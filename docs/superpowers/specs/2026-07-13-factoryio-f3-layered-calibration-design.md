# Factory I/O F3 分层校准重建设计

**日期：** 2026-07-13
**状态：** 用户已批准设计方向，待书面规格复核
**范围：** `l2_factoryio/` 的 Factory I/O Python 直控技术预演；F3 通过前不进入 AB OPC UA 或正式录屏。

## 1. 问题与证据

当前问题不是单一“围栏碰撞”，而是几何认知、控制判据和动力学观察混在一起：

- Claude F2 点表与初版控制器（`d85f1d8`、`32c46c8`）采用 `Right 取货 / Left 入架`，但失败快照显示取货后 `At Load=False`，货物仍在载入位；初版没有验证 `At Load` 清空，完成日志可能是假成功。
- 后续现场探针采用 `Left 取货 / Right 入架`，Left 动作曾使 `At Load` 清空，说明这一方向更符合当前场景；但堆垛机没有 cargo-present 输入，不能仅凭叉位限位证明货物已平稳在车上。
- 调试场景与重新加载的官方场景各有 45 个对象，StackerCrane、Rack、LoadingConveyor、RollerConveyor 的位置完全一致；12 个 Safeguard 仅出现等价四元数 `q/-q`，没有实际位移。因此不以移动输送机、货架或围栏作为首修。
- 现有诊断命令在中间相位结束后执行 `all_stop()`，会清除 Lift，可能改变正要观察的承载状态；“命令成功”不能代替物理画面验收。

官方 Stacker Crane 文档仅定义 Target、Left/Right、Lift、Moving X/Z 和叉位限位，没有场景侧别绑定或 cargo-present 标签：
https://docs.factoryio.com/manual/parts/stations/

## 2. 目标与非目标

### 目标

1. 把入料、取货、带载移动、入架拆成四个可独立证伪的物理门。
2. 每个门同时使用固定画面、I/O 快照和动作日志判定，消除信号假成功。
3. 先使货位 1 单箱流程物理稳定，再扩展到代表货位 30、54。
4. 保留现有超时、互锁、safe-stop 和离线回归能力。

### 非目标

- 不移动或删除官方场景的输送机、Stacker Crane、Rack、围栏或控制柜。
- 不把 Factory I/O 54 格称为 20×20 物理孪生；20×20 仍由 AB 业务层和显式 400→54 映射承担。
- F3 未通过前不进入 AB OPC UA、自研 3D 同源回放或正式素材录制。
- 不以提高物理解算参数掩盖动作顺序、载入落稳或方向错误。

## 3. 设计决策

采用“官方几何不动、控制分层校准、物理画面优先”的方案。

### G0 几何与观察基线

- 重新加载官方 `Automated Warehouse`，使用调试副本恢复 Modbus 映射，但不得改变对象 transform。
- 固定一个能同时看到载入输送带、承载台、货叉和首个货位的俯视/斜俯视相机。
- 画面中区分橙色立柱与实际货叉承载中心；保存基线截图和 XML transform 摘要。

### G1 入料落稳门

- 循环开始先确认堆垛机位于 rest、叉臂 Middle、Lift=False、C0..C6=False。
- 再运行 Entry + Load Conveyor；At Load 被遮挡后立即停止两段输送带。
- 停带后设置独立落稳窗口；窗口结束时 At Load 必须仍被遮挡，货物必须水平、无持续滑动或摆动。
- 若货物此时已经倾斜，停止后续动作，仅调入料时序/落稳窗口，不碰货叉、围栏或物理参数。

### G2 取货承载门

- 初始工作假设采用现场证据较强的序列：Left 伸叉 → Lift=True → 确认 Lift 动作完成且 At Load 清空 → 回 Middle；未经本门现场通过不得升格为定论。
- 到达 Middle 后保持 Lift=True，不由诊断收尾自动清除 Lift。
- 由 Factory I/O Pause 冻结现场，确认托盘/箱体水平位于承载台、未接触围栏或柜体，再清理输出。
- 若 Left 不能稳定取货，才执行一次 Right 空载/单箱对照；方向必须由 At Load 变化和固定机位画面共同裁决。

### G3 带载移动门

- 仅在 G2 通过后，从 rest 向货位 1 做最短带载移动，全程保持 Lift=True、叉臂 Middle。
- 先使用正常 1×；必要时仅为观察切慢速，不把慢速成功当最终验收。
- 到位后 Pause，货物必须保持水平，无掉落、穿透、持续抖动或防护网接触。

### G4 入架门

- 若 G2 证实 Left 为载入侧，则本门使用其对侧序列：Right 伸叉 → Lift=False 下放 → 独立落稳窗口 → 回 Middle；若 G2 裁决相反，两门方向必须成对翻转并同步修正文档与测试。
- 只有确认托盘已由货架梁承载后才允许回叉；回叉后返回 rest。
- 货位 1 连续成功后，再依次验证 30、54；任何失败都回到对应物理门，不跨门调参。

## 4. 控制与诊断边界

- `all_stop()` 保持为异常和流程终态的安全动作；不得再作为带载中间相位的观察手段。
- 中间相位使用“保持安全承载状态 + GUI Pause + 取证 + 显式清理”的诊断协议。
- `position_hint` 只作为进程内提示，不能替代 Factory I/O 实际运动/限位证据。
- 每次实验前 F6 复位，记录命令、时间、I/O 前后快照、固定机位截图和人工物理判定。
- 单元测试验证动作顺序、超时、清理和相位边界；单元测试通过不代表 F3 物理通过。

## 5. 验收标准

每个物理门均须同时满足：

1. **画面：** 货物水平、无碰撞、无滑落、无持续抖动、无穿透。
2. **I/O：** 预期限位/At Load/Moving 状态出现，非预期输出为 False。
3. **日志：** 每个动作有起动、到位/停稳、耗时和失败清理记录。

校准通过条件：货位 `[1, 30, 54]` 各完成一次完整入库，三次均满足上述三重证据。F3 最终通过条件：再重复两轮，形成 `[1, 30, 54] × 3` 共 9/9 成功记录。若任何一次失败，F3 保持红灯并回到对应物理门。

## 6. 备选方案裁决

- **不采用直接恢复 Claude 初版：** 已有 `At Load` 未清空的假成功证据。
- **不采用移动输送机/删除围栏：** 场景 transform 与官方基线一致，现阶段没有几何误移证据。
- **物理参数调整仅作后置 A/B：** 只有动作方向、落稳与承载状态均正确后仍复现抖动，才测试 timestep/solver 参数，并保留 1× 默认参数终验。

## 7. 决策记录

- 2026-07-13：用户批准停止连续试错，采用分层校准重建。
- 2026-07-13：官方几何保持不动；控制流程以 G0→G4 门禁推进。
- 2026-07-13：F3 通过前冻结 AB OPC UA、自研 3D 同源回放与正式录屏。
