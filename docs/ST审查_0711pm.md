## 阻塞(Blocking)

### 【阻塞-1】FC_TMax 对可调参数 rVx/rVy 做无防护除法,同文件其余 3 处同款除法均有防护——本处遗漏
**文件:行**:`03_Functions.st:176-177`,受害调用点 `03_Functions.st:205`(FC_CalcScore 内)

```st
FC_TMax := MAX(INT_TO_REAL(GVL_Param.COLS - 1) * GVL_Param.CELL_W / GVL_Param.rVx,
               INT_TO_REAL(GVL_Param.TIERS - 1) * GVL_Param.CELL_H / GVL_Param.rVy);
```

**问题描述**:`rVx`/`rVy`(堆垛机水平/垂直速度)是 `02_GVL.st:30-31` 声明的**非常量**全局变量,
注释明确写"现场可演示灵敏度"(A4 调参区,管理员可在线改)。全仓对这两个变量做除法的地方共 4 处
(`FC_CalcTravelTime` 03_Functions.st:129-130、`FC_TravelLaden` 03_Functions.st:368-369、以及本处
`FC_TMax`),前两处都先把值拷到局部变量并 `IF vx<=0.0 THEN vx:=1.0`/`IF vy<=0.0 THEN vy:=1.0` 钳位后
才除,唯独 `FC_TMax` 直接对全局 `GVL_Param.rVx`/`rVy` 做除法,没有任何守卫——这是一处防护不一致
的遗漏,不是设计如此。`FC_TMax()` 是 `FC_CalcScore` 的评分归一分母(`tNorm := 行程时间/FC_TMax()`),
而 `FC_CalcScore` 是 `FB_SelectSlot`(score 策略)、`FB_LocalSwapImprove`(重定位+交换两阶段全用)、
`FB_GoodsInput`(交互推荐)共用的核心评分函数——一旦触发,污染面是全链路评分,不是局部小问题。

**触发场景**:答辩/演示时管理员登录后在"调参区"把 `rVx` 或 `rVy` 手误改成 `0`(清空后未及时填值)
或误入负号。IEC/CODESYS 的 REAL 除零通常返回 IEEE754 ±INF 而非触发运行时故障,PLC 不会报警、
不会崩溃、`ViolCnt`(违规数,只查重量/容积)仍为 0——但 `FC_TMax()` 变成 ±INF 或(若两轴都异常)
变成一个负值,`tNorm` 随之变成 0 或负数,`FC_CalcScore` 的效率项/预留项符号错乱,**选位与局部搜索
从此刻起悄悄给出违反直觉的评分排序**(远/重货可能被判定"更优"),且没有任何可见报警提示原因,
比崩溃更难定位——评委若追问"为什么调完速度参数优化结果变差了",现场很难当场查出根因。

**建议修法**:在 `FC_TMax` 内比照 `FC_CalcTravelTime` 的写法,先把 `GVL_Param.rVx`/`rVy` 拷到局部
变量并做 `<=0.0` 钳位(钳到 1.0 或任意正值)后再相除,与全仓其余 3 处除法防护风格统一。

### 【阻塞-2】FB_LocalSwapImprove 的"重定位"阶段未排除 aMaintBlock 冻结货物,与同函数"交换"阶段的防护不对称,违反 A3 文档承诺的"已有货物原地冻结"
**文件:行**:`04_FB_Warehouse.st:290`(重定位阶段判断条件,缺 aMaintBlock 检查)对照
`04_FB_Warehouse.st:343-346`(交换阶段的正确防护写法)、执行搬移的 `04_FB_Warehouse.st:310-313`

```st
290	            IF GVL_WH.aGoods[iGi].Assigned THEN        (* 重定位:只判断已分配,未判断"是否在检修格" *)
...
310	                IF iBx >= 0 THEN
311	                    // 搬移:旧位腾空,新位占用,货物记录/可视化联动
312	                    GVL_WH.aSlots[GVL_WH.aGoods[iGi].X, GVL_WH.aGoods[iGi].Y].Occupied := FALSE;
```
对照交换阶段(同一 FB 内,正确写法):
```st
343	                IF FC_FitsPhysical(x := xjA, y := yjA, ...)
345	                AND FC_FitsPhysical(x := xiA, y := yiA, ...) THEN   (* 双向都查,含各自现居格的 aMaintBlock *)
```

**问题描述**:README_PLC.md §3 与 `04_FB_Warehouse.st:1022` 的 FB_UserAuth 注释都明确写检修挂起
(A3)语义="SAP EWM putaway block:禁新入库(选位/分配/交换均排除),**已有货物原地冻结**"。
交换阶段(phase 2)确实双向调用 `FC_FitsPhysical`,而 `FC_FitsPhysical` 内部检查目标格
`aMaintBlock`(`03_Functions.st:84`)——所以若货物 iGi 现居格被标记检修,`FC_FitsPhysical(x:=iGi现居格...)`
会返回 FALSE,阻止把 iGj 换进来,间接也保护了 iGi 不被换走。但重定位阶段(phase 1,290-330 行)
只判断 `Assigned` 一个条件就直接对该货物做全库最优位搜索,搜索循环虽然正确排除了"检修格作为目标"
(`FC_CheckFeasible` 内含 aMaintBlock 检查),但**完全没有检查这件货物自己当前是否正站在检修格上**——
若找到更优空位,290-333 行会照常把它从(可能是检修中的)原格挪走、原格 `Occupied:=FALSE`,与"原地
冻结"的书面承诺矛盾。两个阶段对同一件事(移动一件已上架货物)防护不对称,像是重定位阶段的遗漏而
非有意设计。

**触发场景**:检修挂起(`CmdMaintToggle`)在 `05_PRG_Main.st` 中位于"L6 常驻段"(第189行之后),
**不受当前状态机状态限制、每周期都读取**,可以在 `STATE_IMPROVE` 分片运行期间随时按下——不需要
"先完成一轮再重新触发"这种迂回路径。典型演示场景:载入演示批→选 AWRA-LS→运行分配,分配完
自动进入 STATE_IMPROVE(20 件小批也会跑重定位+交换分片);此时操作员对着屏幕把某个已经放了
货的格子标记"检修"(演示"设备故障"叙事),如果局部搜索的重定位阶段恰好还没扫到这件货(20~400
件按 `nBatchPerCycle=10` 分片,通常要跨多个扫描周期才扫完一轮),下一次扫到它时会把它判定为
"可继续优化的已分配货物",若别处有更优空位就直接把它挪走——刚标记的"检修冻结"当场被打破,
且没有任何报错提示,只有细心对照坐标才能发现。

**建议修法**:在 290 行的判断条件补一个否定检查,即"该货物已分配 且 其当前格未被检修挂起"才
进入重定位搜索,与交换阶段的防护逻辑对齐(可直接复用 `GVL_WH.aMaintBlock[GVL_WH.aGoods[iGi].X,
GVL_WH.aGoods[iGi].Y]` 取反作为第二个条件;注意 IEC AND 不短路,两个条件都是简单字段读取,无
数组越界风险,可以安全地合并进同一个 IF)。

### 【建议-2】aSlotOps 是无上限、Reset 不清的持久 INT 计数器,理论上可溢出但当前量级极低
**文件:行**:声明 `02_GVL.st:82`;递增点 `04_FB_Warehouse.st:211`(assign)、`04_FB_Warehouse.st:319`
(relocate)、`04_FB_Warehouse.st:360-361`(swap,两格各 +1)

**问题描述**:`aSlotOps`(C6 损耗履历热力图数据源)按设计"Reset 不清"(`02_GVL.st:82-83` 注释),
跨越整个 PLC 运行期持续累加,类型是 `INT`(IEC 范围 ±32767)。任务清单要求核查"计数器/累加器
有无溢出"——严格说这是一个无上限累加器,理论上跑得足够久会溢出。但实际量级测算:单次完整
"分配+局部搜索"跑批(20~400 件),单个格子最多被计入几次(assign 一次 + 局部搜索改进阶段的
个位数次重定位/交换),要撑到 32767 需要约 3000+ 次完整批次重复命中同一格,这在比赛演示的时间尺度
(几天内的现场演示+评委轮番试跑)下基本不可能触发。

**建议修法**:可选(非紧急)——若担心长时间无人值守压力测试/连续演示场景,把 `aSlotOps` 元素类型
升级为 `DINT`(同 `FB_ScanLoadProbe.nSamples` 已经采用的做法),一次性消除顾虑,成本极低。当前
不升级也可以接受,仅作为审查清单条目如实记录。

### 【建议-3】FB_SelectSlot 的浮点等值比较 `rS = rBestScore` 经复核是安全的,但建议加注释防止后人"误修复"
**文件:行**:`04_FB_Warehouse.st:107`

```st
xBetter := (rS < rBestScore) OR ((rS = rBestScore) AND (rT < rBestTime));
```

**问题描述**:表面看是审查清单里的经典陷阱"浮点用 = 直接比较",但深入复核后发现这里的相等
判断是**刻意且安全**的:`rS`/`rBestScore` 都来自同一个确定性纯函数 `FC_CalcScore` 对相同有效
输入的重复求值——而 `FC_CalcScore` 的 `stab`/`energy` 项只依赖 `y` 不依赖 `x`(`03_Functions.st:
208-209`),`eff`/`resv` 项依赖的 `tNorm` 在切比雪夫模型下,只要候选格的水平行程时间不超过垂直
行程时间(`tx<=ty`),不同 `x` 会得到位级相同的 `tNorm`——因此对同一层 `y`、位于"垂直轴主导"
区间内的不同列,`FC_CalcScore` 会返回逐位相同的 IEEE754 值,不是"数学上相等但因舍入误差凑巧
不等"的那种危险相等判断,而是"同一段代码用相同输入重算,结果必然位级相同"的良性相等,用于
在真正平局时把 tie-break 交给行程时间、最终由扫描顺序(x 外层)保留列号最小者。经手工复算多
组 (x,y) 组合验证成立,当前实现没有 bug。

**建议修法**:非必须修复,但建议在此行上方加一句注释说明"此处 = 比较是刻意的位级平局判断,
不是浮点误用",防止未来代码审查或新同学看到 `=` 比较就"顺手"改成 `ABS(rS-rBestScore)<eps`——
那样反而会因为 `eps` 选取不当悄悄改变平局时的选位结果(T36-T40 等选位类用例可能因此变化),
是好心办坏事的风险点。

### 【建议-4】两种"单拍完成"边沿检测写法并存,增加维护心智负担(非缺陷,已逐一验证两种写法各自在其调用方式下正确)
**文件:行**:`FB_InitWarehouse`(`04_FB_Warehouse.st:51`)与 `FB_Stats`(`04_FB_Warehouse.st:463`)
使用 `xOld := xExecute AND NOT xDone`(0711 修复版,专门应对"跨状态多次重入")；而
`FB_AssignAllGoods`(`04_FB_Warehouse.st:158`)与 `FB_LocalSwapImprove`(`04_FB_Warehouse.st:283`)
仍用最朴素的 `xOld := xStart`。

**问题描述**:本条不是 bug——已逐条追踪 `05_PRG_Main.st` 的调用方式:前两个 FB 是"单周期内跑完
全部逻辑"模式,配合状态机同周期转出的调用方式,需要特殊的 `AND NOT xDone` 写法才能在跨状态
重入(尤其 Reset 后)时正确重触发,已验证 Reset 中断/正常访问两种路径都触发正常;后两个 FB
是"跨周期分片"模式,`PRG_Main` 保证 `xStart` 只在状态入口给一个干净的单周期脉冲,朴素写法在此
调用约定下同样正确。也验证了 Reset 在 `fbAssign`/`fbImprove` 分片运行期间强行打断的场景——由于
状态机顶部的复位分支会在同一扫描周期内把 `eState` 切到 `STATE_INIT`,使 `CASE` 分支当周期不再
落入 `STATE_ASSIGN`,`fbAssign` 最后一次被调用时 `xStart` 必为 FALSE(kick 脉冲早已清零),
`xOld` 停留在 FALSE,下次重新进入 `STATE_ASSIGN` 能正确重触发,未发现"重入后卡死/不重触发"
的死角。仅因为同一文件内两套边沿检测习惯并存,留作可读性建议。

**建议修法**:非必须。若后续有时间做代码整理,可考虑统一成一种边沿检测写法并在 FB 头部注释
写明"本 FB 期望的调用约定"(单拍完成 vs 跨周期分片),降低新增 FB 时选错模式的风险。
