# B6 · §4.5 情景参数稳健性 + 免费改进池条目

## §4.5（一）情景参数稳健性

**方法**：针对 canonical 标注"情景假设-显式声明"的三轴——G2 到达强度（峰谷利用率 ρ）、G3 频次估计噪声（σ）、G4 故障画像（MTBF）——补做单参数扫描（OFAT），检验"AWRA-LS 布局优于顺序基线"是否对情景取值敏感。复现脚本 `sim\scan_g2g4.py`，数据 `sim\out\scan_g2g4_agg.csv`（汇总）与 `scan_g2g4.csv`（逐 seed 明细 840 行）。扫描点（`docs\G2G4敏感性扫描_0710.md:11-15`）：G2 ρ∈{0.60,0.75*,0.90}、G3 σ∈{0.15,0.30*,0.45}、G4 MTBF∈{1500,3000*,6000}s（*=canonical 基准）。引擎零改动复用 `realism.py::run_tier3`，30 seed(2026–2055)×7 点×4 策略，CRN + 配对 t 检验。

**结论**：全 7 配置点上 AWRA-LS 相对顺序基线的出库均时优势 100% 稳健：
- base：awra 10.931s vs seq 20.583s，降幅 46.89%，t=−44.04***，30/30 配对全胜（`scan_g2g4_agg.csv:2`）
- G2 ρ=0.60/0.90：降幅 46.89%，P95 随 ρ 486→801s 单调上移（`:3-4`）
- G3 σ=0.15/0.45：降幅 47.06%/46.89%，t=−44.24/−43.66***（`:5-6`）
- G4 MTBF=1500/6000：降幅 46.89%，P95 1047.5s/482.3s（`:7-8`）

机理（`G2G4敏感性扫描_0710.md:37-40`）：出库均时是纯行程口径，G2/G4 不改行程几何→对这两轴免疫；G3 噪声经布局间接影响，差异在 ±0.2pp 内。P95 对 ρ 单调升、对 MTBF 单调降 → 拟真层机制自洽而非调参巧合。

**诚实边界**（`:41-44`，列 `pair_rate_p50_pct`）：P50 口径下 awra<seq 配对优势率降至 83–100%，最弱点=重故障 MTBF=1500（83.3%）——单次 600s 停机砸中哪单具随机性、P50 被 MTTR 主导；重故障场景下布局优化不是主要矛盾、可用性（维修组织）才是。此边界建议原样写入报告。

**图表指向**：G2/G3/G4 三组各绘"降幅%（左轴近水平线）+ P95（右轴单调折线）"双轴图，呈现"均值免疫、分位单调"；逐 seed 明细可作配对优势率箱线图。

## 免费改进池 / 决赛展望：关联存储（零先验双命令感知选位）

**条目**：关联存储 correlated storage——YAGNI 结论 + 零先验替代 link（−9.81%***）。复现 `sim\correlated_storage.py`，数据 `sim\out\correlated_storage.csv`（30 seed 配对 360 行）。三重论证（`docs\关联存储实验_0710.md:46-53`）：
1. **机理**：本项目单元负载（unit-load）堆垛机，correlated storage 经典收益通道（一次巡回取多件）解析恒等不存在，与机型错配（`:13-16`）。
2. **实验**：族先验方案 corr 在无/弱共现下显著有害——θ=0.0 时 +9.12%***、θ=0.5 时 +3.15%***、即使 θ=1.0 也仅 −2.44%**（`:37-39`）。
3. **替代**：零先验 link（选位时于 score 前 10 可行位偏向当期取货位，不依赖族/共现统计）在全部 θ 恒定 **−9.81%*****（连段均时 5.36s vs base 9.40s）；corr_link 恒劣于纯 link——族先验无独立价值。

**代价**（`:41-44`）：link 出库均时 +0.31s（p<0.001）、终局取货 +3.1%（轻）；corr 代价约 3 倍。违规全 0。该文档 0710 夜经独立验证员 fresh-context 对抗核验（CRN 逐 seed 零差、位级同源、约束安全），两处副作用数值经修正（方向使 corr 代价更大、YAGNI 更稳）。

**定位建议**（暂不入主线、决赛可加分）：link 属口径级变更（动了选位逻辑），按保守采纳边界不直接入主口径（`:60`）；收益有限、进主口径需动 `make_score_fn` 全量重验、ROI 不划算（`:64`）；推荐作"决赛展望/免费改进池"独立探索条目，叙事="以实验证否复杂方案、以更简方案验证增益留作展望"。边界（随条目写入）：随机分族为保守侧、"族内频次相似"现实场景未扫、top-K=10 未调参、仅双命令流适用。

# B7 · 吞吐口径边界声明 + L4 阶梯叙事合并

## 一、吞吐口径边界声明（230 次/h）

**是什么**：`230 次/h` 出自 `1_给你看\实验发现.md:17`（F18 表 F8 行，官方 sum 口径，替代切换前 268 次/h）。计算见 `sim\mixed_ops.py:101-105`：`n_ops=2*len(new_goods); thr_dual = n_ops/(tot_dual/3600)`。`tot_dual` 是 `n_cycles`（默认 100）个"存 1+取 1"周期的**运动学仿真行程时间**累加（切比雪夫+梯形加减速，H 口径 `--accel --adaptive`）。230 次/h = "若堆垛机按该批仿真平均单周期行程连续不间断运行，理论每小时完成多少次双命令存取"。固定 SEED=2026，单次运行、单负载序列，非多次重复统计量。

**不是什么**：
- **不是 AC500/AB 实测吞吐**——不经 PLC 扫描周期(10ms)、OPC 写值、I/O 轮询、看门狗、分片开销（这些只在 L4 口径出现）。
- **不是多周期统计量**（无 CI、无多 seed）。
- **不是档3（泊松到达）下成立的量**——`现状与任务.md:281`（深思#2）指出泊松下"吞吐=到达率"、不再反映布局优劣；230 次/h 只在档2（连续满载假设）有意义。
- **不是"AC500 实测吞吐 230 次/h"**——属 D-4 已标错配点，不入报告定论区。

**报告安全表述**（建议在 §3 头条区 + §4.5 拟真阶梯区加边界句）：
> "吞吐 230 次/h 为 H 口径运动学仿真（sim/mixed_ops.py，双命令周期模型、连续满载假设、忽略控制器扫描/通信/机械限位开销）下的**方法论换算值**，用于量化布局优化对理论吞吐上限的提升（较基线 +74%），并非 AC500 在线实测的多周期统计吞吐；控制器侧实测口径见 L4 任务级完成周期数。"

若需"实现档"实测锚点对照，可用 `docs\D2量化_0712.md`（AC500 静态权重真跑=取货 8.76s/降 60.4%/能耗省 67.2%/违规 0），但吞吐本身该文档未给 PLC 实测值，**不要**把 230 次/h 与这组并列成"同一套 AC500 实测"。

# B11 · 数字孪生成熟度阶梯（决赛展望）

本项目技术路线天然分层：Python 建模仿真产出算法/数据、AC500(CODESYS 系)ST 承载控制逻辑、AB 画面+Factory I/O 承载可视化与物理验证——恰对应数字孪生成熟度递进。

## 三层阶梯（文字版结构图，供配图）

```
L0  设计时仿真（Design-Time Simulation）
     Python 建模 + 统计实验，产出策略、参数、上界
     └─ 路径：ST 移植（算法/评分函数 → AC500 ST）
                ▼
L1  运行时画面级孪生（Runtime Visual Twin）
     控制逻辑运行在 AB/CODESYS，画面实时渲染 PLC 内部状态
     └─ 路径：控制信号改走 OPC UA 连独立物理引擎
                ▼
L2  运行时控制在环（Runtime Control-in-the-Loop）
     控制器 ↔ Factory I/O 物理引擎经 OPC UA 闭环，
     驱动"会掉/会堵/会被挡"的虚拟工厂而非渲染函数
```
本质区别：L0 验"算法数学上对不对"、L1 验"控制逻辑跑起来+画面对不对"、L2 验"控制逻辑接到有物理约束的世界还能否正确驱动执行器"——工业界"虚拟调试 Virtual Commissioning"的完整闭环（`2_你要操作\FactoryIO仿真上手教程.md:61`）。
