## 1. 站得住的结论(可用于设计与报告)

| # | 结论 | 置信 | 一手依据 |

| 1 | 高密度数据可视化分三类互补技术:**clustering**(客户端屏幕空间动态分组,随缩放重算)/ **aggregation-binning**(服务端或预定义边界的**静态**聚合,含 hexbin)/ **heat map**(连续密度面)。原则:**「能画不代表该画」** | high | ArcGIS 官方文档(developers.arcgis.com、doc.arcgis.com) |

## 2. 由此得出的设计调整

### 关键重新框定:2D 图「6px/格」不是可视化方法问题,是我自己造成的

2D 货位图当前画布 **133×133**,20×20 → 每格 6.6px。
但 133px 是**左栏空间不足**逼出来的(左栏内容 914px > 可用 850px)。
**若左栏重排后给 2D 图 ~280px,每格 14px,逐格色阶完全可读**——而 warehouse-science 的一手先例正是「2D 平面逐格上色」。

→ **2D 侧的正解是先解左栏空间(R2 步骤2),而不是发明聚合算法。** 这也符合决策梯①「这功能需要存在吗」。

### 3D 侧:保留逐格作细节层,总览层改用「分区聚合」而非「热区包络」

| 层 | 做法 | 依据 |
|---|---|---|
| 细节层(放大/近看) | 货位背板色阶(现状,`cellHeatMesh` + `setColorAt`) | warehouse-science 一手先例支持逐格;three.js 官方 API |
| 总览层(远景) | **分区聚合**:20×20 按 4×4 或 5×5 分区求均值,渲染为少量大色块/轮廓 | ArcGIS aggregation(有一手依据),**取代**证据垮掉的「Golden Zone 包络」说辞 |
| 切换 | 按格子屏幕投影边长触发,阈值**自行论证并记录为本项目经验值**,不谎称引用标准 | ArcGIS LOD 惯例;无统一数字 |

### 诚实标注纪律
- 边缘条带投影若采用 → 标注为**本项目的类比借用/原创设计**,不写「业界成熟做法」。
- 分区粒度(4×4 还是 5×5)**文献没有针对 20×20 的推荐值** → 需原型对比后自行论证。
