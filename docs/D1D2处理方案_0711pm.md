### 1.1 一个待你拍板的措辞选择(不是对错,是分寸;5 秒选一个)

你在任务书里给的示例措辞是:"PLC 主口径采用**经该表标定的稳健静态权重**,运行时切换留决赛展望"。

**核验结果:这句话有一处不够准确,建议不要照抄。** 证据链:

- `plc/02_GVL.st:53-56` 的静态默认值(α1.0/β0.6/γ0.4/δ0.15)在 `canonical_assumptions.md` 里
  属于 **"E 节"(2026-07-03 初标定,单 seed)**,是 **E v2 表(18 场景自适应表)之前**的老基线;
- `canonical_assumptions.md:92`(E v2 节第三条)原文自己说:"上表固定默认(α1.0/β0.6/γ0.4/δ0.15)
  **降级为对照口径与回归锚**,继续用于历史数字复现"——即项目自己的文档把这组静态值定性为"被 E v2
  表取代的旧基线/对照组",不是"E v2 表标定出来的结果";
- `sim/adaptive_weights.py:38`:`DEFAULT = (0.15, 1.0)  # 当前默认(δ, β+γ)` 这行代码把这组静态值
  当作**标定实验的对照基线**(即表格要打败的对手),而不是表格的输出;
- 18 组表值与静态默认逐一核对**大多数不同**(如 (120,"uniform","online"): (0.20, 0.6) vs 静态
  δ=0.15/β+γ=1.0),不存在"表标定出这组静态值"的因果关系。

**结论**:PLC 当前用的固定权重,是"E v2 表出现之前的历史基线",不是"用 E v2 表算出来的稳健值"。
若写成"经该表标定的稳健静态权重",评委只要翻到 canonical_assumptions.md E v2 节自己那句"降级为
对照口径"就能反手拆穿——这是一个自己给自己挖坑的措辞,比不提这件事更危险。

**两个选项**(下面 §1.2 位置6/7/8 的 before→after 默认采用**选项B**,如果你更看重简洁、接受这点

> 定性估算,基于已读代码结构类比,非实测工时;给区间+风险等级,不假装精确。

### 2.1 D-1:补 TOB + CB 两策略执行器 + 检测器分档路由

**现状**(已读代码确认):`plc/04_FB_Warehouse.st:126-234` `FB_AssignAllGoods` 目前只支持
`iStrategy∈{0,1,2}`(seq/near/score)经 `FB_SelectSlot` 逐件评分选位;"AWRA-LS"实际是
`xUseRank:=TRUE + iStrategy:=2` 的组合,再接 `FB_LocalSwapImprove` 做局部搜索
(`05_PRG_Main.st:127-129` 的 `SelStrategy→(xUseRank,iStrat)` 翻译逻辑)。排序阶段用的优先级函数
`FC_CalcPriority`(`03_Functions.st:219-234`)是 `λ1·freq+λ2·weight+λ3·vol` 的加权公式,不是纯频次。

- **TOB(全周转率存储)**:sim 侧(`sim/strategy_lib.py:76-92` `run_full_turnover`)= 按频次(或
  COI=体积/频次)排序全批,再逐件放最近可行位——**放位逻辑与已有的 iStrategy=1(near)完全一致**,
  唯一要新增的是"纯频次/COI 排序"这个优先级函数(对照 `FC_CalcPriority` 加一个变体,几十行量级)+
  新增 `SelStrategy` 枚举值 + `PRG_Main` 分派分支。**结构性风险低**,估 **0.5-1 天**(含新增
  golden-vector 回归用例,仿 T25 做法)。
- **CB(分级存储)**:sim 侧(`sim/strategy_lib.py:95-121` `run_class_based`)= 先按累计频次
  70%/90% 分 A/B/C 三级,再把可行位按行程时间分三段配额,各级只在自己的段内找最近可行位(段内无解
  再兜底全局搜索)。**这是真正的新逻辑**,不能靠已有 FB 简单复用:需要"先算总频次+分级"的两遍扫描
  + 一个"分段限定起点、段内穷尽再回退全局"的选位状态机,比 TOB 复杂不止一档。估 **1-2 天**(含
  回归用例)。
- **检测器分档路由**:`FB_SceneDetect`(`04_FB_Warehouse.st:922-1017`)目前硬编码 holistic 档
  阈值。给 lex/speed 档加路由本身很便宜(`density` 特征已算好,阈值判断是几行 IF/ELSIF,对照
  `sim/strategy_lib.py:160-171` `select_strategy` 的三档分支照抄即可),**但路由做出来若没有
  CB/TOB 执行器可路由到就没有意义**,必须排在上面两项之后。加一个"目标档位选择"HMI 开关(供演示
  切换 holistic/lex/speed)是新画面元素,本项目历史上画面搭建是最慢/最易卡壳的环节
  (`docs/画面输入失效攻坚_0711.md` 全篇即为此事的攻坚记录)——估路由逻辑本身 **0.5-1 天**,
  画面开关另计,**不确定性较高**。
- **回归验证**:`plc/06_PRG_Test.st:16` `N_CASES:=59`(当前 59/0 全绿)。新增两档策略需要各自的
  golden-vector 用例(仿 T25:`sim/export_st_vectors.py` 生成期望值 + PLC 侧比对),估
  **0.5 天**,前提是 TOB/CB 是**新增分支**、不改动现有 SelStrategy=3(AWRA-LS)默认路径——若做到
  这点,对已有 59 条用例应是零风险(纯增量)。

**D-1 小计:约 3-5 天 ST+测试工作**(不含 HMI 新增开关与演示重录)。

### 2.2 D-2:补 FC_ClassifyScene + 权重查表接入

**现状**(已读代码确认,含一条独立新发现):`plc/04_FB_Warehouse.st:988-997` `FB_SceneDetect` 每
周期已经算出 `n/density/skew/w_mean/w_heavy`(与 `sim/strategy_lib.py:125-142` `batch_profile()`
完全对应的特征,**这部分已经"移植"过了,不用重做**)。但 `08_GVL_WeightPolicy_generated.st` 需要的
是另一套**三分类索引**(`iDensity` 0/1/2、`iDist` 0/1/2=uniform/skew/heavy),这不是同一件事。

**⚠️本文件独立核实的一条重要发现,会显著影响这项估工**:sim 侧**从未实现过**"给定一个已存在的
批次,判断它是 uniform/skew/heavy 中哪一类"的分类器。证据:全仓 grep `lookup_weights(` 的全部
调用点(`correlated_storage.py:220-221`、`energy_report.py:106-107`、`headline.py:32-33`、
`mixed_ops.py:124-125`、`realism.py:143-144,164-165`、`scan_g2g4.py:73-74`、
`warehouse_sim.py:487-488`)无一例外把 `case` 参数当**硬编码字符串**传入(几乎全部固定传
`"skew"`);`sim/warehouse_sim.py:157` `gen_goods(n, seed, case="skew")` 证实 `case` 从来只是
**生成批次时选用哪种随机分布的"配方参数"**,不是"事后从一批货物的统计特征反推它属于哪一类"的
分类结果。也就是说,`08_GVL_WeightPolicy_generated.st:5` 注释里那句"iDist……按库存重量均值与频次
变异系数判"描述的是一个**从未被写过、甚至从未在 sim 里验证过的设计构想**,不是"sim 有、PLC 缺"的
移植关系,而是"sim 和 PLC 都没有,需要从零设计"的关系。

这意味着 `FC_ClassifyScene` 不是像 D-1 的 TOB 那样"抄 sim 现成公式翻译成 ST"的低风险移植,而是
需要:①设计一个分类规则(如按频次变异系数 CV 分档);②**回头验证**这个新分类规则,喂给当初标定
18 场景表时用的那些批次,能否重新分类回原本的密度/分布档位(否则查表会查到未经验证的权重组合,
比"没有自适应"更危险——查错表可能比不查表更差);③补一个频次方差/CV 的新统计量(现有
`FB_SceneDetect` 没算这个,需要多一趟累加)。

- **FC_ClassifyScene 设计+验证**:估 **1-2 天**(不确定性主要在②的验证闭环,如果简单阈值规则一次
  就吻合 18 组标定场景会很快,如果需要来回调阈值会拖长——这是本项目里少有的"没有现成答案抄"的部分)。
- **查表接入**:`08_GVL_WeightPolicy_generated.st:16-19` 的查表代码本身只有 3 行,且已经写好在
  注释里,"取消注释+接上分类结果"是几十分钟量级的机械工作,**接入本身不是难点**。
- **回归成本(比 D-1 更高的一点)**:D-1 的新策略是"新增分支",D-2 改的是"AWRA-LS 现有路径的输入
  权重"——`sim/export_st_vectors.py:28` 生成 T25 黄金向量时硬编码 `score_fn = ws.make_score_fn(
  1.0, 0.6, 0.4, 0.15, ...)`(静态默认权重),一旦 PLC 权重变成动态,**T25 现有黄金向量必须重新
  生成**,且需确认其余用到 `rBeta/rGamma/rDelta` 的用例(如 06_PRG_Test.st 的评分相关断言)是否
  也要跟着调整。估 **0.5-1 天**。

**D-2 小计:约 2-4 天 ST+设计+测试工作**,不确定性高于 D-1(卡点是①一个从未被验证过的新分类器,
不是移植风险,而是"design of experiment"风险)。

### 2.3 共同的回归/演示重录成本

- **ab_sync 回归**:两组改动都要过 `tools/ab_scripting/ab_sync.ps1`(编译0错+在线 iPassed/iFailed),
  当前 59/0;需要 AB(Automation Builder)在线会话——**今晚 computer use 三次 user_denied,这项

  `docs/演示脚本v3_0711.md` 末尾"长跑验收清单"式的全量回归(冷启→12步→v4 四镜头),此项目历史上
  这类"洗一遍再长跑"的动作按其自身记录通常要数小时(含冷启动、GUI坑排查);③若时间紧张只做 D-1/D-2
  之一,演示脚本需要相应只增补对应镜头,仍是增量重录而非从零重录。

### 2.4 总工作量与风险定性

| 维度 | D-1(TOB/CB+路由) | D-2(FC_ClassifyScene+接入) |
|---|---|---|
| 定性工期 | 3-5 天 | 2-4 天 |
| 移植/设计性质 | 移植为主(sim 公式现成) | **设计为主**(分类器从未存在,含验证闭环) |

| 是否需 AB 在场 | 是 | 是 |
| 是否需演示重录 | 是(新增镜头) | 是(新增镜头) |

**合计:约 5-9 天 ST/测试/设计工作**,不含 GUI 新增开关与决赛演示重录的确定性工时(项目自身历史
显示 GUI 环节是最慢/最易反复的部分),且全程需要用户在场操作 AB(overnight 无法推进这部分)。
对照当前报告周(`现状与任务.md` §3 排期表)已经临近、且本仓库今晚 computer use 三次被拒的实际
情况,这是一个有真实排期风险的选项,不建议在报告周前作为默认路径,可作为决赛前的独立加分项排期。
