## 1. B1 阈值锚定改写

### 1.1 锚 A:de Koster et al. (2007) "55%/70%" —— 【裁决:站得住】

WebSearch 交叉核对多个独立二手引用源(Google 摘要、学术引用聚合),"order picking 成本高达仓库运营总成本

结论,未发现夸大或误引。这是仓储运筹文献中最常被引用的一对统计量之一,报告草稿用"估计""高达"等
词也做了恰当的不确定性保留。**可放心作硬引用**,报告草稿 1.4 节的判断准确。

（未能拿到 PDF 纯文本做逐字页码核对——两次 WebFetch 对该 PDF 的解析都失败,原始 PDF 是扫描/压缩流,

### 1.2 锚 B:"单杠杆 4%~28%" —— 【裁决:需加固,发现一处此前未被指出的引用陷阱】

WebFetch 直接抓取报告来源清单里的 `journal.ummat.ac.id/.../37750`(PT X 案例研究)原文摘要,发现:

- 该研究的 **Class A 降 23%**、**Class B 降 4%**——也就是说,报告草稿"4%~28%"这个区间的**下限 4% 和
  上限附近的 23%,其实来自同一篇论文的同一张表,只是不同货物分类**,不是"多篇独立研究互相印证"出来
  的两个端点。报告草稿的措辞"多篇学术研究:某产线 ABC 改造行程距离降 23%;……"把这两个数字并列成
  看似独立的证据点,但下限 4% 在正文里连来源都没单独标注,读者无法看出它和前一句"23%"同源。
- **更关键的一处遗漏**:同一研究里 **Class C 货物的行程距离不降反升,从 279m 涨到 705m(涨约 153%)**,
  原作者自己的解释是"C 类走动频率低,可接受"。报告草稿完全没提这一负面结果——而本项目在 B2 节
  反复强调"我们不回避代价、诚实呈现局限"(如 AWRA 漂移退化 49%),但在 B1 引用别人的文献时,只挑了
  同一篇论文里好看的两个数字,略去了它自己的负面发现。这在措辞/立场上不一致,评委如果去查这篇二手
  可查的开放论文(非付费墙,我能直接 WebFetch 到摘要),会发现"4%~28%"这个区间的呈现方式经不起追问。

**加固建议**:
1. 明确标注 4% 和 23% 同源(该产线案例的 Class A/B 两档),不要让读者以为是两项独立研究;
2. 要么补一句"该研究 Class C 品类反而上升(设计者认为可接受,因该类走动频率低)"体现同等的诚实标准,
   要么换一组更干净、真正来自不同研究的区间端点(27.55% 的 ResearchGate 论文因 403 未能核实具体数值,
   若报告周能拿到全文,建议改用它和另一篇的端点组合,避免同源问题)。

### 1.3 锚 B:"62% 联合优化上限" —— 【裁决:需加固,报告已做的降级不够充分】

报告草稿已经如实标注"二手转述、原文 403 未核验",这个态度是对的。但复核后发现还有两层报告没提到
的风险,建议一并补充:

1. **二手来源自己的说法就有出入**:WebFetch 抓取 optioryx.com 博客原文,其原话是"**A study done in
   2020** combining SLAP with picking optimization and routing strategies showed... 62%"——博客说这是
   "一项 2020 年的研究",但报告草稿称其为"Taylor & Francis **2023** 年论文"。这两者可能是同一研究的
   会议/预印本(2020)与期刊发表版(2023),也可能博客本身引用张冠李戴——403 打不开原文,这一层"研究
   到底是哪年做的"本身就没锁定,比报告草稿目前呈现的"只是原文未核验"更不确定一层。
2. **域/规模不可比这一点报告完全没提**:62% measures "picker **travel time**"——从"SLAP+picking
   optimization+routing strategies 联合优化"这个措辞看,这几乎可以肯定是**人工拣选员在多通道仓库里
   走动路径**的场景(涉及 routing/批次拣选),而本项目是**单巷道单堆垛机的 AS/RS**,267 个仓位,没有
   人工路径规划这个维度。这是两种规模、自动化程度都不同的系统,即使原文数字被核实无误,"我们的
   62.0% 与联合优化文献上限 62% 持平"这句话隐含的可比性也弱于表面看起来的样子——本质上类似“单杠杆
   vs 多杠杆”那个坑,只是这次是"人工拣选 vs 自动化堆垛机"维度的不可比。

**加固建议**:1.3 节头条段落("仓储运筹文献长期指出……与文献可查的联合优化上限持平")建议加一句
"该文献数字来自人工拣选场景(涉及拣选路径与批次策略),与本项目单堆垛机 AS/RS 系统规模、自动化程度
不同,数值持平不代表方法可比性持平,仅作数量级参照"——把现在只藏在 1.4 节脚注式限定语的"未核验",
提升为正文里同时讲清楚"未核验+域不同"两层,评委追问时才有完整答案,而不是被追问到第二层就语塞。

### 1.4 ASQ 六西格玛 1.7% —— 【裁决:站得住,一处区间数字建议报告周复核】

WebSearch 确认核心结论(28 家企业均值节省营收 1.7%)与报告一致,这条"排除精益 >20% 泛化说法"的反证
逻辑成立,而且是本报告主动发现的一处好的自我纠错(草稿里"重点澄清一处容易踩坑的说法"这段是本次复审
里少见的、报告主动做对抗性自查的例子,值得肯定)。**唯一细节**:报告写"区间 1.2%~4.5%",我的搜索
结果显示该研究还提到"部分企业累计节省最高达 6.8%"——区间口径可能不止一种(如"年度均值区间" vs
"多年累计上限"),建议报告周花几分钟核对原文本身用的是哪个区间描述,但这不影响"1.7% 与我们的 62%
不是同一把尺子"这一核心论点,即使区间数字略有出入也不影响结论。

### 1.5 B1 小结

de Koster 锚站得住,是本节最硬的部分;"62%"上限报告已知道要降级但降级得不够彻底(域/规模不可比这层

### 2.5 装卸时间零影响(F23 三探针)—— 【裁决:站得住,一处措辞可收敛】

对照 `1_给你看/实验发现.md` F23 与 `sim/handle_probe.py` 源码,探针 A(平移)/B(权重扫描 λ∈{0,1,100},
30 seeds×3 权重=90 次布局)/C(反事实层高耦合,30/30 布局改变)三线逻辑与报告描述一致,数字（90 次
布局 100% 相同、30/30 全部改变）均可在代码中找到对应实现,没有编造。

一个很小的措辞提醒:这个命题("位置无关的加性项不改变 argmin")在数学上是初等事实,不需要实验"证明"
才成立——三组探针做的其实是**实证确认**这一模型确实符合"装卸只依赖重量、与位置无关"的前提,而不是
"发现"了什么新数学定理。报告用"证明"这个词略微把"确认前提成立"包装成了"发现了不显然的结论",
建议措辞从"用三组实验证明"改成"用三组实验实证核验(该结论在'装卸位置无关'这一前提下是代数意义上
必然的,三探针核实了我们的模型确实满足这个前提,并用反事实探针 C 排除了"未必如此"的可能性)"——
这样既保留了"我们想清楚了为什么能简化"这个加分点,又不给评委留"这算什么'证明'"的反驳空间。这是本节
里最轻的一条,不影响结论,只是措辞精度问题。

## 3. B10 oracle 写作 —— 本次复审发现的最主要问题

### 3.1 头条数字本身 —— 【裁决:站得住】

`sim/out/oracle_gap_summary.txt` 逐字核对:"检测器[lexicographic]:全局 gap 1.65%,达 oracle 98.5%,
closed gap 93.3%"、"检测器[holistic]:全局gap 24.69%,达oracle 81.6%,closed gap 0.0%"、"tob=397、

`sim/oracle_gap.py` 源码逻辑(VBS 词典序定义、gap/attain/closed gap 公式)也与文档描述一致。

### 3.2 三条既有限定语(closed gap 非逐位等价/ALL 行池化 CI/energy 口径)—— 【裁决:均站得住】

- **closed gap "同构非逐位等价"**:直接读 `sim/oracle_gap.py:100` 与 `:146-147`,确认代码确实是
  "先对 390 个实例的百分比 gap 取均值,再对两个均值做一次比值",不是官方公式原始定义的"先在原始量纲
  上聚合、再做比值"。报告和 sim 审查文档对这条的措辞("同构但非聚合后逐位等价")准确,不是过度声明。
- **ALL 行 CI 池化**:直接读 `sim/instance_gen.py:171-176` 的 `ci95()` 实现,确认它把传入的一组数值
  当作同一总体独立同分布样本处理,没有任何分层逻辑——13 个异质场景 390 个数值直接池化确实会把"场景
  间系统差异"和"同场景内抽样噪声"混在一起。这条限定语准确。
- **energy vs heavy_tier 口径不同**:直接读 `sim/warehouse_sim.py` 的 `metrics()`,确认 `exp_t`/
  `heavy_tier` 是频次或前 20% 件数的**均值**(已归一化),`energy` 是纯 **求和**(未归一化);13 个场景
  n 从 20 到 240 不等,池化平均确实会让"低 32%"这个数字不是单件效率对比。这条限定语准确。

### 3.3 第四个洞(实为最大的洞):PLC 实际只实现 holistic 档,98.5%/93.3% 属于未部署的 lexicographic 档

这是本次复审**发现的最主要问题**,此前报告草稿、三份审查文档、演示脚本均未提及。证据链:

**(a)PLC 策略库物理上不存在 CB/TOB 执行器**——`grep FUNCTION_BLOCK plc/04_FB_Warehouse.st` 逐一核对
全部 15 个 FB(InitWarehouse/SelectSlot/AssignAllGoods/LocalSwapImprove/Stats/AnimatePath/
ScanLoadProbe/BuildVisuPath/VisuRefresh/GoodsInput/QueryGood/RecordPage/**SceneDetect**/UserAuth/
ParamGuard),没有一个实现"分级存储(CB)"或"全周转率存储(TOB)"逻辑;`02_GVL.st:117` 的
`SelStrategy` 枚举只有 `0=seq 1=near 2=score 3=AWRA-LS` 四档,没有 CB/TOB 的编号位。

**(b)代码注释自己写明了这一点**——`plc/04_FB_Warehouse.st:913-915`:"档位说明:holistic=题面全要素
默认档……**lex/speed 档推荐 TOB/CB,PLC 策略库未含**……ST 版按主打叙事实现 holistic 档。"这句话原本
就在代码里,只是从未被拿出来和 B10 节的头条数字放在一起读。

**(c)FB_SceneDetect(实际跑在 PLC 上的检测器)硬编码的就是 holistic 阈值**——`04_FB_Warehouse.st:
998-1005`:`IF rSkew<0.35 AND nLoc<=20 THEN iRecBatch:=1(near) ELSE iRecBatch:=3(AWRA) END_IF`,
与 `strategy_lib.py` 的 `select_strategy(objective="holistic")` 分支逐字对应,没有 lexicographic/
speed 的代码路径。

**(d)holistic 档的真实数字远不如 lexicographic**——`sim/out/oracle_gap.csv` 逐场景读取:
`det_holistic` 全局 attain=**81.61%**、closed gap=**0.0%**(因为 holistic 在全部 390 个实例
100% 选 awra,与 SBS=awra 完全重合,"检测器"这一步骤对结果零贡献);而 98.5%/93.3% 属于
`det_lexicographic`——一个 PLC 上永远不会被选中(因为它多数场景推荐 tob/cb,PLC 无法执行)的纯
Python 假设性档位。**换句话说:报告反复引用来论证"PLC 硬实时约束下的工程最优形态"的那两个数字,
描述的根本不是 PLC 上跑的那个检测器。**

**(e)演示脚本把这个混淆直接写进了台词**——`docs/演示脚本v3_0711.md` v4-1 行:"Load demo 后…

### 3.4 "检测器既充分又必要"论证链 —— 【裁决:需加固,存在跳步,与 3.3 同根】

§3.3 的必要性论证("没有一个固定策略能在全部场景通吃……这是为什么需要场景检测器")建立在**7 个固定

47% 落后、tob 系统性漏件 397 件,数据核实无误)。但这个"必要性"结论被无缝套用到"holistic 档"这个

awra"(100% 选中率,closed gap 0.0%)**——也就是说,已部署的这个检测器,目前并没有实证地展示出它

"我们已经做到了"这个时态上是超前的——目前更准确的表述是"我们论证了场景检测这一**思路**的必要性
(基于全景仿真),并且实现了这一思路的一个**安全默认档**(holistic),但尚未在当前测试场景库中观察
到这个安全默认档相对固定策略的实证增益;能展示增益的其他档位待移植到 PLC"。
