## 2. FB vs FC(FUNCTION)vs PRG 选用规范

**已核对(多来源一致,可信度较高)**:

- **FUNCTION(FC)**:无内部状态记忆,相同输入永远返回相同输出(纯函数);只能有一个返回值(可搭配 VAR_OUTPUT 附加多输出,但主返回值单一)。适用场景:纯计算,如 MAX(IN1, IN2)。
- **FUNCTION_BLOCK(FB)**:通过实例(instance)携带静态变量,状态在扫描周期间保持;必须实例化后调用(`fbName(...)`),每个实例有独立内存。
  - **状态保持判据**:只要输出依赖"上一次调用时的内部状态"(计时器、计数器、状态机、边沿检测等),就必须用 FB,不能用 FC。
  - 同一逻辑要在程序里多处复用、且各处需要独立状态 → 用参数化 FB,每处单独实例化。
  - 无参数 FB 也可仅用于把大型程序拆分成可命名、可折叠的逻辑段(组织用途,非状态用途)。
- **PROGRAM(PRG)**:必须被挂载到任务(Task)上才能执行,是任务周期调用的入口点;不能被其他 POU 实例化调用(FC/FB 可以互相调用,PRG 通常是顶层)。适用场景:每个任务周期的主入口逻辑。

来源(已核对内容与业界共识一致,权威性中等,非 IEC 标准原文逐字引用):
- https://industrialmonitordirect.com/blogs/knowledgebase/structured-text-functions-and-function-blocks-iec-61131-3-tutorial
- https://www.motioncontroltips.com/faq-plc-function-blocks-iec-61131-3-classify/
- Beckhoff/Schneider product-help 页面(多篇一致印证 FB 状态保持机制)

**对你项目的直接映射**(不属于调研发现,是应用提示):200件批量分配算法里,"逐件评分+局部搜索"若涉及跨扫描周期分步执行(避免单周期算太久导致看门狗超时),评分/搜索状态机必须封装成 FB 而非 FC。
