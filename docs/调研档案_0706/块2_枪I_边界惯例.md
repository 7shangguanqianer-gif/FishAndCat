## 1. IEC 61131-3 / CODESYS 数组越界运行时行为

- **CheckBounds 机制**（已核对原文）：CODESYS 官方文档明确 CheckBounds 是数组下标赋值/访问时**隐式自动调用**的函数（"implicitly called as soon as the values are assigned to the ARRAY variables"）。签名固定：
  ```
  FUNCTION CheckBounds : DINT
  VAR_INPUT index, lower, upper: DINT; END_VAR
  ```
  **默认行为是静默 clamp（钳位）**，不是崩溃、不是异常：index < lower 返回 lower，index > upper 返回 upper，范围内原样返回。即"越界访问会被悄悄重定向到边界元素"，不会有任何报错提示，除非你手动改写这个 POU 加入 `AppGenerateException`（需要 CmpApp/SysExcept/SysTypes2_Itf 库）来变成显式异常。
  URL: https://content.helpme-codesys.com/en/CODESYS%20Development%20System/_cds_obj_pou_checkbounds.html （官方 helpme-codesys，已核对原文）
  URL: https://help.plc.abb.com/AB270_en/_cds_obj_pou_checkbounds.html （ABB AC500 Automation Builder 官方文档，已核对原文，内容与 CODESYS 通用文档一致——AC500 基于标准 CODESYS 内核）

- **仅对"变量下标"生效**（摘要级）：只有当数组下标是运行时变量时才触发 CheckBounds；下标是编译期常量且越界，是**编译错误**，不会跑到运行时。
  URL: https://industrialmonitordirect.com/blogs/knowledgebase/codesys-checkbounds-not-working-with-variable-length-arrays （摘要级）

- **生产环境性能警告**（摘要级）：隐式检查函数会显著增加 CPU 负载，文档暗示不建议在生产系统长期开启／或需要评估开销；这类检查更多是开发调试辅助，不能替代显式防御式编程。
  URL: 同上（industrialmonitordirect 摘要）

- **完整隐式检查函数清单**（已核对原文，来自 CODESYS 官方 POU 列表页）：
  | 函数 | 检查内容 |
  |---|---|
  | CheckBounds | 数组下标越界 |
  | CheckDivInt / CheckDivLInt / CheckDivReal / CheckDivLReal | 除零 |
  | CheckRangeSigned / CheckRangeUnsigned | 子范围类型（如 INT(0..100)）越界 |
  | CheckLRangeSigned / CheckLRangeUnsigned | LINT/ULINT 子范围越界 |
  | CheckPointer | 指针地址与类型对齐合法性 |

  URL: https://content.helpme-codesys.com/en/CODESYS%20Development%20System/_cds_f_obj_pous_implicit_check.html （已核对原文）

- 关于 **IEC 61131-3 标准本身**是否规定越界为 undefined behavior：**未找到**权威一手条文明确说明——搜索只找到 Beckhoff TwinCAT 扩展的 `__TRY/__CATCH` + `RTSEXCPT_ARRAYBOUNDS` 异常机制（这是厂商扩展，非标准强制），说明具体行为是**实现相关（implementation-defined）**，标准本身未强制规定统一行为。
  URL: https://stefanhenneken.net/2019/07/29/iec-61131-3-exception-handling-with-__try-__catch/ （摘要级，第三方博客非官方标准文本）
  结论：**标准层面未找到**明文规定；各厂商（CODESYS 默认钳位、Beckhoff 提供可选异常）行为不同——这点对你的移植很关键，AC500 遵循 CODESYS 默认钳位语义。

## 2. 工业界错误接口惯例：错误码 vs bError+sErrorMsg vs ENO

- **PLCopen 标准 FB 模式**（摘要级，多个厂商文档交叉印证 Beckhoff/Kollmorgen/Siemens/OPC Foundation）：PLCopen Motion Control 等标准库的**主流错误接口形状是 `Error : BOOL` + `ErrorID : DWORD`**（4 字节，非字符串）。ErrorID 第 29 位区分 OPC Foundation 定义的错误码 vs 厂商/PLCopen 自定义错误码。**没有找到**官方 PLCopen 规范强制要求 `sErrorMsg` 字符串输出——这是部分厂商（如 Beckhoff Tc2_Standard 库的一些 FB）自行增加的可读性扩展，不是 PLCopen 强制项。
  URL: https://reference.opcfoundation.org/PLCopen-CFB/v102/docs/6 （摘要级，OPC Foundation 官方参考文档目录页）
  URL: https://webhelp.kollmorgen.com/KAS/Content/3.UnderstandKAS/PLCopen-General-Rules/PLCopen-Function-Block-ErrorIDs.htm （摘要级）
  URL: https://support.industry.siemens.com/cs/attachments/37865127/PLCopen_Blocks_V2_0_en.pdf （西门子官方 PDF，未逐页深入核对，摘要级）

- **ENO（Enable Output）**：IEC 61131-3 标准定义的 EN/ENO 是最原始/最基础的错误传播机制（FB 执行失败时 ENO=FALSE），但**颗粒度太粗**（只有布尔值，无错误原因/错误码），工业实践中很少单独依赖 ENO 做详细错误处理，通常搭配自定义 Error/ErrorID 输出。此点为**背景常识性判断，非本次直接检索到的一手条文**——建议如需引用，另行核实 IEC 61131-3 正式 EN/ENO 条款。

- **推荐形状**（给你的判断，非检索结果）：对你的批量分配算法，若要对齐工业惯例，最常见、最"PLCopen 风格"的是 `Error: BOOL` + `ErrorID: WORD/DWORD`（用枚举定义错误码含义），`sErrorMsg` 作为可选的调试友好扩展、不是主流强制项。

## 3. 除零 / 溢出行为

- **REAL/LREAL 除零**（摘要级但与 IEEE 754 常识吻合，可信度较高）：浮点数除零**不抛异常**，遵循 IEEE 754：非零数除以0 → INF；0除以0 → NaN。CODESYS 文档原话："floating point types don't raise an exception, but return the values Infinity or NaN as appropriate"，同时明确"Division by zero may have different results depending on the target system"——即**具体行为跟目标运行时/PLC 型号相关，不是标准统一保证**。
  URL: https://content.helpme-codesys.com/en/CODESYS%20Development%20System/_cds_operator_div.html （已核对原文，但页面本身未给出逐字数值细节，需结合 CheckDivReal 独立页）

- **INT/DINT 整数除零**（摘要级+已核对交叉）：由 `CheckDivInt`/`CheckDivLInt`/`CheckDivReal`/`CheckDivLReal` 隐式函数监控，具体默认返回值（是否是 0、还是保留原分子、还是触发异常）**未在本次抓取的页面中找到逐字数值**——这几个 POU 各自独立页面本次未单独 WebFetch，只能确认"存在监控机制"这一事实，**未找到**它们各自的默认返回值规格，建议后续对 `_cds_obj_pou_checkdivint.html` 等具体页面单独抓取核实。

- **整数溢出/wraparound**（摘要级，通用背景知识非 CODESYS 官方逐字确认）：INT 范围 -32768~32767，超出范围时是否 wraparound（如 32767+1 = -32768）还是被 CheckRangeSigned 钳位，**取决于变量是否声明为子范围类型（subrange type）**——普通 INT 类型运算溢出行为在本次检索中**未找到 CODESYS 官方逐字确认**其是否 wraparound 还是 saturate；只有当变量声明为子范围类型（如 `INT(0..100)`）时才会触发 CheckRangeSigned 钳位。此为你 400 库位场景重要边界点，**建议后续针对这一点专项二次核实**（例如查 IEC 61131-3标准算术运算符溢出条款或 CODESYS 编译器行为测试）。

## 5. 公开 PLC 边界测试框架/用例集组织方式

- **CODESYS Test Manager**（官方商业插件，已核对部分原文/摘要级混合）：测试用例是遵循固定接口约定的 IEC 函数块（"test case is a function block which follows a specific scheme"），可用测试表格（Test Tables）自动生成，也支持"IEC Unit Test"风格自由编程测试步骤。**未找到**其边界测试专门命名规范的逐字文档细节，官方 Data Sheet PDF 本次未逐页深入抓取。
  URL: https://store.codesys.com/media/n98_media_assets/files/2101000003-F/4/CODESYS%20Test%20Manager_en.pdf （官方 Data Sheet，未深入抓取，仅列出存在此产品）
  URL: https://content.helpme-codesys.com/en/CODESYS%20Test%20Manager/_tm_basic_function_iec_unittest.html （官方文档，未深入抓取正文）

- **co♻e (counit) 开源单元测试框架**（已核对原文，WebFetch 直取）：这是**免费开源**、社区维护的 CODESYS 单元测试框架，组织方式更明确、更适合你直接参考：
  - 每个功能块（FB）在独立的 "Test" 文件夹下配一个对应的 `_Test` 后缀测试套件（如 `FB_Sum` → `FB_Sum_Test`）
  - 每个测试用例是一个独立命名的"方法"，官方明确**不强制命名规范**（"the framework in no way enforces to use any standard for the naming"），但推荐用描述性名字让测试名本身即文档，例如 `TwoPlusTwoEqualsFour`、`ZeroPlusZeroEqualsZero`
  - 零值等边界情况通过普通描述性命名覆盖，无专门的"边界测试"标记体系
  URL: https://forge.codesys.com/lib/counit/tutorial/Tutorial/ （官方项目 Forge 页面，已核对原文）

  **对你 40+ 边界用例的可迁移建议**（非检索结果，基于以上事实的判断）：可借鉴 co♻e 的"`_Test` FB + 描述性方法名"结构，按你的算法输入维度（库位数=0/1/400/401，货物数=0/1/199/200/201，评分并列，容量耗尽等）逐条建测试方法，命名用 `场景_预期结果` 模式（类似 `ZeroPlusZeroEqualsZero`），比自造命名体系更贴合社区惯例。
