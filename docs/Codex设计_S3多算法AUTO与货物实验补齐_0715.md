## 1. 最终目标

S3 不是单件搬运动画，而是一张可交互、可复核的工业作业台：

1. 右侧用唯一固定镜头完整展示 20 列 × 20 层单巷道仓储系统、入口队列、堆垛机、货叉、货物、出口扫码交接和退出过程。
2. 左侧在同一画面同步展示当前作业、算法选择依据、实时轴位、队列、相位、候选评分与多 seed 统计，不另设“过程数据页”。
3. 四条主策略链可在同一场景、同一 seed、同一逻辑周期与相位检查点直接切换比较。
4. `AUTO` 使用现有完整规则路由器；默认安全词典序自动选择 AWRA / CB / TOB，且保留人工覆盖。
5. Tier 1 / 2 / 3 与 uniform / skew / heavy 可切换，用于同一算法跨拟真档、跨货物画像比较。
6. 新增风险完备的货物实验，检验相关性、密度、阈值、联合压力、漂移和 AUTO regret；不宣称穷举连续参数空间。

### 非目标

- 不把 S3 称为 20 × 20 PLC/Factory I/O 物理闭环。
- 不称 AUTO 为机器学习、在线学习或自学习。
- 不在右侧 3D 视口叠加图表、算法按钮或第二摄像头。
- 不为展示效果改写 canonical 运动学、容量规则或已有实验数字。
- 不把候选位评分分量当成系统级 KPI。

## 2. 不可改真相与可调画面参数

### 2.1 不可改真相

| 维度 | 锁定值 |
|---|---|
| 坐标 | X=列，Z=层高，Y=纵深；`camera.up=(0,0,1)` |
| 仓储结构 | 单巷道、单排 20 × 20 货位；I/O 位于左下角外侧 |
| 预占规则 | `(列 + 层) mod 3 = 0`，共 133 个预占禁用位 |
| 物理尺寸 | 单格 1 m × 1 m × 1 m；货物落在合法格中心 |
| 水平轴 | `VX=2.0 m/s`，`AX=0.5 m/s²` |
| 垂直轴 | `VZ=0.5 m/s`，`AZ=0.3 m/s²` |
| 双轴运动 | 同时起动、独立梯形积分，快轴先到后钳位，整程由慢轴决定 |
| 机构约束 | 桅杆 X 恒等于载货台/货叉 X；货叉只沿 Y 伸缩 |
| 轨迹 | 由逐轴 `axis_pos` 同时采样得到，不用贝塞尔/样条伪圆滑 |
| 口径 | 新页面、实验、提示词与统计全部标 SIM；显示时间与 sim 时间分离 |

### 2.2 可调画面参数

以下可为构图和识别度调整，但必须通过截图门：相机焦点/半径/方位角、灯位和强度、面板宽度、字号、材质粗糙度、货物饱和度、地面范围、标签偏移、轨迹管径和动画时间压缩率。

## 3. 信息架构（已锁定方案 A）

### 3.1 整体布局

- 左栏占可用宽度 31%~34%，右侧 3D 占 66%~69%。
- 页面只有一个工作视图，不存在“三维作业 / 过程数据”两页切换。
- 左栏纵向填满，任何 720p 截图不得出现无信息的大块留白；次要解释收入不超过两层的 `<details>`。
- 右侧只保留单镜头 3D、清晰的目标 DOM 标签和底部一句 SIM 来源说明。

### 3.2 左栏从上到下

1. **运行头**：当前算法模式、Tier、货物画像、seed、周期号、SIM 标识。
2. **一级选择器**：`AUTO / SEQ / NEAR / SCORE / AWRA-LS` 五个常驻按钮。
3. **二级选择器**：Tier 1/2/3 常驻；货物画像 uniform/skew/heavy 用紧凑选择器。
4. **AUTO 路由卡**：画像 `n/density/skew/w_mean/w_heavy`、目标口径、选择结果、阈值理由、备选策略、人工覆盖状态。
5. **当前作业卡**：同一 gid、入库位、出库位、货物重量档、当前所有者、七步作用说明。
6. **实时轴位**：X/Z/F 三轴数值与速度；不得再放静态伪数据。
7. **货位规则图**：20 × 20 小图，必须能辨认 133 个斜纹预占位、库存、目标、设备和随载货物。
8. **过程与对比**：七相位紧凑轨道、入口队列、四算法同场景 P95 迷你比较；不另开数据页。
9. **系统指标**：P50/P95、设备利用率、fail/viol、单位货物能耗代理。hover/focus 展示定义、样本、公式、方向与来源。

### 3.3 评分与系统指标分层

- `efficiency / stability / energy / reserve` 只能标题为“候选位评分贡献”。
- tooltip 必须显示原始输入、归一化分母、权重、贡献值和“总分越低越优”。
- `energy` 明标 `Σw·h` 代理或其单位货物版本，不写 kWh，除非完整换算假设同时可见。
- 系统 P95、利用率等放在独立“多 seed 结果”区域，不与候选贡献同色同层。

## 4. 算法模式与切换协议

### 4.1 四条手动主策略链

| UI 名称 | 初始布局 | 后续在线入库 |
|---|---|---|
| SEQ | seq | seq |
| NEAR | near | near |
| SCORE | score | score |
| AWRA-LS | AWRA-LS | score |

四条链沿用 `sim/mixed_ops.py` 和 `sim/realism.py` 的既有定义。

### 4.2 AUTO 完整路由

- 默认目标：**安全词典序**。离线标定顺序为 `零额外失败 → exp_t（期望取货时间）CI 并列 → 重货平均层高`；运行时只执行由该标定蒸馏出的固定阈值，不在线重算 CI。这里不把动态系统响应 P95 偷换进 v1 路由规则。
- S3 与本轮新增实验必须显式传入 `objective="lexicographic"`，不得依赖 `strategy_lib.select_strategy()` 当前仍为 holistic 的函数默认值。是否修改全项目默认值超出本规格范围，需另做影响审查，不能为 S3 顺手改动。
- 已知批次：现有规则路由可选择 AWRA / CB / TOB；CB/TOB 是 AUTO 可见结果，不新增为常驻手动按钮。
- 未知批次：现有规则在 near / score 内降级选择。
- 综合口径与纯吞吐口径收入高级设置，不占用首屏；切换后必须显示目标口径已改变。
- `AUTO` 是确定性、可解释的阈值路由；UI 用语固定为“SIM 场景路由（规则阈值）”。
- 当前代码虽计算 `w_mean/w_heavy`，选择分支尚未实际使用重量特征。实施前不得写“重货驱动选路”；是否增加重量分支由 §7 的预注册证据门决定。

### 4.3 CB/TOB 的动态链定义

CB、TOB 为批量初排策略；进入混合流后统一使用 score 做逐件回库，形成 `CB→score`、`TOB→score`。这样与 `AWRA-LS→score` 的批量/在线职责分离一致，不伪造不存在的逐件 CB/TOB。

### 4.4 公平切换

切换算法时保持：

- 同一场景、seed、逻辑 cycle、phase kind、归一化 phase progress；
- 同一货物/请求序列与同一外生 arrivals；
- 加载目标策略在该检查点的库存快照，不把上一策略的物理库存偷渡过去；
- 用 200~300 ms 的“对比检查点重载”遮罩解释库存布局变化，不重置到周期 0。

故障按共同底层随机流生成，但不同策略的作业时长不同，故障不承诺落在同一 cycle；UI 不用绝对 sim 时间强行对齐故障事件。

## 5. 货物与出库所有权状态机

每个 gid 在任何时刻只能有一个所有者：

`QUEUE → CARRIER_IN → RACK → CARRIER_OUT → OUTFEED → SCANNED → EXIT_MASK → CONSUMED`

硬约束：

1. 入库队列、载货台、货架、出库线不得同时出现同一 gid。
2. 取货时先从库存快照移除，再绑定到货叉/载货台；不可在“货架和载货台”双重出现。
3. 回到 I/O 后，货物从载货台转移到短出口输送机，载货台才为空。
4. 货物通过扫码门后状态变为 `SCANNED`，继续沿短线前进；通过红色遮挡退出帘后才进入 `CONSUMED` 并隐藏。
5. 下一件入库货物不得替换仍在载货台或出口线上的上一 gid。
6. 左栏“当前所有者”和 3D 位置必须同源；自动测试逐帧检查所有权唯一性。

## 6. 单镜头、场景与美术

### 6.1 镜头硬门

- 固定一个主镜头，无自动轨道、无第二视角、无过程特写切镜。
- 正常 UI 不开放 OrbitControls；仅 `?debug=1` 可用于开发校准。
- 货架、I/O、堆垛机、出口短线、扫码门和退出帘必须同时在画面内。
- 1280×720、1366×768、1600×900、1920×1080 四种视口逐一截图。
- 3D 主体横向占右视口 75%~92%；左右安全边各 ≥3%；货架顶部距上边 4%~12%；地面/出口不得被底边裁切。
- 任何验收截图中，右侧大面积空白、货架裁切、I/O 消失或标签出界均为阻断。

### 6.2 场景几何

- 货架与预占封板作为纯竖直背景平面，取消无功能的前突装饰和会遮挡货物的厚边。
- 地板在货架与作业区两侧合理外延，1 m 地缝与货位/导轨方向一致；不得出现左右无缘由的不对称截断。
- 堆垛机至少包含行走底座、贯通桅杆、升降载台、双货叉、导轨和必要连接件，避免抽象成一根杆。
- 出口短线、扫码门、退出帘属于背景设备，但必须能看清 gid 完成交接。

### 6.3 色彩与照明

- 货架/地面继续使用正式版低饱和中性灰。
- 货物重量档保持蓝系语义，但提高与货架的明度/饱和度差；活动货物增加琥珀轮廓和轻微 emissive，不改变其重量档底色。
- AB 状态色只用于状态：占用绿、推荐黄、路径蓝、报警红、查询青、检修紫；不把状态色误作重量类别。
- 红色只用于 ABB 品牌细节、报警与退出遮挡帘，不作普通强调。
- 使用主方向光、相机补光和低强度环境补光；全货架阴影视锥更新投影矩阵，避免货物与柜体融成一片。

### 6.4 标签

- 目标位、随载货物、扫码完成全部使用 DOM overlay，不使用模糊 WebGL 文字贴图。
- 目标标签最小 15 px、700 字重、黄色底黑字，自动夹在右视口安全区内。
- 标签必须有清晰指向线，不能遮住载货台、目标格或出口交接点。

## 7. 新增货物实验：预注册设计

“覆盖所有情况”定义为覆盖已知风险维度、决策阈值和关键交互，不宣称穷举连续参数空间。

### 7.1 复用现有证据

- 13 静态场景 × 30 seeds × 7 策略：`sim/instance_gen.py`。
- 三档动态混合流：`sim/realism.py`。
- G2/G3/G4 七点 × 30 seeds × 四策略：`sim/scan_g2g4.py`。
- 30-seed untouched 静态泛化：`sim/final_test_30seed.py`。
- 权重网格：`sim/adaptive_weights.py`，仅作为 2-seed 设计表，不能升级为在线学习。
- 生命周期与重排只作旁证，不能嫁接成当前 Tier 3 的同口径主结论。

### 7.2 S0：确定性路由边界测试

先做不含随机仿真的直接画像断言，再用真实 `batch_profile()` 做可达边界回环：

- `n=19/20/21`；
- 直接谓词：`density=0.249/0.250/0.251`、`0.699/0.700/0.701`；
- 生产可达值：`n=65/66/67`（跨 `density<=0.25`）与 `n=186/187/188`（跨 `density>=0.70`），逐一断言三位小数 round-trip；
- `skew=0.349/0.350/0.351`；
- batch_known TRUE/FALSE；
- holistic/lexicographic/speed 三目标。

同时做“未使用特征”变形测试：只改变 `w_mean/w_heavy` 时，当前 v1 选择结果应保持不变，防止 UI 宣称不存在的重量选路。

### 7.3 S1：18 场景动态筛查

每格 10 seeds、100 个混合流周期，比较 7 固定策略链 + AUTO：

| 编号 | 场景 | 主要风险 |
|---|---|---|
| E01 | 120 uniform | 无画像信息基准 |
| E02 | 120 skew | 当前主基准 |
| E03 | 120 heavy | 重货基准 |
| E04 | 20 tiny-light | 极小批与 near 分支 |
| E05 | 60 light-sparse | 低密度轻货 |
| E06 | 120 flat-info | 频次项失效 |
| E07 | 60 heavy-sparse | 低密度重货与安全并列键 |
| E08 | 200 dense-skew | 约 75% 密度、失败门 |
| E09 | 240 dense-skew | 极限密度、失败门 |
| E10 | 120 hot-heavy | 频次与重量正相关 |
| E11 | 120 hot-light | 高频轻货、低频重货 |
| E12 | 120 bulky-heavy | 重量与体积正相关 |
| E13 | 120 near-limit | 重量/体积接近但不越合法边界 |
| E14 | 120 illegal-mix | 3 超重 + 3 超容积及动态恢复 |
| E15 | 200 skew + ρ=0.90 | **扩展 Tier 3 压力情景**；非 canonical G2、非 H |
| E16 | 120 skew + σ=0.45 | 高频画像噪声 |
| E17 | 120 heavy + MTBF=1500 | 重货与严苛故障 |
| E18 | 99-cycle uniform→skew→heavy | 三个批次边界的分布漂移 |

固定策略链定义：`random→random / seq→seq / near→near / score→score / awra→score / cb→score / tob→score`。AUTO 只在每个独立 run 的初始批次选择一次，运行中不无成本重排既有库存。

E15 的 `extended_pressure=true`、`canonical_g2=false`、`headline_eligible=false` 必须同时写入场景注册表和 CSV；UI tooltip/导出标题固定显示“扩展 Tier 3 压力情景（非 canonical G2、非 H）”，不得只靠说明文档兜底。

相关性场景不能只改标签。每个 seed 先按既有边际分布生成重量、频次、体积三个多重集，再做确定性 rank coupling：

- E10 要求 Spearman `ρ(weight,freq) ≥ +0.75`；
- E11 要求 Spearman `ρ(weight,freq) ≤ -0.75`；
- E12 要求 Spearman `ρ(weight,volume) ≥ +0.75`，频次边际保持基准；
- 重排前后每个属性的边际多重集必须逐值一致；实测相关系数、目标和容差写入输出；未达阈值则该 seed 拒绝进入聚合。

所有场景写入版本化注册表 `sim/scenarios/cargo_dynamic_v1.json`，并锁定：

- S1 探索 seeds=`2026..2035`；S2 独立确认 seeds=`31001..31030`；S3 长跑 seeds=`41001..41010`；若触发重量感知 v2，untouched seeds=`51001..51030`；四段互不重叠。
- 每个场景必须分别声明 `initial_goods_generator`、`inbound_goods_generator`、`request_pool_policy`、`invalid_inbound_policy`；不得只给初始库存换画像。
- E01/E02/E03 的初始货和整批入库货分别调用 `ws.gen_goods(n,seed,case)` 与同 case 的版本化 inbound generator；E04~E09 使用与 `instance_gen.SCENARIOS` 相同的 light/heavy/flat/dense 参数生成初始货及整批入库货；源文件和注册表 hash 进入 manifest。
- E10~E12 从同 seed 的既有边际多重集出发，按 `(属性值,gid)` 稳定排序，相关性重排 RNG=`seed+101`，并记录实际 rho。
- E10~E12 的 100 件 inbound batch 单独执行同一 rank-coupling 规则，不能把相关初始库存接上默认 skew 新货。
- E13 的初始货与入库货均固定 `weight~U[90,99.5]kg`、`volume~U[0.90,0.99]m³`、频次沿现有 Pareto，属性抽样顺序与 RNG 偏移写入注册表。
- E14 初始库存固定为 120 件合法 skew 货；100 件 inbound 中按 `seed+31` 稳定选 6 个索引，交替改为超重/超容积。非法货在入口校验即拒收、不进请求池；该周期不派发配对出库，记 `REJECT_NO_DISPATCH`，下一周期继续。其余 94 个有效周期仍严格一入一出，终局库存保持 120。所有策略共享相同非法索引与属性。
- workload 生成器在保持 `mixed_ops.build_workload()` 默认行为兼容的前提下参数化：入库批次基础 RNG=`seed+7`、请求 RNG=`seed+13`；每个有效周期从当前合法库存池按 `freq` 加权抽一件、移除后再把当期合法 inbound 加入池。请求只依赖 gid/频次，不看仓位；请求权重、到达参数和函数 hash 进入注册表/manifest。
- E18 初始库存=120 uniform；共 99 周期，0~32 的 inbound=uniform、33~65=skew、66~98=heavy。请求池始终是当前合法在库货物，按各货物实际 `freq` 加权演化，不强行按段标签抽货。AUTO 初始选择保持到 run 结束；在周期 33/66 前分别对当时真实库存画像计算并记录“下一独立批次建议”，不立即生效、不重排库存。E18 衡量 stale-route 鲁棒性，不宣称在线改策闭环。

### 7.4 公共随机数与公平性

同一 `(scenario, seed)` 的八个模式必须共享：

- 初始 goods 身份与属性；
- 新货和出库请求序列；
- arrivals 基础流；
- 画像噪声基础随机数；
- 故障基础随机流：先用固定 RNG 预生成足以覆盖最长 horizon 的“忙时故障间隔”序列，策略只按 busy-time 游标消费，不在条件分支中重新抽 RNG。

输出 `workload_hash / arrivals_hash / noise_hash / fault_stream_hash`；任一策略间 hash 不一致则整格实验作废。策略时长不同导致事件实际落点不同是正常现象，不能通过重排事件制造绝对时刻相同。

### 7.5 指标

主指标固定显示全称：**Tier 3 系统响应 P95（含排队/装卸/故障）**，不得简称 C3“期望取货时间”。每个 seed 内沿现有共享引擎约定：先升序排列响应数组，P50=`resp[n//2]`，P95=`resp[min(n-1,int(n*0.95))]`；跨 seed 报“seed 级 P95”的均值和 t-95%CI。  
护栏：正常场景 `fail=0 && viol=0`；illegal-mix 用 `expected_reject=6 && excess_fail=0 && viol=0`。  
次指标：响应 P50、平均取货行程、设备利用率、队列峰值、排空时间、单位货物能耗代理、重货平均层高。  
AUTO 专项：picked strategy、选择稳定率、相对同场景最佳固定策略 regret、策略切换次数、阈值理由一致性。

E14 的 6 个 `REJECT_NO_DISPATCH` 周期不进入响应 P50/P95、利用率或双命令时间分母；CSV 固定输出 `scheduled_cycles=100 / valid_cycles=94 / rejected_cycles=6 / response_sample_n=94 / reject_service_s=0(modelled)`。拒收耗时尚无 canonical 参数，因此记 0 且单列，不把它伪装成真实门检节拍。

`regret` 预注册为两层，禁止混报：

1. **静态路由 attainment**：沿用现有 oracle-gap 的 exp_t 词典序口径，仅用于验证路由器标定，不替换成动态 P95。
2. **动态运行 regret**：同一 `(scenario,seed)` 内先剔除 `excess_fail>0` 或 `viol>0` 的固定策略，以 Tier 3 系统响应 P95 最小者作事后 oracle；报告 `P95_AUTO-P95_oracle`、配对 95% CI、可比 seed 数和 AUTO/固定策略失败差。AUTO 自身不可行时记 `INFEASIBLE`，regret=`NA`，不得进入均值；若该 seed 无任何可行固定策略，则 oracle/regret=`NA`，单列不可行率。它是诊断上界，不宣称路由器事先知道事后 oracle。

另做 untouched 评估：S1/S2 用于筛查与确认，路由规则不因结果临时改写；若 §7.8 触发 v2 提案，必须在未参与拟合的独立 seeds 上单列 regret，不能复用训练 seeds。

### 7.6 S2：独立 30-seed 确认

S1 只作 `2026..2035` 探索。按以下预注册规则选 6 格后，在完全独立的 `31001..31030` 上重新运行 30 seeds；S1 与 S2 不合并、不把最差格筛选后的同一 seeds 称为独立确认：

1. AUTO regret 最大格；
2. 响应 P95 退化最大格；
3. 任一非预期 fail/viol 格；
4. AUTO 发生策略改变的两个不同画像格；
5. 属性相关性对 winner 影响最大的格；
6. 若以上重复，则按 E 编号从小到大补足。

报告均值、标准差、95% CI；同 seed 比较用配对差与配对 CI。

### 7.7 S3：长跑

选择 S2 中 regret、P95、队列峰值最坏的三个不同场景，用独立 `41001..41010` 各 10 seeds × 1000 周期，验证无状态泄漏、库存计数平衡、队列可排空和 gid 所有权唯一。

### 7.8 重量感知路由的证据门

只有当 E10/E11/E12 在 30 seeds 下满足以下全部条件，才允许提出 v2 重量分支：

1. 与同密度/同 skew 对照相比，词典序 winner 因重量相关性发生稳定改变；
2. 正常货物 `fail=0 && viol=0`；
3. 配对 CI 不跨 0，且重货层高安全键有一致改善；
4. 新规则在未参与标定的 untouched seeds 上不增加 regret。

否则保持 v1 路由，`w_mean/w_heavy` 只作画像展示，不声称参与选择。

## 8. S3 trace 包

### 8.1 展示组合

生成 `Tier 1/2/3 × uniform/skew/heavy × 5 UI 模式` 共 45 个可切换 trace。AUTO trace 内部记录实际 picked strategy，可能为 CB/TOB。

实验 E04~E18 的聚合结果进入左栏比较/高级抽屉，不全部制作 3D 录像，避免把数千个 run 伪装成逐一可视化。

### 8.2 trace 必备字段

- schema_version、source、sim_only、seed、tier、profile、strategy_mode；
- router profile/objective/picked/alt/reason/thresholds；
- workload/arrivals/noise/fault hashes；
- 初始库存、逐周期前后库存快照、队列、所有权事件；
- 每阶段 t0/t1/from/to/u0/u1/cargo owner；
- 统计定义、样本量、seed 范围与源 CSV；
- 生成命令、Python 版本、关键依赖版本、事件引擎/策略/场景定义文件 SHA256；
- trace 内保存 canonical payload hash（计算时排除 hash 字段）；完整 JS/JSON 文件 SHA256 只写入外部 manifest，避免自指；
- 禁止在 JS 中硬编码与 trace 不一致的 KPI。

### 8.3 文件组织

优先生成一个索引 + 按组合拆分的小型 JS，按需加载；审计 JSON 可另存但不随页面预载。file:// 使用动态本地 `<script>` 注册表，不走 fetch。不得引入外网依赖。

- 单个浏览器 trace 上限 800 KB，索引上限 100 KB；超限先移除重复快照，不降低数值精度。
- 完整性测试逐一核对 45 个 manifest 条目、文件存在性、payload hash 和 schema 版本。
- 运行时最多缓存 3 个 trace；淘汰时释放库存 mesh、路径 geometry、数组引用和全局注册项。
- 加载失败必须在左栏显示具体组合和可重试按钮，禁止右侧静默空白。

## 9. 验收门

### 9.1 Python / 数据

- selector 边界测试全绿；
- 45 trace schema、所有权、库存计数、阶段连续性全绿；
- 每个公平比较格四个 hash 一致；
- 正常场景无额外 fail/viol，非法货按预期拒收且系统继续；
- AUTO pick 与 `strategy_lib.select_strategy` 逐格一致；
- 多 seed 聚合数字从 CSV 读取，页面无手抄漂移；
- 负结果、不可行格和不稳定选择如实显示。
- exporter 不得另写一套 Tier 3 业务循环；实验与 trace 必须消费同一个事件引擎。若暂时无法抽取共用引擎，则逐 job 对账 arrivals、slot、phase、fault、所有权事件、库存终态和 artifact hash，任一不一致即阻断。

### 9.2 JavaScript / 交互

- JS 语法检查通过，浏览器 console error=0；
- 五模式、三 Tier、三画像均可切换；
- 算法切换保持逻辑检查点，不回到开头；
- hover 与键盘 focus 都能读到指标定义；
- 暂停、重播和循环不会重复 gid 或泄漏 mesh；
- file:// 与本地 HTTP 两种打开方式均工作。
- 删除或彻底移出运行路径的旧内联动画状态机；最终页面只能有一个 requestAnimationFrame 驱动器、一个暂停源和一个 QA 状态源，禁止继续用全局开关压制旧循环。

### 9.3 3D 物理与过程

- 桅杆/载台 X 始终一致；货叉伸缩不带回托板或让货物掉落；
- 等待货物、活动货物与 trace gid 一致；
- 七步全过程连续：装入→入库→放货→联程→取货→回口→扫码卸出；
- 出库货物在退出帘之后才消失；下一件货物不提前替换；
- 133 个预占位在货架与规则图上均肉眼可辨；
- 目标标签清晰、指向正确、不模糊、不遮挡机构。

### 9.4 浏览器截图

每种视口至少检查：初始算法证据、入口装载、远端高层、放货、取货、回口、扫码、退出、算法切换、画像切换。故障帧只对实际含故障的 trace 检查；另从 S1 中按固定规则选“最小 seed 且含故障”的专用 showcase trace，标 `SIM 故障展示 / 不进入性能统计`。普通 trace 无故障时验收项记 NA，不伪造事件。

自动几何断言只作第一层；最终必须保存真实截图并由主 agent 与独立 critic 分别目检。任何一方判定主体裁切、空白失衡、标签不可读、货物错位或所有权穿帮，即阻断提交。

每张验收截图的 manifest 必须记录：候选 HTML/JS/trace SHA256、浏览器与版本、视口、DPR、算法/Tier/画像/seed/cycle/phase/progress、抓取时间和人工结论。旧 hash 对应的截图不得验收新代码。

## 10. 实施顺序

1. 先把 `realism` 与 exporter 收敛到一个共享事件引擎，再固化数据 schema、策略链和公平性测试；
2. 再增补实验生成器并跑 smoke；
3. 监督 agent 审查实验设计和 smoke 结果；
4. 跑 S1，按预注册规则选 S2/S3；
5. 生成 45 trace 并做数据级验收；
6. 重构左栏与交互，不先动 3D 美术；
7. 实现货物所有权、出口交接和切换检查点；
8. 最后校准镜头、灯光、颜色、标签与构图；
9. 多视口截图 + 独立 critic 复核；
10. 用户确认候选后，才讨论覆盖 `样张_S3定稿.html` 和视频。
