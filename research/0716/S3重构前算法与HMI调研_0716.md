## 0. 结论

1. 当前 S3 候选版的运动连续性、碰撞限定域、视口适配和性能自动 QA 仍然成立；但独立反向审查对“展示可发布性”给出 **NO-GO**。两者不矛盾：前者证明页面按自己的契约稳定运行，后者指出契约把 20×20 逻辑 SIM、54 格 Factory I/O 物理验证和未证实的扫码/离场状态混成了一个可信物理场景。
2. 下一版不能以“把蓝色粗线换成灰色设备”为主任务。必须先决定视觉语义：
   - 20×20 是算法逻辑仓位 SIM；
   - Factory I/O 是 54 格物理 I/O 与时序证据；
   - 两者可以同屏关联，但不能伪装成同一几何孪生。
3. AWRA-LS 与 SCORE 使用同一个货位评分函数。AWRA-LS 的真实增量是批次优先级排序、初始分配后最多 5 轮局部重定位/交换；动态逐件阶段回落为在线 SCORE。不能宣称 AWRA-LS 有另一套更高级评分公式或全局最优保证。
4. 当前有逐候选 `score` 的运行时计算，但没有把候选总分、四项分值、加权贡献或排名导出到 trace/CSV。新版若要显示“为什么选这个格”，必须先改仿真证据导出，前端不能临时重算或伪造。
5. AUTO 是启动时一次规则选路，不是在线学习、逐货切换或重量感知；当前 S3 九条 AUTO 代表回放全部选中 `TOB → SCORE`。
6. 三种入口的推荐结构是：
   - 主入口：预设情景 E01–E18；
   - 次入口：同一情景内的规则推荐与 8 算法对照；
   - 独立沙盒：用户输入单件货物，只展示合法性和单件决策，不继承统计 KPI。
7. 当前三档 Tier 不是“推荐档位”，而是拟真假设层。算法推荐与拟真层必须拆开。

## 2. 算法真相源

### 2.1 可行候选集与运动时间

对货物 `g`，候选货位必须同时满足：货位为空、层限重允许、体积不超过 1.0 m³；现行预占规则 `(col+tier)%3==0` 的格位不进入空位集。[warehouse_sim.py:118](F:/abb_wh_work/sim/warehouse_sim.py:118) [warehouse_sim.py:181](F:/abb_wh_work/sim/warehouse_sim.py:181)

运动时间有两档实现：

- 匀速：`max(|Δx|/VX, |Δy|/VY)`；
- 加减速：X、Z 轴分别使用三角/梯形速度曲线，再取并行两轴的较大时间。

基准参数为 `VX=2.0 m/s`、`VY=0.5 m/s`、`AX=0.5 m/s²`、`AY=0.3 m/s²`。[warehouse_sim.py:40](F:/abb_wh_work/sim/warehouse_sim.py:40)

### 2.2 SCORE 与 AWRA-LS 共用的评分

候选位总分为：

`S = α·h_eff + β·h_stab + γ·h_energy + δ·h_resv`

其中：

- `h_eff = (f/f_max)·(T/T_max)`：高频货物放得越远，惩罚越大；
- `h_stab = (w/w_max)·(tier/19)`：重货放得越高，惩罚越大；
- `h_energy = w·tier·CELL_H/(w_max·H_MAX)`；
- `h_resv = (1-f/f_max)·(1-T/T_max)`：保护近端资源的预留项。

实现见 [warehouse_sim.py:248](F:/abb_wh_work/sim/warehouse_sim.py:248)。平局按 `(score, travel_time, tier, col)` 继续裁决。

两个必须披露的负结果：

1. `H_MAX=19·CELL_H`，所以当前 `h_energy == h_stab`。β、γ 虽然名称不同，数值信号完全相同，实际效果相当于 `(β+γ)·同一项`。[warehouse_sim.py:48](F:/abb_wh_work/sim/warehouse_sim.py:48) [warehouse_sim.py:262](F:/abb_wh_work/sim/warehouse_sim.py:262)
2. 归一化没有 clamp。动态阶段的分母冻结为初始批次 `f_max/w_max`，噪声后的 `freq_est` 或后续货物可能使分项超过 1；当前 `t3_skew_score.js` 已出现 `freq_est/f_max≈3.674`。

因此，下一版不能把四项画成四个互相独立且恒在 0–1 的仪表。若要继续保留四项概念，需先决定是：如实显示当前实现、修正公式、还是只显示加权表达式与限制说明。

### 2.3 八种模式的真实差异

| 模式 | 决策方式 | 动态阶段 | 可展示边界 |
|---|---|---|---|
| RANDOM | 在可行位中按 seed 随机抽取 | RANDOM | 无 score，不应伪造排名 |
| SEQ | `(col,tier)` 顺序扫描第一个可行位 | SEQ | 不是最近位 |
| NEAR | 最小化 `(travel_time,tier,col)` | NEAR | 不看频次、重量与预留 |
| SCORE | 最小化四项加权总分 | SCORE | 逐件在线，不知未来，不重排 |
| AWRA-LS | 先按 `0.50f+0.35w+0.15v` 排批次，再用 SCORE 分配，最后局部搜索 | SCORE | 启发式、最多5轮、无全局最优证明 |
| CB | 按累计频次分 A/B/C，再从动态剩余空位序列的类起点寻找 | SCORE | 不是固定物理三区 |
| TOB | 按频次排序后逐件选择当时最近位 | SCORE | 当前默认不是 COI 变体 |
| AUTO | 根据 `n/density/skew` 与 objective 在 run start 选一次固定策略 | 固定所选链 | 不是在线学习，不使用重量统计路由 |

实现入口：[warehouse_sim.py:270](F:/abb_wh_work/sim/warehouse_sim.py:270) [strategy_lib.py:58](F:/abb_wh_work/sim/strategy_lib.py:58) [strategy_lib.py:96](F:/abb_wh_work/sim/strategy_lib.py:96) [strategy_lib.py:145](F:/abb_wh_work/sim/strategy_lib.py:145)

AWRA-LS 和 SCORE 最准确的对照是：

```text
SCORE：原到达序 → 每件用 score_fn 选位 → 完成

AWRA-LS：按货物优先级重排批次
         → 每件用同一个 score_fn 选位
         → 最多 5 轮重定位/成对交换
         → 完成

动态逐件阶段：AWRA / CB / TOB → SCORE
```

### 2.4 AUTO 的真实语义

当前阈值包括：`density_low_max=0.25`、`density_high_min=0.70`、`online_skew_low=0.35`、`tiny_batch_max=20`。[strategy_lib.py:71](F:/abb_wh_work/sim/strategy_lib.py:71)

S3 固定使用 `batch_known=True`、`objective=lexicographic`。三个 profile 初始都是 120 件，密度约 0.449，所以九条 AUTO trace 全部命中 `TOB → SCORE`。`strategy_switches=0`；E18 在周期 33、66 重新计算的建议也只记录为 advisory，不执行切换。

新版可以展示：输入统计、命中阈值、推荐原因码、固定算法链。不能展示：在线学习、逐货自适应、重量感知、最优保证。

## 5. 三种输入模式的可行性

### 5.1 A：单件自定义沙盒

最小输入：`ID、名称、重量kg、频次、体积m³`，另需当前库存快照、算法和“预览/提交”语义。

可以做：

- 输入合法性；
- 可行候选数；
- 推荐仓位；
- 单件行程；
- 补齐导出后的评分分解。

不能继承：P50/P95、吞吐、利用率、年化能耗、regret、18×8 聚合结论。

额外风险：PLC 仅拒绝负值，Python 要求重量/频次/体积严格大于 0，并拒绝重量超过 100 kg、体积超过 1 m³；统一口径前不能称完全同源。[04_FB_Warehouse.st:1050](F:/abb_wh_work/plc/04_FB_Warehouse.st:1050) [cargo_scenarios.py:312](F:/abb_wh_work/sim/cargo_scenarios.py:312)

### 5.2 B：预设情景主入口

以 `scenario_id + mode + tier + seed + evidence_stage` 为统一键。E01–E18 可直接复用 registry、runner 和 S1 证据；E01–E03 还能加载现有三维回放。E04–E18 没有回放时必须写“无代表性回放”，不能借用相似 profile 的 KPI。

这是当前最稳妥、证据同源性最强的主入口。

### 5.3 C：规则推荐 + 八算法对照

在同一情景中先显示 AUTO 的输入统计、阈值和推荐理由，再高亮 8 模式 S1 表里的对应策略。可比较 `n_feasible、P95 mean±CI、fail/viol/reject、regret`，但必须限定同一情景、同一 workload、同一 stage。

不得：

- 写“AI 最优算法”；
- 跨情景平均排名；
- 把规则推荐等同 oracle；
- 把 Tier 当成算法推荐档位。

### 5.4 推荐融合

```text
入口页
├─ 预设情景（主）
│  ├─ 情景事实与证据阶段
│  ├─ 规则推荐及原因
│  ├─ 八算法同场对照
│  └─ 若有：代表性三维回放
└─ 单件沙盒（独立）
   ├─ 输入校验
   ├─ 货位决策

```

拟真层 Tier 1/2/3 作为情景的第二维，而不是和算法放在一个“推荐档位”选择器里。

## 6. Factory I/O 对三维建模的实际启示

Factory I/O 官方 Automated Warehouse 的目标是运输、存储和取回货物，而不是展示算法 20×20 逻辑矩阵本身。[官方场景说明](https://docs.factoryio.com/manual/scenes/automated-warehouse/)

### 6.1 入口和出口连续性

- [Emitter](https://docs.factoryio.com/manual/parts/emitter/) 在发射体积被现有物品占用时不会继续生成，天然避免“同一点无脑叠箱”；
- [Remover](https://docs.factoryio.com/manual/parts/remover/) 在物品进入移除体积后删除，避免货物突然在输送中途消失；
- S3 可以借鉴“明确的生成边界—物理链—明确的消费边界”，但不能把这三个逻辑状态伪装成本地 FIO 已验证站点。

### 6.2 设备速度和演示倍率

- Factory I/O [重载辊筒输送机](https://docs.factoryio.com/manual/parts/heavy-load/) 的数字/模拟最大速度约为 0.45/0.8 m/s；
- [轻载输送机](https://docs.factoryio.com/manual/parts/light-load/) 可达 0.6/3 m/s；
- 因此页面应把“物理基准速度”和“画面演示倍率”分开。倍率只能改变播放时钟，不能回写 sim KPI。

### 6.3 布局与碰撞

Factory I/O 的[创建场景指南](https://docs.factoryio.com/getting-started/creating-a-scene/)会用红色边界提示部件相交，并在 Edit→Run 时销毁相交部件。这说明下一版不能只凭截图好看；入库链、载货台、货架、地面和安全空间都必须进入几何碰撞 oracle。

### 6.4 镜头

Factory I/O [User Interface](https://docs.factoryio.com/manual/user-interface/)支持 Orbit、Fly、Follow、固定 Cameras 和时间倍率。S3 最适合采用：一个经过截图验收的统一预设镜头 + 可恢复的自由观察；不再堆叠无意义的额外视角。

## 7. 工业 HMI 与信息架构约束

[ISA-101](https://www.isa.org/standards-and-publications/isa-standards/isa-101-standards)把 HMI 的设计、可用性和性能纳入生命周期；其委员会范围明确包括菜单层级、导航、图形/颜色约定、动态元素和报警。[ISA101 Committee](https://www.isa.org/standards-and-publications/isa-standards/isa-standards-committees/isa101)

[ABB Operator Workplace and Process Graphics Best Practice](https://library.e.abb.com/public/3d78c2290d004e058f33ed16fbaf2d71/3BSE068129%20en%20B%20Operator%20workplace%20and%20process%20graphics%20-%20Best%20practice%20guidelines.pdf)强调以人的任务为中心、统一过程图形和工作场所设计。对本项目的落地约束是：

- 每一时刻只有一个“当前动作”主入口；
- 顶部保留周期级任务和目标，底部只保留设备遥测、镜头和必要口径；
- 正常状态使用低饱和底色，黄色/红色只用于当前目标、阻断或异常；
- “查看全部 8 种算法”必须是明确动作，不是看起来像统计标签的“小按钮”；
- 术语以中文主标签呈现，只保留已解释的算法/统计缩写。

可访问性底线：

- [W3C Use of Color](https://www.w3.org/WAI/WCAG21/Understanding/use-of-color.html)：货物等级、预占、目标和告警不能只靠颜色；
- [W3C Content on Hover or Focus](https://www.w3.org/WAI/WCAG21/Understanding/content-on-hover-or-focus.html)：效率/稳定等指标的解释必须同时支持 hover 与 keyboard focus，并可保持阅读；
- [W3C Name, Role, Value](https://www.w3.org/WAI/WCAG21/Understanding/name-role-value.html)：自定义选择器、滑块、弹窗入口要暴露名称、角色、状态和值。

## 8. 对当前页面问题的裁决

| 问题 | 裁决 | 下一版硬约束 |
|---|---|---|
| 20×20货架像FIO物理孪生 | 阻断 | 常驻标注逻辑SIM，并与54格FIO证据分层 |
| 扫码/遮罩/消费被画成物理链 | 阻断 | 删除物理实体或明确标成SIM外部边界 |
| 蓝色粗线堆垛机 | 严重 | 先定逻辑SIM或FIO参照，再定造型；不能只换灰色 |

| “全量8”难发现 | 严重 | 改为“查看全部8种算法对照”显性入口 |
| 地图/3D货色不一致 | 感知严重，不是数据映射bug | 最终截图像素互认 + 文字/形状冗余编码 |
| 入场200s+ | 原语义错误已修复为静态排队 + 物理入场；但叙事仍需澄清 | 七步真实时间轴与演示倍率分离 |
| 预占格与货架断层/贴片感 | 视觉语义失败 | 与格口同平面、满格体积、无凸起，并用纹理+文字区分 |
| 英文和冗余元数据 | 信息噪声 | 建立白名单，仅保留AUTO/SEQ/P95/SIM等已解释缩写 |
| 默认镜头 | 仍需盲看验收 | 5秒内能回答当前动作、目标货位、货物流向、20×20口径 |

## 9. 为什么之前的自动 QA 与代理监督没有发现

问题不在截图没有生成，而在 oracle 定义错误：

- 合同测试把 `OUTFEED / SCANNED / EXIT_MASK / CONSUMED` 当成必须存在的正确答案，因此不会追问这些状态有没有 FIO 证据；
- 拓扑测试验证“同轴、两侧、不共带”的内部约定，没有对照 `io_map.md`；
- 颜色测试只看源码常量，没有看最终相机、光照和色调映射后的像素；
- “全量8”只检查 DOM 文案存在，没有做可发现性测试；
- 中文清理只扫描少数禁词，没有比较顶部与底部的语义重复；
- 缺少“观众会如何理解”的反向断言。

下一版监督必须新增：

1. 可见实体证据登记：来源、FIO/逻辑SIM、是否计入KPI、是否仅示意；
2. 真相源反向检查：任何“物理/FIO/孪生”元素都对照 canonical 与 `io_map.md`；
3. 5 秒盲看：20×20是什么、FIO验证到什么、当前动作是什么；
4. 最终截图颜色互认，不只比 hex；
5. 唯一当前动作规则；
6. affordance 测试：首次观看者能否找到全部8算法、详情和暂停；
7. 反主张测试：页面不能让观众推断出项目未完成的物理/算法能力。
