## F5 复核：部分接受，核心结论需打回重判

### F5-1 正确链接约定：不是显式 ADR/Slot/handle

本地 ABB ModbusTcp 库文档和官方在线文档一致：`ModTcpMast` 公开接口只暴露 `Eth/IPAdr/UnitID/Fct/Addr/Nb/Data/ParallelProcessing`，没有可由用户传入的连接句柄、Slot 或 `ADR(ModTcpConfig)`。

关键证据：

- 本地 `ModTcpMast`：`C:\ProgramData\AutomationBuilder\AB_LibDoc_2.9\ABB\ModbusTcp\1.1.10.3\Default\KaJXUV6jW7md1AoQHZXBQ3jeEUM\modtcpmast.html:61-65` 显示 `Eth BYTE`，初始 `ETH1`，并注明 “input is ignored, existing for compatibility reasons only”。
- 官方在线同页：`ModTcpMast` 行 135-141 同样说明 `Eth` 是兼容性输入且被忽略；行 143-188 列出 `IPAdr/UnitID/Fct/Addr/Nb/Data`，无显式 config handle。
- 本地 `ModTcpConfig`：`C:\ProgramData\AutomationBuilder\AB_LibDoc_2.9\ABB\ModbusTcp\1.1.10.3\Default\KaJXUV6jW7md1AoQHZXBQ3jeEUM\modtcpconfig.html:39-40` 是配置 FB；`:52-78` 只有 `RespTimeout/KeepAlive/ByteOrder/Port`，无 Eth、IP、Unit、handle。
- 官方应用例 3ADR010980：第 5.1 节说明 `ModTcpMast` 使用 `ModTcpConfig` 设置的一般配置，`ModTcpMast2` 不使用一般配置而使用本地连接参数（官方 PDF 行 157-160）。

因此，Claude 对“不是缺共享句柄/ADR 链接”的方向判断基本成立；`Eth:=0` 本身也不太可能是主因，因为文档说 `Eth` 对 `ModTcpMast` 是兼容字段且被忽略。但 `Eth:=0` 的记录仍建议改成“字段被库忽略，取值不是决定因素”，不要强调它是正确接口索引。

### F5-2 更可能遗漏：设备树 Client Protocols / Modbus TCP Client

官方示例不是“只写 POU + 引库”：

- 官方 PDF 3ADR010980 行 146-156：服务端需要在 `ETH1` 下添加 Modbus TCP Server；客户端需要在 `Protocols (Client Protocols)` 下添加 Modbus TCP Client；`ModTcpMast` 才用于读写 Modbus 数据。
- 现有 F5 记录 `F:\abb_wh_work\l2_factoryio\5_ac500_modbus_master_实验记录.md:23-28` 只证明新增 `PRG_FIO_Bridge`、修 `Execute`、修 `Eth:=0`、Build、Login、fbCfg `NO_ERROR`；没有记录已在设备树下添加 `Client Protocols / Modbus TCP Client`。
- 对 `F:\abb_wh_work\plc\AB_Project` 只读 grep 未发现 `Client Protocols`、`Modbus TCP Client`、`ModTcp`、`PRG_FIO_Bridge` 的可读 XML 命中；工程主文件为二进制，不能完全排除 GUI 内已有对象，但目前缺证据。

这使 `pFC22/pFC23=NULL` 更像“连接资源未由设备树/协议对象注册”或“当前项目配置未完整”，不能直接判成 VirtualAC500 出站 socket 不支持。若确实没加 Client Protocols，对 Claude 的“仿真 stub 静默失败”结论应打回。

### F5-3 FB 调用节奏也不能轻描淡写

`ModTcpConfig` 和 `ModTcpMast` 都继承 `AbbETrig3`，其语义不是“Execute 永远 TRUE 就每周期执行”：

- 官方/本地文档：上升沿启动单次操作；第一周期读取输入并缓存；`Done` 或 `Error` 后，需要 `Execute=FALSE` 才复位输出；再次执行需另一个上升沿（官方 `ModTcpConfig` 行 86-91，`ModTcpMast` 行 92-99）。
- F5 记录 `F:\abb_wh_work\l2_factoryio\5_ac500_modbus_master_实验记录.md:46` 写“改为每扫描周期调用 `fbCfg(Execute:=TRUE)`”，这只能保证 FB 被调用，不等价于重新配置成功；持续 TRUE 不会再产生上升沿。
- `ModTcpMast` 输出错误码里 `BUSY` 明确是 “call again”，`ERR_INTERNAL_IO_RETRY` 也是“function to be called again”；而 `ERR_INTERNAL_UNEXPECTED_STATE` 是 fatal internal error（本地 error-id `:202-205`, `:232-241`；`ModTcpMast` 文档 `:134-138`）。所以 master FB 应该每周期被调用直到 `Done/Error`，完成后拉低 `Execute` 再发下一次。每隔一扫描 xToggle 调用/触发读写，会显著增加状态机异常风险。

调用节奏不能解释全部 `pFC=NULL`，但足以使“运行期负结果”不能当作最终能力边界。

### F5-4 现有错误码怎么解释

如果只是远端 Factory I/O 不通，库更应报：

- `ERR_FAILED_CONNECT`（failed to connect to Server）；
- `ERR_TIMEOUT`；
- `ERR_CONNECTION_CLOSED`；
- 或 Modbus exception。

而当前记录是内部连接指针 NULL、连接级字段全 0、`ERR_INTERNAL_UNEXPECTED`。结合官方示例需要 Client Protocols，这更像“本地连接/协议栈资源没建好”而不是“远端 server 没响应”。

## 给 Claude 的重做要点（用户在场时）

1. 不要先宣称 VirtualAC500 不支持出站 Modbus。先确认 AB 设备树是否已添加 `Protocols (Client Protocols) -> Modbus TCP Client`，并截图/记录对象名、IP、Port、Unit、是否 Active。
2. 保留 `ModTcpConfig`，但按 `AbbETrig3` 语义做配置：`Execute` 上升沿一次，等待 `Done/Error`，然后 `Execute:=FALSE` 至少一周期；不要用“每扫描 TRUE”当作重配置。
3. 读写事务用 `ModTcpMast` 或 `ModTcpMast2` 每周期调用直到 `Done/Error`。若要规避 general config 不确定性，优先尝试 `ModTcpMast2`，因为官方 3ADR010980 说明它使用本地连接参数而不是 general config。
4. 一次只发一个请求：先 FC2 读 input1，读通后再试 FC5 写 coil7；`ParallelProcessing:=FALSE`；每次事务完成后拉低 Execute，再触发下一次。
5. 判据改为：若配置完整且节奏正确后报 `ERR_FAILED_CONNECT/TIMEOUT/CONNECTION_CLOSED`，才讨论远端/仿真 socket 边界；若仍 `pFC=NULL/ERR_INTERNAL_UNEXPECTED_STATE`，需进一步查 Client Protocols 与库内部资源注册。

## 最终结论

- F4：接受。
- F5：不接受“首选=仿真不支持出站 Modbus 主站连接”。当前更稳妥裁决是：`编译/加载级可行性已证；运行期事务未通；但现有证据未排除 AB Client Protocols 配置缺失与 AbbETrig3 调用节奏错误，不能签成仿真能力边界。`
