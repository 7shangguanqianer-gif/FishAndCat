# N4-P2 PLC 状态机、权限与满数组边界（0712）

## 总结

本轮新增两条可直接导致错误状态的阻塞项：权限守卫执行顺序允许越权参数先产生永久副作用；演示批重复载入可写出 400 件数组。另确认策略切换、可调分片参数和统计违规计数均缺少运行期保护。

## N4-14【阻塞】ParamGuard 在消费者之后执行，越权值可先改变布局再被回滚

证据链：

- 主状态机及 fbAssign/fbImprove/fbStats 在 plc/05_PRG_Main.st:57-187 执行。
- 登录与 fbGuard 到 :198-206 才执行。
- 守卫注释声称即使绕过画面锁，非管理员改动不超过一个扫描周期：plc/04_FB_Warehouse.st:1060-1064。
- T55 只单独调用守卫并检查 rDelta 值跳回，没有把分配消费者放在守卫前：plc/06_PRG_Test.st:440-449。

反例：操作员绕过 HMI 写 rDelta/rAlpha/xUseAccel/nBatchPerCycle；同一拍 fbAssign 先以越权值选位、写 aSlots/aGoods/aSlotOps，随后 fbGuard 才把参数值回滚。参数表面恢复了，布局副作用不会撤销。

裁决：当前“画面锁 + ST 守卫双闸”只能防值长期驻留，不能保证越权值从未被消费。若要承接“双闸”话术，guard 必须在所有参数消费者之前执行，或运行开始时把参数复制到受保护快照、忙碌期只读快照。

## N4-15【阻塞】FC_LoadDemoGoods 无容量检查，重复载入会写 aGoods[401]

- 货物数组上限是 1..400：plc/02_GVL.st:71-74。
- PRG_Main 每次 Load demo 上升沿直接调用 FC_LoadDemoGoods，没有 MAX_GOODS 判断：plc/05_PRG_Main.st:100-107。
- 生成函数从当前 iGoodsCount 开始无条件连续加 20：plc/07_GVL_Data_generated.st:84-131；生成器源同样无保护：sim/export_st_vectors.py:116-128。

空表连续载入 20 次达到 400；第 21 次第一件即写 aGoods[401]。若此前已有超过 380 件手工货物，一次演示载入即可越界。段尾命令复位只防卡 TRUE 风暴，不防合法重复点击。

建议：FC_LoadDemoGoods 返回成功/错误码并先判断 iGoodsCount+N_DEMO<=MAX_GOODS；PRG_Main 显示 DEMO_FULL，不得部分写入。

## N4-16【高】货物 ID 可为 0/负数/重复，破坏查询与 GoodsId 哨兵语义

- FB_GoodsInput 只检查满表与重量/频次/体积负值，随后直接写 iId/sName：plc/04_FB_Warehouse.st:743-756。
- ST_Slot.GoodsId 用 0 表示“无货”：plc/01_DUTs.st:28-39。
- FB_QueryGood 对 iId<=0 直接视为未找到，并假定“序号唯一，命中即止”：plc/04_FB_Warehouse.st:812-837。

因此：

- Id=0 的货可入表、可占位，但查询永远找不到，且槽位 GoodsId 与“空”哨兵冲突。
- 负 ID 同样可入表但不可查询。
- 重复 ID 可入表；查询只返回第一件，槽位层无法唯一追溯货物。
- 空名称也没有校验，属于数据质量缺口。

建议至少拒绝 iId<=0 与重复 ID；若题面允许业务主键另行输入，应新增内部唯一索引，不能复用 0 哨兵。

## N4-17【高】策略在开始时部分锁定、结束时又读实时值，存在中途切换竞态

- 进入 ASSIGN 前将 SelStrategy 转成 xUseRank/iStrat：plc/05_PRG_Main.st:127-131。
- 分配结束后是否进入局搜却重新读取实时 GVL_Visu.SelStrategy：:141-147。

反例：

- 以 3=AWRA 启动，分配中切到 2，完成后跳过 LS。
- 以 2=score 启动，分配中切到 3，完成后反而进入 LS，但本轮并非按 AWRA 的启动语义运行。

建议启动时锁存 runStrategy，整个批次只读锁存值；忙碌期禁改策略或把改动排到下一批。

## N4-18【高】管理员可把分片参数改成 0/负数，使状态机永久不前进

- nBatchPerCycle、nPairsPerCycle、nMaxLsRounds 都是 HMI 可调全局变量：plc/02_GVL.st:63-66。
- Assign phase 的进度只发生在 k<nBatchPerCycle 的 WHILE 内：plc/04_FB_Warehouse.st:198-232。值<=0 时 iCur 永不前进。
- Improve relocation 同样依赖 nBatchPerCycle：:285-333。
- Improve pair phase 依赖 nPairsPerCycle：:335-381。值<=0 时 iI/iJ 永不前进。
- nMaxLsRounds<1 仍会先执行一整轮，因为 iRounds 初值为 1：:270-280、383-389。
- ParamGuard 只做权限快照，不做范围校验：:1078-1111。

这会产生 xBusy 永远 TRUE、状态驻留、无 xError/ErrorID 的“静默挂起”。建议用下限钳位或拒绝式校验，并在 FB 内设最大扫描数/错误码，不能只依赖 HMI 输入框 Min/Max。

## N4-19【高】运行中参数没有批次快照，可生成无法归属单一口径的混合布局

FC_CalcScore/FC_CalcTravelTime/FC_CalcPriority 在每次调用时读取 GVL_Param；Assign 和 Improve 跨多个扫描周期运行：plc/03_Functions.st:126-137、204-213、231-233；plc/04_FB_Warehouse.st:152-235、270-392。

管理员可在批次运行中改：

- 评分权重；
- xUseAccel/xSimultaneousXY、速度/加速度；
- Rank λ；
- nBatch/nPairs/nMaxRounds。

前半批与后半批会按不同参数运行，ExecTime 和 Score 也混杂；最终统计没有记录参数版本。应在启动拍锁存完整参数结构与 run_id，忙碌期只读该快照。

## N4-20【高】VisuViolationCount 只查重量/体积，不是完整“约束违规数”

- FB_Stats 对 Assigned 货物仅比较 MaxWeight 和 MaxVolume：plc/04_FB_Warehouse.st:417-431。
- 报告却把违规列解释为“超限重/超容积/占用冲突”的硬约束计数：sim/build_report.py:509-513。

当前未检查：

- Assigned 坐标是否 0..19；非法坐标还会先造成数组越界。
- aSlots[x,y].Occupied 是否为 TRUE。
- aSlots[x,y].GoodsId 是否等于该货 ID。
- 一货一位、一位一货、重复 ID。
- 是否落到 PresetOcc 或 MaintBlock。
- aSlotFreq/VisuSlotState 与货物表是否一致。

所以 0 只能称“成功记录中的承重/容积复查为 0”，不能称全部约束/占用冲突为 0。建议新增全仓一致性扫描与细分错误计数。

## N4-21【中】SceneDetect 最大负载的单周期安全只是估算，没有实测/测试锚

- 初次触发单拍遍历 iGoodsCount：plc/04_FB_Warehouse.st:944-964。
- 插入排序每周期 50 件，注释自估 n=400 最坏约 2 万步并直接断言不触发看门狗：:910-920、968-985。
- T49-T53 只覆盖 5 件和 21 件画像/分支：plc/06_PRG_Test.st:385-427。

没有 400 件、逆序频次、与 fbVisu/fbAssign 同任务并发时的最大扫描耗时证据。此处应标“静态估算，待 AB/目标机压力测”，不能并入“10 ms 已证安全”。

## N4-22【中】快照可捕获推荐/查询等瞬态状态，并非纯布局快照

- CmdSaveSnap 直接整体复制 VisuSlotState：plc/05_PRG_Main.st:233-239。
- VisuSlotState 同时承载推荐状态 3、查询高亮 6、已占 2 等：plc/02_GVL.st:86-90；查询会写 6：plc/04_FB_Warehouse.st:797-835。

若在 WAIT_INPUT、查询高亮或分配中途保存，后续“优化前 vs 优化后”会混入瞬态颜色或半成品。建议快照从 aSlots/GoodsId 重建纯状态，或只允许 DONE/明确基线状态保存。

## N4-23【已知阻塞复核】“检修存量冻结”仍未实现

- GVL/README 说已有货物原地冻结：plc/02_GVL.st:76-83；plc/README_PLC.md:40-44。
- LocalImprove relocation 只要 Assigned 就可从维护源位搬出：plc/04_FB_Warehouse.st:286-326。
- FC_FitsPhysical 注释又说“检修位上的货可被换走”：plc/03_Functions.st:82-86。

这是 N3/Claude 既有 B-2，本轮状态机复核未发现反证，仍是未修阻塞。必须在“冻结”与“只禁入、允许出库”中二选一，代码和话术统一。

## N4-24【已知阻塞复核】FC_TMax 对可调速度仍无除零保护

- FC_CalcTravelTime 会把非法速度钳到安全值：plc/03_Functions.st:126-136。
- FC_TMax 直接除 rVx/rVy：:174-177。
- ParamGuard 允许管理员保存 0/负值，且不做范围验证：plc/04_FB_Warehouse.st:1078-1111。

这是既有 ST 审查阻塞，本轮未修；同时 N4-01 新增发现它还不随 xUseAccel 改归一分母。

## N4-25【正面】若输入保持合法，核心数组与路径的常规上界设计较稳

- COLS/TIERS/MAX_GOODS 是常量并与数组维度一致：plc/02_GVL.st:9-26、69-84。
- FC_CheckFeasible/FC_FitsPhysical 先查坐标再访问数组：plc/03_Functions.st:59-67、79-87。
- L 形路径最大点数为 39，落在 VisuPath[0..39] 内；BuildVisuPath 先验坐标范围：plc/04_FB_Warehouse.st:563-595。
- AnimatePath 对运行中路径清零有负索引护栏：:488-516。

这些正面项成立，但依赖入口先保证 iGoodsCount、ID 与可调参数合法，不能抵消上述满数组/参数问题。

## 修复优先级

1. 先把 guard 移到消费者前或引入批次参数快照。
2. 给 LoadDemo 原子容量检查。
3. 锁存 runStrategy；给三项分片参数做范围校验和 xError/ErrorID。
4. 拒绝非正/重复 ID。
5. 把 ViolCnt 改为全仓一致性审计，不再把重量/体积两项外推成“全部约束”。
6. 400 件最坏负载与 SceneDetect 在 AB/目标机做压力测试。
