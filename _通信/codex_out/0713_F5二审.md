# F5 二审：VirtualAC500 能否作为 Modbus TCP Client 直驱 Factory I/O

> 执行日期：2026-07-14  
> 范围：只读复核；未操作 Automation Builder、VirtualAC500、Factory I/O，未运行 `tools/ab_scripting`，未执行 Git。  
> 裁决对象：`l2_factoryio/5_ac500_modbus_master_实验记录.md` 附录 B 与 `l2_factoryio/media/f5_redo_0713/01-07*.png`。

## 一、裁决

**裁决：部分接受证据解释，接受项目级停止条件。**

1. **项目级结论：当前安装环境中的 VirtualAC500 主站直驱不可用，应停止继续把 F5 当作可交付联调路线。**适用边界为：Automation Builder 2.9、当前 `VirtualAC500_V3` 3.9 系列实例、`ABB_ModbusTcp_AC500` 1.1.10.3、当前工程与本机。五项重做后，合法 FC2 请求仍在触发周期同步返回 `ERR_INTERNAL_UNEXPECTED_STATE`（16#8005），502 与无人监听的 5020 无连接层差分；这足以证明本次失败早于远端 TCP 连接/响应阶段。
2. **产品级结论：证据不足以声称“所有 VirtualAC500 版本永远不能做 Modbus TCP Client”。**ABB 官方仿真帮助只给出通用限制：模拟 PLC 功能有限，部分 POU 会正常编译/下载但不工作；未找到 ABB 对 `ModTcpMast2 + VirtualAC500` 的逐项支持矩阵或明确禁用声明。因此对外应写“当前 VirtualAC500 实例未能提供该主站功能，项目按不可用处理”，不要写成全产品线定理。
3. **Claude 对 `pFC22/pFC23=NULL` 的机理解释需撤回。**这两个名称与库公开类型 `ETH_MOD_FCT22_TYPE` / `ETH_MOD_FCT23_TYPE` 精确对应，明显是功能码 22/23 的专用内部对象，不是已被文档证明的 TCP 连接句柄。本次请求是 FC2，它们为 NULL 不能证明“连接资源没有注册”。同理，截图 07 在 `Execute=FALSE`/输出复位期看到的 `wPort/byUnitID/byFunctionCode=0` 只能作为现象，不能作为决定性机理证据。

建议状态：**F5 结案为“VirtualAC500 负结果（当前环境）；真机 AC500 或非 ABB 虚拟 PLC 路线待做”**。C4/C5/F6 不应再等待 VirtualAC500 内继续试参数；若要证明 AC500 直驱，只能转 PM5650 实机，或用 CODESYS/软 PLC 作为独立联调替身，并严格标注平台。

## 三、未试配置面的逐项裁决

### 1. ETH1 NetConfig / 接口索引

不构成继续试验的高价值方向。

- 本地 ABB 1.1.10.3 文档 `modtcpmast2.html:61-66` 明确：`Eth` 输入被忽略，仅为兼容保留。
- 本次目标为 `127.0.0.1`，即便 VirtualAC500 的外部网卡映射有问题，正常到达 socket 层也应产生 `FAILED_CONNECT`、`TIMEOUT` 或 porting 类结果，而不是每两周期同步 `16#8005`。
- 防火墙放行仍是 ABB 对网络仿真的一般提醒，但当前错误发生在连接结果之前；继续改防火墙/ETH1 IP 缺乏针对性。

### 2. 协议对象 Bus cycle task

不是首选根因。

- 安装设备描述 `C:/ProgramData/AutomationBuilder/AB_Devices_2.9/32802/1020 0005/3.9.0.0/device.xml:75-84` 对该 Client 对象写明 `needsBusCycle="false"`；它主要引入 `AC500_ModbusTcp` placeholder library，并无可下载的 IP/端口参数。
- 图 03 的 tooltip 虽显示 `Bus cycle task: Task`，但设备描述自身表明该对象不依赖周期总线驱动。因此再换任务绑定缺乏证据基础。

### 3. CM 通信模块级对象

不需要。ABB 官方 Modbus TCP 示例把 Client 放在根级 `Protocols (Client Protocols)` 下；Modbus TCP Client 走处理器板载 Ethernet，不要求额外 CM5640。当前图 02/03 已符合这一结构。

### 4. `ModTcpConfig` / `ModTcpMast` 共享配置

已被 `ModTcpMast2` 合理排除。`ModTcpMast2` 自带 `RespTimeout/KeepAlive/ByteOrder/Port/ConnectTimeout`（本地文档 `modtcpmast2.html:154-198`），因此不依赖 `ModTcpConfig` 的全局参数。重做后仍同步失败，不能再归因于 config-mast 链接遗漏。

### 5. Virtual runtime 协议/POU 支持边界

这是目前最符合全部观测的解释，但应写成“当前实例的能力边界”，不是逆向证明出的库内部实现事实。

- ABB 官方《Testing in simulation mode》明确提示 Virtual AC500 功能有限，某些 POU 会编译和下载成功但没有功能；网络应用还可能需要 Windows 防火墙放行。
- 当前 runtime cfg 确实装有 `SysSocket` 与 `SysSocketABB`（`C:/Program Files/ABB/AB2.9/AutomationBuilder/VAC500/V3_2.9.0/CODESYSControl.cfg:48,113`），所以不能写“runtime 完全没有 socket”。更准确的说法是：**当前 `ABB_ModbusTcp_AC500` 的 Client 执行路径未能在 VirtualAC500 上进入可用 socket 状态。**
- ABB 错误表只把 16#8005 定义为 “Unexpected state. Fatal internal error”（本地 `error-id.html:232-235`），未公开触发源码；不能声称已经坐实某个具体空指针分支。

## 四、对 `pFC22/pFC23` 证据的纠错

图 07 展开的字段类型显示为 `POINTER TO ETH_M...`。本地库目录同时公开且仅公开两个同名专用结构：

- `eth-mod-fct22-ty.html:38-53`：`ETH_MOD_FCT22_TYPE`，包含 AND/OR mask；
- `eth-mod-fct23-ty.html:38-39`：`ETH_MOD_FCT23_TYPE`。

因此最合理解释是 `pFC22/pFC23` 服务 Modbus 功能码 22/23 的特殊数据结构。本次读请求为 FC2，它们为 NULL 很可能是正常未使用态。除非 ABB 源码或符号说明明确证明其是连接资源，否则后续文档不得继续用“pFC NULL = TCP 连接句柄未注册”作证。

这项纠错不推翻项目级负结论，因为负结论仍由“规范调用 + 同步 16#8005 + 502/5020 无差分 + ABB 明示仿真 POU 可能无功能”共同支持。

## 五、推荐路线与对外口径

### 项目执行路线

1. **停止 VirtualAC500 F5 参数试错**：该路径已完成一次有控制变量的负结果，继续切 ETH/IP/Task/Port 的信息增益很低。
2. **真机路线**：用 PM5650 的板载 Ethernet 跑同一最小 FC2→FC5 桥，作为“AC500 直驱 Factory I/O”唯一可升级为硬件闭环的证据。
3. **无真机期间**：Factory I/O 保留 Python/pymodbus 直控，标“Python 直控技术预演”；若使用 CODESYS 软 PLC，则标“软 PLC 联调替身”，不得写成 AC500 实测。
4. **可选的一次性诊断，不作为继续主线的前置**：若 Claude 希望补齐内部状态，可只在下次真机/受控场次调用 `ModTcpInfo` 记录 `Rdy/ErrorID`，或向 ABB 支持索取 VirtualAC500 3.9 对 `ABB_ModbusTcp_AC500` 的支持矩阵。没有厂商答复前，不再逆向猜 16#8005 的具体触发点。

### 建议写入账本的口径

> 在 Automation Builder 2.9 当前 VirtualAC500 V3 实例中，已按 ABB 文档添加 Modbus TCP Client、改用 ModTcpMast2 并遵循 AbbETrig3 边沿节奏；FC2 对监听端口 502 与无人监听端口 5020 均在连接前同步返回 ERR_INTERNAL_UNEXPECTED_STATE，未形成 TCP 事务。故本项目将 VirtualAC500 主站直驱判为当前环境不可用；该负结果不外推到实机 AC500 或其他 VirtualAC500 版本。
