## 三、N4 对上一轮最重要的增量纠正

### A. 立即纠正 8.757 的能力标签

N4-01 发现：

- Python `make_score_fn` 在 `ACCEL=True` 时使用动态 `t_max_now()`，评分分母为约 `39.666667 s`；
- ST `FC_TMax` 仍固定使用匀速公式，分母为 `38 s`，不随 `xUseAccel` 切换；
- 现有 T19 只锁匀速评分，T22 只锁加减速行程时间，没有锁加减速评分。

因此上一轮若把 `8.757/60.4` 写成“PLC 可实现代理”“与 PLC 数学式同源”或类似表述，请继续降级：

- `8.757`：**Python 引擎的加减速 + 静态权重 sim 结果**；
- `8.794`：按当前 ST 固定 `FC_TMax`、ST tie-break/epsilon 做的更近似 Python 只读代理，约 `8.794 s / 60.24% / 67.07% / heavy tier 0.5417`；
- `8.794` 仍不是 PLC/AB/AC500 实跑，也不能证明当前二进制工程逐值复现；
- `6.443` 仍只是匀速 + 静态、贴近 PLC 默认开关方向的 sim 代理；
- `8.40`（5-seed 头条）、`8.6232`（sum 单 seed canonical 锁）、`7.6202`（legacy and 锁）的定义不变。

不要为了“修正 8.757”而把 `8.794` 升格成新头条；它的用途是揭示 Python 与 ST 的评分分母差异。所有头条仍必须显式标注 sim 设计口径。

### B. 检查上一轮重生的图、报告和 PPT 是否仍被旧图源污染

重点处理 N4-26～N4-34：

- `sim/plots.py` 的主要计算仍是静态权重且默认 `ACCEL=False`；
- fig1/fig5 图内仍可能写 `and`，代码实际使用 `sum`；
- PPT 的 H 主结果页嵌入 fig1，报告 H 段嵌入静态 fig4；仅改正文再重生，会继续把错图注入二进制；
- `sim/run_all.py` 不是完整产物 DAG：漏 oracle 测试、专项实验、报告和 PPT，却会把目录中既有旧图打印成“产物”；
- `build_report.py` 缺输入时仍可能保存并 exit 0；
- `1_给你看/项目全景说明书.docx` 是生成链外的孤儿快照，仍可能保留旧口径并绕过 `doc_check`；
- 当前二进制内嵌图片哈希与当前 figs 一致、关系完整，这是正面证据；问题在当前图源语义，不是二进制里藏有另一套旧图片。

处理顺序必须是：**数值/口径真相源 → 图表计算与图内标签 → 报告/PPT 生成器 → 二进制重生 → 文本、图片哈希和视觉验收**。不要先重生二进制再修图源。

### C. 不再把现有绿灯当成完整证明

重点处理 N4-02～N4-13：

- `test_oracle_gap.py` 的“真实重跑”实际读取冻结的 `instgen_detail.csv`；
- golden test 会写正式 `sim/out`，测试不是隔离只读；
- `run_all.py` 当前只覆盖 21 个 Python 测试，不是 23 个；
- H 锁、legacy 锁、零违规类测试缺 `failed == 0` 断言；
- T25/T58 在单扫描里做 10 万/100 万级 `WHILE`，存在 watchdog 风险；
- `N_CASES` 可在线改；T58 的 `aSlotOps >= 1` 可被历史状态垫绿；部分测试有顺序依赖。

如果上一轮已经更新了“全量回归、一键复现、全部约束零违规”等文字，请再次核对这些断言是否有真实测试门。没有门就改为限定表述，不能只改测试名称。

### D. PLC 项先做独立修复面，不在无授权时直接改 ST

重点处理 N4-14～N4-25：

- `ParamGuard` 在主消费者之后执行，可出现“越权值回滚，但布局副作用不回滚”；
- `FC_LoadDemoGoods` 无容量原子检查，空表第 21 次加载可写 `aGoods[401]`；
- Goods ID 接受 0、负数和重复值，破坏哨兵/查询语义；
- 策略开始时部分锁存，完成时又读实时 `SelStrategy`；
- `nBatchPerCycle`、`nPairsPerCycle <= 0` 可让状态机不前进；
- 运行中参数无批次快照，可能得到混合口径布局；
- `VisuViolationCount` 只查重量/体积，不能称“全部约束违规数”；
- SceneDetect 的 400 件最坏单周期负载未实测；快照会捕获部分瞬态状态；
- 已知的“检修存量冻结”缺口和 `FC_TMax` 除零风险仍成立；
- 合法输入下常规数组与路径上界设计较稳，这是正面确认，不要为了修问题而否定它。

若没有 PLC 修改授权，请给出逐文件补丁计划、测试反例和风险顺序后停下。若已有明确授权，优先修会产生永久错误状态的 N4-14、N4-15，再修输入契约和口径快照；每项都要补能在失败前变红的测试。

### E. 把“可复现”和“可提交”拆成独立验收门

重点处理 N4-35～N4-41：

- 当前没有依赖锁/环境清单，干净机器只装 Python 3.8+ 不能复现全部流程；
- `sim/out`、`sim/figs` 被忽略，但 `run_all` 又不能全部重建；
- 没有提交 allowlist/manifest，整仓包含缓存、备份、锁文件、内部沟通材料等；
- 多个工具绑定本机 F:、用户目录或旧 spike 路径；
- 没有许可证和第三方材料再分发清单；
- 当前 `doc_check` 基线是 11 个断链 + 1 个过时 DOCX、exit 1，不能写成“收尾全绿”；
- 相对路径和固定种子的核心基础较好，可保留此正面判断。

不要把“本机能跑”“输出目录里有文件”“二进制图片哈希一致”合并解释成干净机可复现。

## 四、执行步骤

1. **当前态盘点**：列出上一轮已经改动/重生的真相源、生成器、Markdown、DOCX、PPTX、HTML 和测试；记录当前内容，不假设 N4 审计时状态仍原样存在。
2. **N4 全项裁决**：建立 N4-01～N4-41 矩阵，逐项写当前状态、当前 `file:line`、影响产物、处理动作和验证方式。41 项必须齐全；正面项也要登记。
3. **先止扩散**：搜索当前工作区中仍把 `8.757` 绑定 PLC、把静态图说成自适应、把 21 写成 23、把非全量 `run_all` 写成一键复现、把部分 ViolCnt 写成全部约束的地方。
4. **按授权处理**：只修上一轮授权范围内仍存在的问题。对越权范围只交补丁计划，不静默扩大修改面。
5. **重生前验源**：若获准重生 DOCX/PPTX，必须先验证图源、生成器输入和失败退出行为，再重生；对用户文档遵守备份和视觉 QA。
6. **验证与回执**：运行最小必要验证。任何没有 AB 实跑的项明确写“静态审计/测试代理”，不要写“PLC 已验证”。
