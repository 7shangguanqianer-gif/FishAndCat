# N4-P1 测试、假绿与覆盖缺口（0712）

## 总结

当前“23 个 Python 测试 + 59 个 PLC 测试”不能等价理解为“核心声明已有 82 个独立护栏”。发现 13 组测试口径问题，其中最关键的是：加减速评分并未做 Python↔ST 一致性测试，oracle golden test 也没有重跑它自称会覆盖的策略性能。

## N4-01【阻塞】加减速评分归一分母 Python↔ST 不一致，现有测试完全漏检

证据：

- Python make_score_fn 在构造评分函数时调用 t_max_now，分母随 ACCEL 改变：sim/warehouse_sim.py:110-115、241-251。
- ST FC_TMax 始终使用匀速距离/速度公式，不看 xUseAccel：plc/03_Functions.st:174-177；FC_CalcScore 再以它作分母：:204-205。
- 导出器先把 ACCEL 置 FALSE 生成 ExpScore，之后只生成 ExpTimeAccel，没有 ExpScoreAccel：sim/export_st_vectors.py:30-41、70-72、99-103。
- PLC T19 只测匀速评分；T22 打开 xUseAccel 后只测行程时间，不测评分：plc/06_PRG_Test.st:101-143。

只读量化（120 件、skew、sum、5 seeds、静态权重、ACCEL=TRUE；ST 代理还采用 tie_break=st、ls_eps=1e-6）：

| 项目 | Python 当前 P* | 按 ST 固定 FC_TMax 的更近代理 |
|---|---:|---:|
| 最远时间归一分母 | 39.666667 s | 38.000000 s |
| AWRA exp_t 均值 | 8.756567 s | 8.794145 s |
| AWRA exp_t 标准差 | 0.402707 | 0.375609 |
| 能耗代理均值 | 14395.52 | 14448.32 |
| 重货平均层 | 0.5500 | 0.5417 |
| 较 seq 降幅 | 60.407% | 60.237% |
| 能耗降幅 | 67.190% | 67.070% |
| 每 seed 平均位置差 | — | 38.4/120 件 |

逐 seed exp_t（Python P* → ST 分母代理）：

- 2026：8.786004 → 8.916762，位置差 61。
- 7：8.144789 → 8.210813，位置差 15。
- 42：9.087189 → 9.075371，位置差 52。
- 123：8.624977 → 8.645331，位置差 29。
- 999：9.139877 → 9.122447，位置差 35。

裁决：数值卖点只小幅变化，但布局大量变化，故 8.757/60.4 不能再称“当前 ST 可逐值实现的代理”。应称“Python 加减速+静态权重代理”；更接近当前 ST 数学式的只读估计为 8.794/60.24/67.07，仍不是 AB/AC500 实跑。

## N4-02【高】oracle golden test 的“真实重跑全流程”说明不成立

- 测试说明称 og.main 会使 warehouse_sim 评分/可行性漂移变红：sim/tests/test_oracle_gap.py:13-17。
- oracle_gap 明确是纯后处理，load_detail 读取冻结 sim/out/instgen_detail.csv：sim/oracle_gap.py:21-22、55-64。
- og.main 会用当前检测器重算画像/选型，但各策略 exp_t、fail、viol 等性能仍来自冻结 CSV：sim/oracle_gap.py:75-84。

因此它能捕获检测阈值/FIXED 表变化，不能捕获评分函数、可行性、策略执行或实例生成器性能变化，除非先单独重跑 instance_gen.py。

## N4-03【高】oracle golden test 会修改正式产物，测试不是只读

- 测试直接在真实路径调用 og.main：sim/tests/test_oracle_gap.py:71-73。
- main 会重写三份 CSV、summary 与 fig13：sim/oracle_gap.py:149-156、212-213、257-258。
- 测试说明却称真实输出“不受影响”、mtime/size 不变：sim/tests/test_oracle_gap.py:22-26。

即使内容确定性相同，mtime 与容器写入仍改变；测试会掩盖“产物是否由本轮实验生成”的新鲜度证据。本轮因此没有运行它。

## N4-04【高】run_all 所谓“回归测试”漏掉新增的 2 个 oracle 测试

- 仓库共 23 个 test_*：test_sim.py 21 个，test_oracle_gap.py 2 个。
- run_all 直接执行 tests/test_sim.py，而不是 unittest discover：sim/run_all.py:19-20。
- 所以一键底盘只跑 21 个；docs/overnight-report_0711.md:87 的“23/23”来自另一次独立全量测试，不能外推给 run_all。

## N4-05【高】H 锁、legacy 锁与“全场景零违规”没有断言 failed=0

- H 锁只断言 exp_t 与 viol：sim/tests/test_sim.py:174-190。
- legacy H 锁同样只断言 exp_t 与 viol：:194-205。
- 全场景测试只断言 viol：:209-214。
- metrics 的 exp_t 只对成功放置货物计算：sim/warehouse_sim.py:384-389。

当前锁定数据的 failed 确为 0，本发现不推翻现值；但未来若部分货物失败，幸存者仍可能给出漂亮 exp_t 与 0 viol，测试会假绿。应显式断言 len(failed)==0 或把 failed 纳入复合 KPI。

## N4-06【中】δ=0“退化为 near”的测试只证明占位集合相同

- test_f2_delta_zero_degenerates_to_near 只比较两种策略的仓位集合：sim/tests/test_sim.py:139-144。
- 不比较货物→仓位映射、exp_t、hot_t 或 score。

所以可保留的结论是“该 seed/场景下占用位置集合相同”，不能扩成“算法严格等价”。N3 已量化同一集合下映射/KPI仍可不同。

## N4-07【高】PLC T25/T58 把跨扫描状态机塞进单扫描大循环，可能触发看门狗

- T25 在一次 xRunTests 触发中循环调用分配最多 100000 次、局搜最多 1000000 次：plc/06_PRG_Test.st:161-186。
- T58 同样最多循环 100000 次：:475-492。
- README 建议 PRG_Test 与 PRG_Main 同挂 10 ms 任务：plc/README_PLC.md:24。

这能得到终局，但不再是“每扫描只跑一片”的执行形态；置 xRunTests=TRUE 时存在单周期看门狗风险。测试也没有断言循环是否因 xDone 结束，还是因上限超时退出。59/0 不能证明测试触发本身满足实时性。

## N4-08【中】N_CASES 不是常量，可被在线改写造成隐藏测试或越界

- N_CASES : INT := 59，不在 CONSTANT 区：plc/06_PRG_Test.st:11-17。
- 初始化固定写 aResult[1] 到 aResult[59]，汇总却 FOR 1 TO N_CASES：:42、513-520。

若在线写成小于 59，后部失败会被忽略；写成大于 59 会访问 aResult 越界。应改为常量，或汇总直接使用数组固定上界。

## N4-09【高】T58 的 aSlotOps 断言可被旧履历“垫绿”

- fbInit 明确保留 aSlotOps：plc/06_PRG_Test.st:475-478。
- 断言只要求目标格 aSlotOps>=1：:493-496。

如果目标格在此前运行已经大于 0，本次分配即使漏加计数仍会通过。应在分配前快照两个目标格的计数，并断言本次精确 +1；同时不能只检查 aSlotFreq[10,10] 一个任意格为 0。

## N4-10【中】T21/T55/T59 的名字比实际断言强

- T21“与 Python 同源”只查 iGoodsCount==N_DEMO，没有逐货值或哈希：plc/06_PRG_Test.st:127-130。
- T55 声称守卫全部可调参数，但只验证 rDelta：:440-449。
- T59“层计数正确”只断言 VisuTierOcc>=1，不验证精确值：:498-508。

这些测试可作为烟雾测试，不能作为“全字段同源”“25 参数全部被守卫”“层计数精确”的证据。

## N4-11【中】PLC 测试存在顺序依赖与不自包含状态

- T56 依赖 (5,5) 初态未被外部改写；T58 又故意依赖跨 Reset 保留的 aSlotOps。
- T19 评分使用全局权重但测试前未统一重置全部 GVL_Param；管理员曾调参时可改变结果。
- T25/T58 达循环上限没有独立 timeout 失败码。
- sLastFail 只保留最后一个失败编号：plc/06_PRG_Test.st:513-518。

59/0 的可靠解释应限定为“在已知干净初态、当前参数与当前任务配置下的在线结果”。

## N4-12【中】自动测试覆盖集中在核心几何，新增模块多数没有保护

仅有两份 Python 测试。没有直接自动化覆盖：

- adaptive_weights/verify_policy 的 failed 处理；
- realism 的 common-arrival 公平性；
- lifecycle 的尾部/积压结算；
- dwell 的重定位成本；
- build_report/build_ppt 的语义和图文口径；
- CSV schema、参数元数据、输出新鲜度；
- 生成器与二进制文档一致性；
- 400 件 PLC 极限、重复 ID、策略中途切换、守卫副作用。

## N4-13【高】doc_check 既不够宽，当前也并非全绿

设计缺口：

- 只识别少数目录和 md/docx/png/csv 等扩展，不检查 pptx、html、json、视频、根文件或二进制内部文本：tools/doc_check.py:23-31。
- 只扫描 3 份“无状态文档”，正则很窄，抓不到 59/0、23→59、旧 H 数字和 sim/PLC 边界：:33-37。
- 用户 Word 只用 mtime 判断，不核正文：:49-65。
- 项目全景说明书.docx 没有同名 md，完全绕过检查。

本轮只读实跑结果：

- 路径断链 11 处。
- 2_你要操作/AB可视化搭建操作卡_块3增补.md 的 Word 版旧于 md 1 处。
- 因此当前 run_all 若执行到最后的 doc_check 步骤会失败，不存在“当前一键全绿”。

## 建议最小修复顺序

1. 先补 ExpScoreAccel 与 sum+ACCEL 端到端差分锁；同步修正 8.757 的代理称谓。
2. 把 oracle 性能重算与后处理拆成纯函数/临时目录，测试不得写正式 out。
3. H/legacy/全场景测试补 failed==0。
4. T58 改前后差值；N_CASES 改常量；T25/T58 加显式 done/timeout 断言。
5. run_all 改 unittest discover，且把“底盘”和“全报告再生”拆成两个诚实名称。
