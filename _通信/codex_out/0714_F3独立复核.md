# F3 全程独立复核终稿（SOL 修订版，覆盖 0714 旧结论）

## 裁决

**部分接受；不接受“F3 已完全安全闭环”的签绿。**

接受 H5 在当前 work 场景下的 **9/9 功能成功**：cell 1/30/54 各 3 次、每次均完成六个实际诊断相位并有下放 Z 启动日志。

不接受“Target 名义高度就是已充分证实的唯一根因”及“LOWER_STALLED 是已验证 fail-safe”的表述。F3 应降为：

> **H5 9/9 运行成功；Target 恢复是当前 work 场景下下放成功的必要操作条件。根因机理与异常分支安全性待补证，F3 不得以完整闭环口径签绿。**

## Findings

### F3-R1｜阻断｜`LOWER_STALLED` 在 H1.3 后与自身前置条件矛盾，测试把不可达分支 mock 成可达

**事实错误**

`place_lower()` 先写 `Target=cell`，随后发生 `NO START` 才调用 `_zero_drop_hold_confirmed()`；后者又硬性要求现场 `Target==0`。

- `l2_factoryio/f3_stacker_control.py:989-997`
- `l2_factoryio/f3_stacker_control.py:1042-1050`

因此，在正常 Modbus 写入保持语义下，H1.3 后的 `NO START` 会看到 `Target=cell`，自检必失败，转为 `LIFT_POSITION_UNKNOWN` 持久锁；`LOWER_STALLED` 不可达。它只有在外部异步把 Target 清零（例如残留 force、其他写入者或读写异常）时才可能可达，而这本身不应被解释为安全放置。

现有测试不能反证该问题：它直接将 `_zero_drop_hold_confirmed` mock 为 `True`，绕过真实 `Target=cell` 状态。

- `l2_factoryio/test_f3_stacker_control.py:321-335`
- 同文件 `:339-364` 反而明确断言 `target=30` 时自检应为 False。

**影响**

旧报告所称“仍可执行的 LOWER_STALLED → 人工确认 → retract”并非当前正常 `NO START` 的实际路径；但代码、CLI 和测试仍保留一个若被外部状态触发便可把异常解释为“已放置”的分支。这既不是已验证 fail-safe，也不是可复现的异常处理。

**可行替代**

删除该正常放行语义：任何 `place-lower` 的 `NO START` 均记录持久故障、禁止 retract/travel，直到完整重启和新 epoch。新增不使用 mock 的测试：模拟 `Target=cell + NO START`，断言必落锁；模拟外部 Target 被清零，也仍不得产生 `PLACEMENT VERIFIED`。

### F3-R2｜严重｜“Target 是名义高度 Z setpoint 的唯一根因”证据不足，存在未排除的同观测解释

**缺失证据**

账本只陈述“写回 Target=30 后 0.1 s / 34 采样 Z 启动”，但当前项目中未找到该 34 点原始采样、执行脚本、场景哈希或处理记录；可追溯来源仅为叙述：

- `docs/overnight2-report_0712night.md:182-185`
- `l2_factoryio/f3_stacker_control.py:962-968` 是代码注释，不是原始测量。

H5 的九次日志确实表明“`Target=cell` 后 `Lift=False` 可启动 Z”，例如：

- `l2_factoryio/logs/F3_diagnose_20260713_191012.log:2-5`
- `l2_factoryio/logs/F3_diagnose_20260713_191119.log:2-5`
- `l2_factoryio/logs/F3_diagnose_20260713_191818.log:2-5`

但它不能区分以下解释：

- Target 为非零时重新武装/刷新了控制器，而非“名义高度”本身作为 Z setpoint；
- 该次写入同时消除了此前 Target force 或其他场景状态的影响；
- 场景移植、force 清理和相位时序修复与 Target 写入不是隔离变量。

**影响**

修复策略目前有效，但根因表述超出证据。将其写成唯一机理会误导后续排障和工程论证。

**可行替代**

将结论收缩为“Target=cell 是已观察到的必要操作条件”。后续在固定场景副本和哈希下保存原始 CSV，至少对照 `Target=0` 与 `Target=cell`，同步采集寄存器读回、Lift 线圈、Moving Z、叉位、force 状态和时间戳。

### F3-R3｜严重｜`goto()` 的“NO START + 静止 = 已在位”仍是实际盲区；后续人工门只是程序性缓解

**事实错误**

`goto()` 在没有位置传感器的情况下，把任意“目标写入后不动”的状态解释为已到位，并更新 `position_hint`：

- `l2_factoryio/f3_stacker_control.py:647-656`

这同样兼容驱动被抑制、目标未生效、场景状态异常等失败。项目自身已确认没有 X/Z 实体位置反馈，只能依赖视觉：

- `docs/overnight2-report_0712night.md:137-143`

`place-extend` 的确要求下一步传入 `--confirm-position`，且相位链要求上一步为 travel：

- `l2_factoryio/f3_stacker_control.py:1307-1321`
- `l2_factoryio/f3_stacker_control.py:1332-1342`

但这只是操作员布尔确认，不是机器独立验证；同时 travel 日志和 phase state 已可能被误记为成功。

**影响**

标准 H5 操作规程能通过人工视觉降低风险，但不能把该 fallback 称作自动 fail-safe，也不能以其日志作为精确位置的独立证据。

**可行替代**

无既有可信 `position_hint` 时，禁止 `NO START` 自动转“已在位”；改为 `POSITION_UNVERIFIED` 故障或强制显式、可追踪的人工确认后才继续。增加测试覆盖“静止但不在目标位/目标被抑制”。

## 对旧报告 `0714_F3独立复核.md` 的覆盖

- **正确**：H5 9 次成功、cell 1/30/54 各三次的 retract 终验日志；work 场景位置/驱动/force-off 的结构核查；零行程放置不能仅凭执行器静态状态自动判真。

- **遗漏**：Target 根因的原始 34 点数据缺失及混杂变量；`goto` 的静止即在位盲区；H5 实际为 54 份、9×6 日志；Target 时序与 LOWER_STALLED 真实可达性的测试缺口。

- **需删除的旧结论**：任何“LOWER_STALLED 当前可实际导致错误收叉”的确定性表述。应替换为“设计意图不安全，但当前正常路径因前置条件矛盾而不可达；一旦外部异步清 Target，仍可能变成不安全分支”。
