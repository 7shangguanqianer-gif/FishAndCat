### 1.4 B4/H1：safe_stop 真值表重写

- 无论读回是否成功，先独立 best-effort `Target=0`：`f3_stacker_control.py:490-505`。
- 等 Target=0、Moving X/Z=False，并要求连续两次完全一致的停止快照：`f3_stacker_control.py:460-488`。
- 只有“恰好一侧 fork hold 线圈=True + 对应 one-hot Left/Right 限位”或“两侧 False + one-hot Middle”才接受现场叉真值：`f3_stacker_control.py:511-546`。
- 双叉、线圈/限位不符、陈旧 `fork_hold`、全假但非 Middle、读回失败时：不写 C2/C3、不写 Lift，只尝试独立清输送带并返回失败：`f3_stacker_control.py:551-572`。
- stop 从不自动回抽叉；持叉时强制保留 Lift：`f3_stacker_control.py:548-572`。
- 最终再验证 Target/Moving、输送带、叉输出与限位；任何不确定均抛错，由外层记录持久 fault：`f3_stacker_control.py:573-618,1366-1407`。

### 1.5 B5：Windows 单写者与竞态闭合

- 同线程可重入、同进程跨线程互斥、跨进程 byte lock；锁覆盖“读状态→判锁→整个命令→状态完成”：`fault_lock.py:94-179,319-336`。
- 锁文件用 `a+b` 原子 open-or-create，避免双首启 `r+b→w+b` 截断窗口：`fault_lock.py:94-105`。
- 本地锁与 OS 锁共享总 timeout，避免同进程争用无限挂起：`fault_lock.py:142-179`。
- 每次状态提交单调增加 revision，状态写仍使用 fsync + `os.replace`：`fault_lock.py:213-231,263-272`。
- Windows 真实双进程探针：父持锁时子进程 `TIMEOUT/rc=23`；释放后子进程 `ACQUIRED/rc=0`。

### 1.6 H2/H3：write-ahead、强杀锁与跨进程相位链

- `begin_transition` 只接受与 lease 匹配的动作，设备写前持久化 token/action/cell/mode/epoch/base_revision/预期前后态：`fault_lock.py:537-581`、`f3_stacker_control.py:85-92,1450-1480`。
- 任意低层动作写还会再次检查 pending 的 `command_mode`；只有同一外层租约能继续：`fault_lock.py:390-406`。
- 正常结束仅能由相同 token + 相同 command mode 清 pending；`stop` 虽是锁态白名单，也不能清理被强杀的 diagnose pending：`fault_lock.py:584-614`。
- phase 状态只允许 diagnose lease、匹配 token/pending 后写入；下一进程 `place-extend/place-lower/retract` 消费精确的 phase/cell/load/fork/lift 链：`fault_lock.py:460-533`、`f3_stacker_control.py:1247-1313`。
- 动作后的 RuntimeError/Modbus/Stop/E-stop/KeyboardInterrupt 等 BaseException 都进入保守 stop；保留 pending，并记录 `COMMAND_ABORTED` 或 `SAFE_STOP_UNCONFIRMED`：`f3_stacker_control.py:1366-1407,1503-1511`。
- 进程强杀无法运行清理，但 write-ahead pending 已 fsync，下一进程看到 pending 即 fail-closed。

### 2.3 外部故障注入复核（临时目录）

在没有 Claude 新签批次、也不触碰 Factory I/O 的前提下，追加六个真实异常序列并固化到 `test_fault_lock.py`：

1. 子进程 `begin_transition` 落盘后 `os._exit(97)`：pending 保留、diagnose 被挡、stop 不能跨 mode 清除、OS byte lock 自动释放。
2. 子进程持 byte lock 时 `os._exit(98)`：Windows 在进程死亡后释放锁，下一进程可取得租约。
3. command lease/pending 已建立后，模拟外部删除 state：低层写因找不到匹配 write-ahead 被拒，缺失 state 继续 fail-closed。
4. audit 介质抛 `OSError`：reset 不提交解锁；transition completion 不清 pending；fault 当前态在 audit 抛错前已提交为 locked。
5. 真实子进程在 action lease 内并发 `record_fault`：先阻塞，动作租约释放后 fault 成为最终锁状态。
6. 真实子进程在 reset lease 内并发 `record_fault`：先阻塞，reset 租约释放后 fault 成为最终锁状态。
