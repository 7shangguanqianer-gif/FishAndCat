## N5-34｜高｜oracle 的时间百分比在“失败更多”时会显示成超过 100% 的 attainment

- oracle 真正按 `(fail, exp_t)` 词典序选最优：`sim/oracle_gap.py:92-94`。
- 但随后无条件只按时间计算 `gap=(t-vbs)/vbs` 与 `attain=vbs/t×100`，再另列 `excess_fail`：`:95-107`。
- 全量 3900 行中有 58 行 `gap_pct<0` 且 `attain_pct>100`；58/58 都是 `excess_fail>0`，没有一行是同失败数下真正快过 oracle。
- 聚合表出现：dense_200 TOB `gap=-21.515% / attain=128.04% / excess_fail=3.53`；dense_240 `-18.681% / 123.00% / 9.70`；ALL `-3.092% / 103.93% / 1.02`。

裁决：算术和聚合代码一致，但“oracle attainment”这个展示在不可行/失败更多的候选上没有序关系，>100% 会让读者误以为超过 oracle。建议：`excess_fail>0` 时把 gap/attain 置 `N/A (more failures)`；或先分失败档，再只在同一失败档计算时间 gap。任何 98.5%/99.9% 话术也必须同时带 `excess_fail=0` 条件。

## 建议验收门

1. 每个 CSV 只允许一个 schema；另附 `params.json` 或 provenance manifest。
2. CI 扫描：非空、矩形、主键唯一、数值有限、detail↔agg 基数与聚合重算。
3. 所有 percent 字段声明合法域；oracle 指标在 `excess_fail>0` 时禁止生成正常百分比。
4. 日志头写 script hash/mtime、scenario count、seed count；脚本更新后旧日志自动判 stale。
5. 打包前从当前源码一次性重生 CSV、日志、图、DOCX/PPTX，禁止混用不同日期产物。
