## 二、已独立验证成立的部分

### 2.1 CB `min(start,len-1)`：默认口径逻辑等价

Python 每件货先重建当前空闲序，再从 `min(start,len(order)-1)` 开始找，失败后从头回退：`sim/strategy_lib.py:112-120`。ST 对当前未占位计数，正常从 `iSkip` 起找；若剩余空位数不大于 `iSkip`，先试最后一个空位，再从头回退：`plc/04_FB_Warehouse.st:429-465`。

逐分支对照结果：空序、`len<=start`、`len>start`、尾位不可行、从头回退、最终失败六类语义一致。另做只读差分探针：

- `rule ∈ {sum,and}`；
- `n ∈ {1,3,20,60,120,200,260}`；
- 每组 30 seeds，共 **420 个实例**；
- 终局位置和失败 ID：**420/420 一致，mismatch=0**。

因此，**默认固定运动学和正常唯一 ID 输入域内**，未发现 CB 钳位实现错误。

## 三、需修发现

### R1【高】唯一 ID 契约只守住交互入口，演示批加载可直接制造重复 ID

**证据链：**

- `FB_GoodsInput` 拒绝 `iId<=0` 和已存在 ID：`plc/04_FB_Warehouse.st:1025-1041`；主程序也处理 `iErrId=4`：`plc/05_PRG_Main.st:115-117`。
- 但 `FC_LoadDemoGoods` 只检查容量，然后无条件追加固定 ID 1..20：`plc/07_GVL_Data_generated.st:139-190`；模板来源是 `sim/export_st_vectors.py:149-166`。
- 主程序允许在 `iGoodsCount+20<=400` 时反复追加：`plc/05_PRG_Main.st:120-133`。第二次点击即可得到两套 ID 1..20。
- T61 只测“395 件拒载、空仓正常载入”：`plc/06_PRG_Test.st:538-545`；T70 只测 `FB_GoodsInput`：`:750-770`。没有“第二次 LoadDemo 必须拒绝/重映射”的断言。

**影响：** `FB_QueryGood` 命中第一个同 ID 即退出：`plc/04_FB_Warehouse.st:1104-1129`；两个格可以同时写同一 `GoodsId`，查询、快照解释和 Python 的 `cls_of[g.gid]` 字典语义均变得不唯一。容量护栏解决了越界，但没有解决身份契约。

**建议：** 最小安全方案是仅允许 `iGoodsCount=0` 时载演示批；若必须支持追加，则加载前事务式扫描冲突并拒绝，或生成全局唯一新 ID。补“连续载入两次”测试。

### R2【高】`FB_Stats` 没有实现规格中的“双向 + 一货一位”一致性扫描

规格要求 `Occupied↔GoodsId` 双向和一货一位：`docs/ST补丁计划_轨B_0712.md:98`。当前实现只从每个 `Assigned` 货物检查其坐标格是否 Occupied、GoodsId 是否等于自身，再查承重/容积：`plc/04_FB_Warehouse.st:690-715`；后面的全仓循环只统计 `nUsed`：`:727-733`。

以下静态反例都会得到 `ViolCnt=0`：

1. 某非预占格 `Occupied=TRUE, GoodsId=999`，但货物表没有 999（幽灵格）。
2. 某空格 `Occupied=FALSE` 却残留非零 `GoodsId`。
3. 两件 `Assigned` 货物使用同一业务 ID，分别位于两个都写该 ID 的格——每件的单向检查都通过。
4. 同一货物 ID 出现在两个占用格；只要货物记录指向其中一个，另一个不会被扫描为违规。

T72 只把货物记录改到无货格，覆盖单向 `good→slot`：`plc/06_PRG_Test.st:795-802`，不能证明双向或一货一位。

**建议：** 加 slot→good 扫描并显式处理合法预占格（`PresetOcc=TRUE, GoodsId=0`）；同时统计每个 Assigned 货物的格匹配数必须恰为 1。R1 修复后再锁唯一 ID。

### R3【高】批次参数快照规格未落全；CT 的“快照”不保护 TOB 和最终统计口径

规格要求建立完整运行参数快照，并让 Assign/Improve/新 FB 全程只读快照：`docs/ST补丁计划_轨B_0712.md:71-76,97`。当前有三处偏差：

2. CT 只拍 `rVx/rVy/xSimultaneousXY`：`plc/04_FB_Warehouse.st:282-317`。TOB 根本不使用这份序表，而是每件调用读实时参数的 `FB_SelectSlot`：`:416-427`；该 FB 最终进入 `FC_CalcTravelTime/FC_CalcScore`，后者读实时 GVL：`plc/03_Functions.st:126-142,211-220`。

## 四、REAL 平局与 CB 类界：AB 已证明和未证明的边界

### 4.1 已证明

T60 的默认参数均为可精确表示的二进制值，构造 `(0,1)` 与 `(4,0)` 同为 2.0 秒，并断言按 `(time,tier,col)` 选择 `(4,0)`：`plc/06_PRG_Test.st:518-535`。最后一轮现存 AB 日志 75/0 包含该断言。因此，**默认参数这一条精确等时向量已 AB 实测**。

### 4.2 未证明；存在可构造的 REAL32 舍入反例

只读 IEEE-754 REAL32 代理探针找到：

- `rVx=2.5379269123`、`rVy=2.9993681908`，切比雪夫档；
- 横向格 `(11,0)` 的 Python 双精度时间 `4.3342461702`；纵向格 `(0,13)` 为 `4.3342461389`，Python 应选 `(0,13)`；
- 两者在逐步 REAL32 运算后都为 `4.3342461586`，ST 进入“相等”分支，再按低层优先选 `(11,0)`。

这不是 AB 实测故障，也不影响默认 2.0/0.5 锚；它证明 `rT=rBestTime` 的安全域仅覆盖当前固定参数/已测向量，不能外推到任意管理员速度。

CB 类界也有同类输入量化边界：频次 `[7.0, 2.9999999]` 在 Python 双精度下首件略高于 70%→B，而第二个数进 PLC REAL 后可量化为 3.0，使首件恰为 70%→A。T63 明确把频次放在远离 70%/90% 的位置：`plc/06_PRG_Test.st:556-560`，所以现有 AB 不覆盖阈值邻域。

**建议：** 不建议简单加 epsilon 后仍声称与 Python 全等。先定义跨端输入量化契约：Python 验证入口把 HMI 参数/频次量化为 float32，或限制为受支持的速度预设；随后为每个预设加入 near tie 和 70%/90% 邻域黄金向量。
