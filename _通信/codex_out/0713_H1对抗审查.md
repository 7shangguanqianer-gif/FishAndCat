## 3. 阻断级发现

### B1. 删除或合法降级 `fault_state.json` 可直接解锁

证据：

- 文件不存在时 `load_state()` 返回空字典：`l2_factoryio/fault_lock.py:45-53`。
- `is_locked()` 只判断非空 `fault` 或 `corrupt`：`l2_factoryio/fault_lock.py:82-83`。
- 测试明确把“文件缺失”锁为预期的 unlocked：`l2_factoryio/test_fault_lock.py:37-38`。
- `record_fault()` 虽写入 `cargo_trust=False`：`l2_factoryio/fault_lock.py:97-121`，但 `cargo_trust` 不参与锁判定。

只读临时目录探针结果：

| 状态 | `is_locked` |
|---|---:|
| 文件缺失 | `False` |
| `{"cargo_trust": false}` | `False` |
| `{"fault": {}}` | `False` |
| 正常非空 fault | `True` |
| 损坏 JSON | `True` |

影响：故障后删除文件、覆盖为结构合法但语义降级的 JSON，或部署到没有该运行文件的新副本，均可绕过“唯一路径 reset-epoch”。`.gitignore` 当前已包含 `fault_state.json`，这能避免误提交，但不解决运行时删除/丢失的 fail-open。

要求：运行期一旦启用安全锁，文件缺失、schema 不完整、版本不匹配、`cargo_trust != True` 均应 fail-closed；首次初始化必须是显式流程。若把本地篡改纳入威胁模型，单个同目录 JSON 不能作为安全边界，应使用 PLC/受控守护进程或受权限保护的状态目录。

### B2. “全局故障锁”实际上只覆盖一个 CLI 入口

证据：

- `_assert_unlocked_for_action()` 只在 `f3_stacker_control.py` 的 `main()` 开头调用一次：`l2_factoryio/f3_stacker_control.py:1201-1213,1254-1279`。
- `f3_mechanics_probe.py` 直接实例化 `Stacker` 并写 Lift/Fork，只有 `--confirm-empty`，没有锁检查：`l2_factoryio/f3_mechanics_probe.py:73-111,157-190`。
- `webapi_probe.py emitter-pulse` 可直接强制 Emitter 生成货物，也没有锁检查：`l2_factoryio/webapi_probe.py:142-164,167-188`。
- 任何 Python 调用者也可直接实例化 `Stacker` 调动作方法，绕开 CLI 门。

影响：H1 文案“锁下只许 snapshot/stop/mark-fault”只对 `f3_stacker_control.py main` 成立，不是设备级全局约束。当前实时状态已是 `RECOVERY_UNSAFE`，但 mechanics probe 仍可写机构，emitter pulse 仍可污染场景。

要求：把锁检查下沉到唯一的写入适配层/命令租约，所有 Target、运动线圈、Emitter-on 写入都必须经过同一守门；至少对 mechanics probe 与 webapi emitter-pulse 补锁。`emitter-off` 可作为保守撤险特例，但必须明确白名单。

### B3. `reset-epoch` 不验证应用 epoch，且“空场 preflight”可被非空场骗过

证据：

- “完整重启”只是 CLI 布尔旗标 `--operator-confirmed-restart`：`l2_factoryio/f3_stacker_control.py:1087-1096`。
- `clear_for_new_epoch()` 只接收两个布尔值，没有旧/新进程、Factory I/O session、场景 GUID 或启动时间的比对：`l2_factoryio/fault_lock.py:157-183`。
- `_run_reset_epoch()` 的在线门只调用 `check_safety_inputs()` 与 `assert_action_ready()`：`l2_factoryio/f3_stacker_control.py:1216-1229`。
- `assert_action_ready()` 只检查 Moving X/Z、叉限位、C0..C6 与 Target；不检查 `At Entry/Load/Unload/Exit`：`l2_factoryio/f3_stacker_control.py:221-241`，货物输入定义见 `:65-66`。

只读 Mock 探针把四个 active-low 货物传感器全部设为 `False`（均表示有货），同时维持 Middle/全零输出/Target 0；当前 preflight 仍 `PASS`，并把内存 `load_state` 设成 `EMPTY`。

影响：未完整重启或场内仍有货物时都可能清锁。货架内、承载台、地面散落物本来就没有完整传感器覆盖，单靠当前 I/O 更不可能证明空场。

要求：

1. 若拿不到可信 Factory I/O epoch，就把“完整重启”明确降级为人工程序性声明，不能称程序验证。
2. 至少检查四个货物传感器、Emitter 强制关闭、Target/线圈/限位和稳定时间。
3. 对叉上、货架、地面/跌落区要求可追溯视觉/API 证据；无法证明就保持锁定。
4. reset 必须取得独占命令租约，防止清锁时另一进程正在落新故障。

### B4. 新进程 `safe_stop()` 读回失败时，连 `Target=0` 都不会先尝试

证据：

- `fork_hold=None` 时先 `snapshot()`：`l2_factoryio/f3_stacker_control.py:449-467`。
- snapshot 异常在 `:467-471` 直接抛出。
- `self.target(0)` 在其后才执行：`:472-477`。
- 现有测试只断言没有线圈写入，没有断言 Target 已尝试清零：`l2_factoryio/test_fault_lock.py:187-199`。

探针结果：snapshot 抛 `link down` 时，`target_calls=0`，`coil_calls=0`。

影响：读路径瞬时失败而写 Holding Register 仍可能可用时，软件跳过唯一应最先尝试的 X/Z 停止。通信完全失联时软件当然不能保证物理安全，但仍不应因为叉状态未知而放弃独立的 Target 停止尝试。

要求：先独立 best-effort 写 `Target=0`，再读取叉状态；读取失败时禁止任何叉/Lift 猜测性写入并落持久故障。若 Target 写也失败，明确升级为“软件无法确认停机，需外部 E-stop/安全链”，不能打印安全停机成功。

### B5. 安全状态文件与动作之间没有进程互斥，存在丢锁和 TOCTOU

证据：

- `record_fault()` 与 `record_phase_state()` 都是“读 JSON → 修改内存 → `os.replace`”：`l2_factoryio/fault_lock.py:97-153`。
- 原子替换只防半写，不能防两个进程的 lost update。
- `clear_for_new_epoch()` 同样读取旧状态后整体覆盖：`l2_factoryio/fault_lock.py:157-183`。
- CLI 只在动作最前检查一次锁：`l2_factoryio/f3_stacker_control.py:1254-1279`；检查后另一进程可落故障，当前进程仍继续写机构。

可复现竞态序列：进程 A 读取“无 fault”准备写 `last_phase`；进程 B 写入 fault；A 随后用旧快照 `os.replace`，把 B 的 fault 覆盖掉。reset 与新 fault 也有同类竞态。

要求：用 Windows 可用的进程互斥/文件锁覆盖“读状态—判锁—整个动作事务—写完成态”；状态增加单调 `revision`/CAS，冲突时 fault 胜出。单纯保留 `os.replace` 不够。

## 4. 高严重度发现

### H1. `safe_stop` 对双叉与陈旧 `fork_hold` 会先做危险写入、后报错

证据：

- 新进程只看 C2/C3，循环遇到第一个 True 就 `break`：`l2_factoryio/f3_stacker_control.py:456-466`。
- 随后把除“选中的一侧”之外的运动线圈写 False：`:478-490`。代码自己承认 Fork=False 会主动回中：`:453-455,480-481`。
- 若内存 `fork_hold` 已是 C2/C3，初始 snapshot 被完全跳过：`:449-452`。
- 最终读回只检查输出/Target，不检查叉限位 one-hot 或 X/Z 已真正停止：`:492-509`。

探针：

- C2+C3 双真时选择 C2 保持并写 C3=False，可能主动回抽右叉。
- 内存称 C3 保持、实际 C2=True 时，代码写 C2=False；最终虽然报 held C3 false/active mismatch，但危险回抽已经发生。

要求：只有“恰好一个叉保持线圈为 True，且与 one-hot 叉限位一致，连续两次稳定快照一致”时才允许保留；双真、全假但非 Middle、线圈/限位不一致、内存/现场不一致，一律不改任何叉输出，落 `FORK_STATE_AMBIGUOUS`。Target 0 后还应等 Moving X/Z=False，再考虑非必要输出。

### H2. 三处 Lift 故障锁只覆盖字符串 `NO START`，TIMEOUT/E-stop/Modbus/强杀均不落锁

证据：

- pick：`l2_factoryio/f3_stacker_control.py:685-710`。
- relift：`:723-755`。
- place-lower：`:842-877`。
- 三处都只在 `"NO START" in str(exc)` 时调用 `record_fault()`。
- `wait_motion_cycle()` 在已看到 start 后仍可能在 settle 阶段 TIMEOUT：`:416-438`。
- Stop/E-stop 在 `check_safety_inputs()` 抛异常：`:159-165`；主异常处理只尝试 `safe_stop`，不一定写 fault：`:1296-1308`。

探针确认：place-lower 的 started-but-not-finished `TIMEOUT` 产生 `fault_calls=0`；observe 窗口发生 E-stop 也 `fault_calls=0`。

影响：Lift/Fork 已被写入后，只要故障形态不是恰好 `NO START`，下一进程可能看到缺失/无 fault 状态而继续。进程强杀更不会执行异常处理。

要求：执行器写入前写 durable `transition_pending`（动作、cell、预期前/后态、epoch、revision）；只有真实 settle 与视觉门完成后才清 pending。执行器写入后的任何异常/取消/Stop/E-stop/KeyboardInterrupt/下次启动发现 pending，都必须 fail-closed。

### H3. G4 跨进程窗口没有可消费的事务状态

正面事实：

- `place` 融合入口已禁用：`l2_factoryio/f3_stacker_control.py:777-782`。
- `place-extend` 只伸 Right，不写 Lift：`:784-807`。
- 新进程 `place-lower` 会复核 Right、静止、Lift=True、C3=True、Target 0，并要求 `--confirm-clearance`：`:809-840,1139-1152`。
- 成功后 `_finish_diagnostic()` 保留 Right hold：`:1184-1198`。

缺口：

- `record_phase_state()` 在相位完整成功后才写：`:1281-1287`；在 C3 写入后、记录前强杀不会留下 intent。
- 写入的 `lift_command=None`，无法证明交接时 Lift 状态：`:1281-1287`。
- `last_phase` 目前只写不读：`l2_factoryio/fault_lock.py:125-153`；控制器没有把它作为授权条件。
- observe 允许 GUI Pause，仅监视 Stop/E-stop：`l2_factoryio/f3_stacker_control.py:982-992`；但输出保持、Modbus 断连、Stop/E-stop 后恢复的物理语义没有现场证据。

因此正常拆门流程比旧版安全，但非计划退出后并没有可审计的恢复语义。应采用 write-ahead intent/completion，并把 session/epoch 连续性纳入下一相位授权。

### H4. fault 日志不是追加审计链，后写可覆盖前一故障

`record_fault()` 只保存一个 `state["fault"]`：`l2_factoryio/fault_lock.py:109-121`；`clear_for_new_epoch()` 也只搬运一个 `cleared_fault`：`:175-183`。多次 `mark-fault` 或新的自动故障会覆盖旧故障细节。对调试可用，但不能称完整审计证据。

要求：把当前锁状态与 append-only JSONL 事件分开：锁文件用于当前事务，事件日志记录每次 intent/fault/clear/actor/revision，不能相互覆盖。

## 5. 中等级/设计冲突

### M1. 故障锁与 recover-lift/recover-retract 的可达性冲突

place-lower `NO START` 会立即落 persistent fault；而 `_assert_unlocked_for_action()` 会阻止所有 diagnose，包括 recover-lift/recover-retract：`l2_factoryio/f3_stacker_control.py:1201-1213`。因此这些恢复命令在它们最需要处理的故障后无法由新进程到达。

两种路线必须选一条：

1. **推荐：完全重启/清场策略。** 删除或明确废弃现场恢复命令，任何带载未知状态都只允许外部安全处置。
2. **受控恢复策略。** 在独占锁、固定恢复状态机、人工双确认和 write-ahead journal 下只开放特定恢复动作；不能把一般 `diagnose` 解锁。

当前两套语义并存会让操作者误以为可以恢复后继续。

### M2. `recover-lift` 允许无 Z 边沿，与 `LIFT_POSITION_UNKNOWN` 的严格门不一致

`recover_place_hold_lift()` 在 START_TIMEOUT 内无 Z 时仍打印“may already be raised”并继续：`l2_factoryio/f3_stacker_control.py:884-933`。这在已有可靠实体高度反馈前不能证明 Lift 已抬起。若采用完全重启策略，应移除；若保留恢复策略，必须用可信位置反馈或视觉双证据，不得仅凭 Lift 命令线圈。

## 7. 修复优先级

### R0：立即停止带货自动试验

当前 H1 不能作为 H3 安全门。只允许只读取证、保存现场、人工外部安全处置；不要用现有 `reset-epoch` 解锁继续旧货。

### R1：重写 `safe_stop`

1. 独立 best-effort `Target=0`。
2. 等待/验证 Moving X/Z=False；失败即升级外部安全链。
3. 双快照校验叉线圈 + one-hot 限位。
4. 任何歧义都不写叉/Lift，落持久故障。

### R2：把锁变成设备级单写者事务

进程互斥/租约 + schema/version + revision/CAS + 缺失 fail-closed；守门下沉到所有写入路径。fault 优先级高于 phase/reset。

### R3：引入 write-ahead 机构事务

每次 Lift/Fork/Target 写入前记录 pending；完成后写 completion；强杀或 session 变化后只能进入受控恢复/清场。

### R4：重做 reset epoch 证据

可信 session/启动标识（能拿到则验证）+ 全货物输入 + Emitter off + 场景/视觉证据 + 独占锁。拿不到 session 标识时，文案只能称人工确认。

### R5：补测试后再开一次单箱、空载优先现场门

先验证 safe_stop/Stop/E-stop/Pause/断连，不直接进入 9/9 带货。任何异常立即锁定并保存证据。
