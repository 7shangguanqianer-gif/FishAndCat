## T2.1 AWRA-LS 移植的五条施工裁决

1. **FB 接口 = PLCopen 通用模式**:Execute(上升沿触发)/Busy/Done/Error/ErrorID(WORD 枚举)。
   四输出互斥;Execute 边沿适配"一次触发、跨周期跑完"的批量算法(枪K,Beckhoff infosys
   已核对)。报告表述写"借鉴 PLCopen 通用 FB 设计模式"——**不写"运动控制标准"**(避误导)。
2. **长计算 = 状态机分片**:Busy 期间每扫描周期处理固定份额(如 10 件/周期),进度存 FB
   静态变量。依据:看门狗超时=任务挂起(枪G);eCo 级 CPU 浮点 1000 条 1.2ms,200×400
   评分循环单周期跑完会爆(枪J 官方 datasheet)。
3. **大数组传参 = VAR_IN_OUT**(传引用零拷贝,CODESYS 官方已核对;原地改写=排序/局搜
   天然形态;位类型不可传、常量不可传两条限制记入编写标准)。
4. **排序 = 自实现插入排序+分片**:标准库/PLCopen/厂商均无排序库(枪G 交叉确认不存在);
   200 元素插入排序最坏 ~2 万比较,分片下周期可控;选择理由=确定性上界+代码可审查性,
   希尔排序无 PLC 实测数据不冒进(桌面 CS 结论不能直接搬,枪G 提醒)——报告可写这个
   工程判断本身。
5. **CPU 型号(0707 官方答复关闭)**:ABB 工程师答复"eCo 与 AC500 标准型均可选,
   建议选 AC500 较合适,内存空间大"→ **按标准型 PM56xx 级设计**,eCo 10kB 紧箍解除:
   数据结构可读性优先(REAL 字段无妨,PM56xx 带 FPU,枪G"定点化无必要"结论适用);
   状态机分片**保留**(看门狗与任务实时性是任何 CPU 的通用约束+PLCopen 模式自然形态)。
   报告写"依 ABB 工程师建议选用 AC500 标准型(PM56xx),并说明 eCo 兼容性考量"。
   仿真器不保证真实时序(枪J)→性能结论按指令周期表最坏估算,报告如实声明无真机。

## T2.2《ST编写标准》的骨架裁决

- **前缀体系:沿用现有 Beckhoff 惯例系**(b/i/r/s/a/st;现有 plc/ 全库已用 iPassed/
  bBatchKnown 风格)。合规依据:PLCopen 官方"前缀方案可自定义,但必须文档化"(枪H)
  ——《标准》第一条就是这句+我们的前缀表。注意 PLCopen 自家示例是 x/i/r/str(BOOL=x
  非 b),两套不可混用,文档里写明选型理由(存量一致性)。
- 头部注释块模板 = 已拍板四件套(功能/IO 表/边界/示例)+ Beckhoff 版本历史表样式
  (枪H 模板 A);Siemens 风格指南的"规则/建议"两级分类法借用为条目权重。
- FB vs FC 判据:跨周期状态→FB;纯函数→FC(现有 03_Functions 全 FC 判定正确)。
- PLCopen Coding Guidelines v1.0(60+ 条,CP/N/C/L/E 五类)为骨架引用;**PDF 机读失败,
  逐字引用前需人工打开**(下载 URL 在枪H 报告);ABB/CODESYS 无自家 ST 规范(枪H 确认)
  →自建标准合法且必要。
- 每 POU 行数上限:PLCopen 未找到具体数字→用定性原则(单一职责),不编数字。

## T2.3 边界用例 40+ 的设计依据

- **CheckBounds 默认静默钳位**(不崩溃不报错;CODESYS+ABB 官方双源已核对,枪I)——
  越界不会被运行时抓,**显式输入验证是必须而非可选**,这句话+官方 URL 直接进报告。
- 隐式检查家族:CheckBounds/CheckDivInt/CheckDivReal/CheckRangeSigned/CheckPointer
  (官方清单已核对);生产环境开启有 CPU 代价(摘要级)→我们走显式防御主线。
- REAL 除零=INF/NaN 不抛异常且"结果依目标系统而异"(官方已核对)→除法前显式判零
  (现有 sim 的 `or 1.0` 模式移植为 ST 显式 IF)。
- **整数溢出行为未确认(专项缺口)**→设计上回避:中间量用 DINT+显式范围检查;
  测试加溢出探针用例。
- 错误接口统一:bError + wErrorID(枚举 DUT)——PLCopen 主流形状(枪I);
  sErrorMsg 只做调试扩展非主接口。
- 用例组织借 counit 惯例:每 FB 配 `_Test` 后缀测试 FB,用例名=场景_预期
  (如 `Overweight_RejectedWithErrId3`),边界维度:库位 0/1/400/401、货物 0/1/199/200/201、
  满仓、零货、超重、超容、非法坐标、重复入库、评分并列、容量耗尽。
