# N5-P7 权限、多客户端与审计留痕复核

结论：新增 4 项问题、正面确认 1 项。当前 A4 是“单个全局管理员位 + 全局 PIN/命令 + 全局参数守卫”的单客户端演示机制，不是按 HMI 客户端隔离的用户权限系统。源码注释已经如实承认这一边界，但测试与部分报告仍把单实例功能测试概括成“权限安全”，容易被误读为多用户隔离已经验证。

范围：只读静态跟踪 `GVL_Visu → PRG_Main → FB_UserAuth/FB_ParamGuard/FC_MaintToggle`，构造双客户端交错时序；未开 Automation Builder，未连接 WebVisu，故以下是代码必然语义，不冒充 AB 动态实测。

## N5-27｜高｜一个客户端登录会给所有客户端开放同一个全局管理员位

- 权限输入与输出都在 `VAR_GLOBAL`：`plc/02_GVL.st:187-192`。
- 主程序只有一个 `fbAuth` 实例，并把其 `iLevel` 回写全局 `GVL_Visu.iUserLevel`：`plc/05_PRG_Main.st:198-206`。
- 参数守卫和检修权限门都只接这一个全局级别：`plc/05_PRG_Main.st:206-210`；`plc/04_FB_Warehouse.st:1065-1078`；`plc/03_Functions.st:269-277`。

必然结果：客户端 A 正常登录后，客户端 B 看到并继承同一个 `iUserLevel=1`；B 的参数写入和检修命令也以管理员身份执行。客户端 B 登出又会把 A 一并登出。这里不存在 client/session owner 字段。

裁决：在一台工程站、一个演示客户端的限定下可用；一旦启用多个 WebVisu/本地 visu 客户端，就不具备会话隔离，不能称“多用户权限”。

## N5-28｜高｜共享 PIN 与共享脉冲命令允许跨客户端消费和竞态

双客户端静态时序：

| 扫描前事件 | PLC 本扫描读取 | 结果 |
|---|---|---|
| A 写 `InPinCode=2026`，尚未点 Login；B 写全局 `CmdLogin=TRUE` | `fbAuth(xLogin=CmdLogin,iPin=InPinCode)` | B 的命令消费 A 的 PIN，全局升为管理员 |
| A、B 在同一扫描前分别写不同 PIN/坐标/参数 | 只有一组 GVL 值 | 最后一次网络写覆盖前一次，无 actor 归属 |
| 任一客户端写 `CmdLogout=TRUE` | 全局 `fbAuth.xLogout` | 所有客户端同时失去管理员位 |

证据：共享输入/命令 `plc/02_GVL.st:178-192`；PIN 在任一登录/登出命令后统一清零 `plc/05_PRG_Main.st:199-205`；命令在段尾统一消费复位 `plc/05_PRG_Main.st:248-264`。

裁决：“PIN 消费即清”能减少单客户端残留，但不能提供多客户端保密或归属；相反，它使跨客户端消费成为确定行为。

## N5-29｜高｜现有 T54–T56 只验证单实例分支，不能证明客户端隔离或端到端权限安全

- T54 直接调用单个 `fbAuthT`，依次测错 PIN、正确 PIN、登出：`plc/06_PRG_Test.st:430-439`。
- T55 直接把 `iLevel:=1/0` 传给 `fbGuardT`，未经过 GVL、HMI 客户端或并发写入：`:440-450`。
- T56 直接传常量 `iLevel` 给 `FC_MaintToggle`：`:452-463`。

这些测试能证明功能块的单线程分支逻辑正确，却没有覆盖：A/B 客户端交错、PIN 所有权、会话超时、并发登出、全局管理员位对其他客户端的扩散，也没有覆盖 N4 已发现的“守卫调用晚于部分消费者”副作用。

裁决：`59/0` 可称“单实例权限函数/回滚分支通过”，不可单独支撑“权限安全已验证”或“多客户端可用”。

## N5-30｜中｜没有会话超时、有效锁定、身份审计或参数变更审计

- `FB_UserAuth` 只有 login/logout edge、`iLevel`、累计失败数：`plc/04_FB_Warehouse.st:1027-1057`；没有时钟、超时、失败延迟/锁定、用户 ID、客户端 ID 或角色表。
- `iFailCnt` 只递增，不参与拒绝逻辑；P3 已证明它还是 16-bit `INT` 且无饱和。
- 管理员参数变化被 `FB_ParamGuard` 直接吸收为新快照：`plc/04_FB_Warehouse.st:1078-1094`，不记录 actor、时间、字段、旧值、新值。
- 检修只保留最终布尔状态和当前数量：`plc/03_Functions.st:281-288`；`aSlotOps` 只是一格一个计数，也不含谁/何时/何动作：`plc/02_GVL.st:80-83`。

裁决：当前“留痕”是状态/热度，不是安全审计日志。演示材料应称“演示 PIN + 单客户端回滚守卫”；若进入真实部署，至少要接 User Management、每客户端会话、超时/锁定和 append-only 变更日志。

## N5-31｜正面确认｜源码对演示边界的自我声明是准确的

- `plc/02_GVL.st:23-25` 明说 PIN 是演示码、非真实凭据，生产化换 visu UserMgmt。
- `plc/04_FB_Warehouse.st:1020-1026` 明说 `iLevel` 非 RETAIN、单客户端演示、多客户端需 `CurrentClientId` 隔离。
- 断电后管理员位回到 0 的 fail-safe 默认也是合理的。

裁决：不要删除这些限定；应把它们同步到报告、答辩 QA 和部署说明。问题在于上层叙事不能比源码声明更强。

## 建议验收门

1. 单客户端演示：明确限制只允许一个 HMI 会话，并在演示前关闭其他 WebVisu 标签页/客户端。
2. 多客户端部署：用平台 User Management 或以 `CurrentClientId` 为键的会话表；PIN、命令、角色与超时均按客户端隔离。
3. 所有有副作用的命令在同一扫描内绑定 actor/session，并在执行点二次鉴权，而不只依赖画面锁。
4. 测试至少加入 A/B 交错序列：A 输入/B 登录、A 登录/B 改参、B 登出/A 操作、并发不同 PIN、超时后命令。
5. 审计日志记录 timestamp/client/user/action/target/before/after/result；热力计数不能替代安全日志。
