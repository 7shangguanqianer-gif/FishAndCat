# N5-P1 浮点精度与决胜规则独立复核（0712）

## 结论

新增 3 项发现、1 项正面确认。最严重的不是 float32，而是 **near 平局规则两端并不一致**：Python 按 `(time,tier,col)`，ST 实际按严格时间比较后保留 `col→tier` 扫描先者，即 `(time,col,tier)`。全量 13 场景×30 seeds 的同货物反事实中，390/390 个运行都出现布局分歧；期望取货时间因平局同程而完全相同，但能耗代理最大相差 12.18%。

本批未开 AB；REAL32 结果是明确标注的 Python 单精度代理，不是 PLC 实跑。

## N5-01【高】near 平局规则 Python↔ST 相反，当前测试只覆盖唯一最短位

### 原始证据

- Python `strat_near` 明确按 `(travel_time, tier, col)`：`sim/warehouse_sim.py:278-283`。
- ST 扫描顺序是列外层、层内层：`plc/04_FB_Warehouse.st:86-88`；near 分支只做 `rT < rBestTime`：`:100-102`。相等时不替换，所以次级顺序实际是较小列优先。
- ST 注释却写“时间、层、列”并称与 Python 一致：`plc/04_FB_Warehouse.st:59,101`，与执行代码矛盾。
- T37 只测全库唯一最短的 `(1,0)`，没有构造等时平局：`plc/06_PRG_Test.st:242-248`；T47 同样只测唯一最短及其检修绕行：`:369-377`。

### 最小反例

匀速口径下 `(0,1)` 与 `(4,0)` 都是 2.0 s。若只剩这两个可行位：

- Python 因 tier 0 更低，选 `(4,0)`；
- ST 因 col 0 先扫描，选 `(0,1)`。

## N5-02【中】现有一致性探针没有真正模拟 REAL32；分层样本已出现小分叉

### 原始证据

- 所有评分、优先级和统计字段均声明为 `REAL`，例如 `plc/01_DUTs.st:15-24`、`plc/03_Functions.st:187-213`。
- ABB 官方 Automation Builder 帮助明确：REAL 占 32 bits，LREAL 占 64 bits：<https://help.plc.abb.com/REAL_LREAL.html>。
- Python 默认 float 为 64-bit；当前 `consistency_probe.py` 只是切 `tie_break` 和 `ls_eps`：`sim/consistency_probe.py:39-64`，没有把每个 ST 临时值量化成 binary32。
- `export_st_vectors.py:1-9` 知道“双精度 vs PLC REAL 单精度”，但测试只以 `1e-3` 容差比单点值，不能证明高密度终局布局不分叉。

### 量化

用“每个具名 ST 临时值/结果转 IEEE-754 binary32”的保守代理，对 13 个场景各取 seed=2026 做 AWRA 端到端分层检查：

- 1/13 运行出现布局差异；
- `illegal_mix/2026` 有 2 件位置不同；
- exp_t 最大差 0.009926 s；
- energy 相对差 0.0717%；
- 13 组 Rank 顺序均未改变；
- 20 件 T25 演示批布局差 0、exp_t 差 0。

### 限定

这是 Python REAL32 代理，不能替代编译器实际表达式提升/舍入与 AB 运行。它证明的是“现有探针的覆盖声明过强”，不是证明 PLC 当前一定有上述 2 件分叉。正确结论应为：20 件黄金向量已由历史 AB 回归锁定；高密度 float32 端到端仍缺 AB 差分门。

## N5-03【高】Rank 平局 Python 按 Goods ID，ST 按数组下标；合法非顺序 ID 也会分叉

### 原始证据

- Python Rank 平局按 `g.gid`：`sim/warehouse_sim.py:323-326`。
- ST 平局比较的是 `aRankIdx` 数组下标，而不是 `aGoods[idx].Id`：`plc/04_FB_Warehouse.st:177-188`。
- HMI 货物 ID 是独立输入字段：`plc/02_GVL.st:108-113`；`FB_GoodsInput` 把它原样写入货物表：`plc/04_FB_Warehouse.st:743-756`。当前没有“ID 必须等于数组下标”的契约。

两个属性完全相同、到达顺序 `[Id=100, Id=1]` 的合法正 ID 反例：

- Python 排序 `[1,100]`；
- ST 按数组位置保持 `[100,1]`。

N4-16 已指出 0/负数/重复 ID 未校验；本项是新的后果：即便 ID 正值且唯一，只要不等于到达下标，两端 Rank tie-break 仍不一致。T25 的生成货物恰好 `Id=index`，因此不会覆盖此分支。

## N5-04【正面】当前演示批对 REAL32 代理稳定

20 件、seed=2026、sum+匀速、ST tie/epsilon 下，双精度与 REAL32 代理的 AWRA 终局布局及 exp_t 均无差异。它支持保留 T25 作为“小批演示锚”，但不能外推到 near 平局或 120/240 件高密度。

## 建议最小修复顺序

1. 明确 near 的权威 tie-break：若以 Python 为真，ST 在等时分支显式比较 tier/col；若以 ST 扫描序为真，修改 Python、历史数字和图表。
2. 新增 near 等时反例向量，不能再只测唯一最短位。
3. Rank 平局统一按 `Good.Id`，或把“ID 必须连续且等于数组下标”做成输入契约与测试；前者更符合现有 Python。
4. 将 `consistency_probe` 的“完整 ST 语义”改成“tie/epsilon 代理”，另加 REAL32 分层探针；决赛前在 AB 跑至少 120/240 件差分。
