# N4-P4 跨机复现、依赖、许可与提交包卫生（0712）

## 总结

当前仓库适合继续开发，不适合直接称“任意机器克隆后一键恢复全部”，也不适合整仓压缩提交。缺少依赖锁、完整产物 DAG、提交清单和第三方许可清单；生成物默认被忽略，而 run_all 又不能补齐它们。

## N4-35【阻塞】没有依赖声明，干净机器只装 Python 3.8+ 无法跑完整底盘

- 全仓未找到 requirements*.txt、pyproject.toml、environment.yml、Pipfile、uv.lock、setup.py 等。
- docs/Codex总对接提示词.md:49-50 却称任意机器装 git+Python3.8+、克隆后跑 run_all 即可重建。
- warehouse_sim.py 本体可零依赖运行，但 run_all 的 plots 步骤至少需要 matplotlib、numpy：sim/plots.py:19-24、79。
- 专项链还需要 scipy、python-docx、python-pptx、Pillow、pyvista、imageio_ffmpeg 等，见 sim/lap_optimal.py:38-42、sim/build_report.py:23-26、sim/build_ppt.py:18-20 及 tools/render_* 导入。

结论：应分别提供 core、report、3D/AB-tools 三档锁定依赖，并记录 Python/包版本。否则干净机复现无法作为验收项。

## N4-36【高】sim/out 与 sim/figs 默认被忽略，而 run_all 又不能全部再生

- .gitignore:4-5 忽略 sim/out 与 sim/figs。
- README.md:36-37 称它们“可删、全部可再生”。
- N4-26 已证明 run_all 只再生基础子集，且不生成报告/PPT/大量专项 CSV/图。
- L4 CSV 还依赖 AB 在线任务周期实验，无法仅靠 Python 在新机器重建。

因此正常克隆不能假设携带正式报告、PPT、oracle 数据、L4 数据和专项图；删除本机 out/figs 也不能靠一条命令恢复。要么建立发布制品包并校验哈希，要么扩展全量再生 DAG。

## N4-41【正面】核心数值脚本的相对路径和确定性种子基础较好
