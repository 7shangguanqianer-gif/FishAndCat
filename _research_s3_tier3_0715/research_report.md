# S3 档3全流程可视化架构研究（2026-07-15）

## 执行摘要

当前 `样张_S3_三方向候选.html` 不是完整档3。严格判定为“档2运动学外壳 + 档3静态标签”，且由于等待队列、被运输货物和图表分别由独立假数据驱动，三者不可能身份一致。右侧镜头失败同样不是微调问题：固定坐标镜头没有按“完整货架 + I/O + 等待队列 + 当前设备 + 当前目标”的动态边界取景。

建议的决赛终态是“方案 C：双幕算法证据 + 档3事件回放”，工程底座采用方案 A 的单一事件总线，交付形式采用方案 B 的确定性离线轨迹回放。它既能解释 AWRA-LS 为什么选这个货位，也能展示泊松到达、混合流双命令、装卸驻留、频次噪声和设备故障下的全过程；同时避免浏览器现场随机仿真带来的演示风险。

## 前提审计

| 前提 | 裁决 | 证据 |
|---|---|---|
| 当前页面已经是档3 | 错误 | 本地档3定义要求泊松到达、装卸、频次噪声、故障；当前页面只有单件循环和静态队列示意 |
| 等待货物会自然变成运输货物 | 错误 | 等待箱是三块无 ID 的独立 mesh；运输箱是单独的 `cargo` mesh |
| 修镜头坐标即可解决 | 不充分 | 当前相机不根据活跃对象组边界求解，固定坐标必然在不同宽高比和作业位置发生裁切/留白 |
| 当前 `realism.py` 可直接导出完整真值轨迹 | 未验证且存在阻塞 | 现脚本聚合统计为主，没有逐事件 trace exporter；canonical U2 三档装卸与 `realism.py` 常数 15s 存在口径冲突 |
| 故障后在线改派已经是真实档3算法行为 | 错误 | `realism.py` 的故障是设备停机/修复；旧动画的库位故障改派明确标为演示编排 |

## 本地真相源

1. `docs/L7_拟真度阶梯设计.md`：档1为匀速/固定权重对照；档2为梯形运动学与场景权重；档3叠加装卸、共同泊松到达、频次噪声、故障。
2. `docs/3D动画设计_0706.md`：已定义左侧紧凑图表、右侧3D、实体队列和统一事件 schema，但 `export_anim_frames.py` 未落地。
3. `docs/Codex任务_S3与导览重做规格_0714.md`：0714 S3 被明确降为单目标位循环，真实队列/KPI/故障不在范围。这是当前能力收缩的直接原因。
4. `sim/realism.py`：单堆垛机事件驱动、共享 arrivals、双命令、响应分位与设备故障，但输出是聚合指标，不是可视化逐事件轨迹。
5. `tools/anim3d_formal.py`：多货物和实体队列由同一手工事件对象驱动，是正确方向；但事件手工编排，故障改派不是真实算法输出。

## 外部研究结论

- AnyLogic 官方把 3D、控件、图表和运行时数据放在同一 presentation 中，并允许相机在运行时跟随运动对象。对本项目的含义是：不应把过程数据和3D拆成彼此无关的画面，也不应依赖固定相机坐标。
- FlexSim 官方的 Statistics Collector 直接监听 3D 对象或流程活动事件，并把同一事件写成统计行。对本项目的含义是：队列、货箱、KPI、曲线必须由同一事件记录更新。
- MathWorks SimEvents 官方同时暴露队列/服务台统计和实体事件序列，用于观察实体如何在系统中移动。对本项目的含义是：货物 ID 的生命周期必须可追踪，而非用三个装饰箱代表队列。
- ISA-101 强调 HMI 的态势感知。对本项目的含义是：左侧不应该堆满同权重图表，只保留评委此刻判断所需的“任务、决策理由、约束、队列、阶段、结果”。
- 可解释优化研究指出，若用户无法理解推荐决策，优化系统难以获得信任。对本项目的含义是：演示不能只显示“去 15/04”，必须显示候选过滤、得分分解、策略选择和局部搜索改进。

## 三种交付架构

### 方案 A：真档3事件总线 + 浏览器实时播放

- `realism.py` 导出逐事件 trace；货物从 arrival 到 waiting、dequeue、load、move、unload、stored/retrieved 全程保持同一 `gid`。
- 3D、货位图、队列曲线、速度曲线、KPI、字幕全部只订阅这一份 trace。
- 单一连续镜头，按活跃对象组动态 fit，并设置安全边距和最大移动/旋转速度。
- 优点：结构最纯、可交互、可切 seed/场景；缺点：实现和浏览器运行风险最高。

### 方案 B：确定性真实轨迹回放

- Python 先用固定 seed 生成并验证 trace；HTML 只做离线播放器，不在浏览器中重跑随机仿真。
- 同样保持唯一事件源和货物 ID；镜头可逐帧预演并做确定性 QA。
- 优点：双击即播、录视频最稳、数字完全可复现；缺点：现场不能自由切换任意参数。

### 方案 C：双幕算法证据 + 档3事件回放（推荐终态）

- 第一幕 20–25s：解释 AWRA-LS 的 `Rank → Assign → Local Search`，显示硬约束过滤、候选 Top3、四项 score 分解、重定位/交换带来的目标下降；LAP 只作为离线批量最优锚。
- 第二幕 55–70s：使用方案 B 的真实档3 trace，连续展示 6–9 个作业、轻中重货、队列、入库/出库、双命令联程、设备故障停机与恢复。
- 在线新到货如实使用 score，不伪装成每件都重跑 AWRA-LS；AWRA-LS 只负责初始批量布局。
- 优点：评委既看到“为什么优”，又看到“真实工况下怎么跑”；缺点：信息设计要求最高，需要严格控制左侧只展示当前决策。

## 推荐镜头规则（单镜头，无多余视角）

1. 主构图对象组：完整货架、I/O、可见等待队列、堆垛机、当前目标。
2. 启动时按对象组 bounding box / sphere 自动 fit；左右与上下至少保留 10% 安全边距。
3. 运行时只在“总览中心”和“当前动作中心”之间缓慢插值；不得切镜头、不得生成画中画。
4. 相机跟随受 master framing 钳制：任何时刻不得裁掉 I/O、当前货物或目标位。
5. 所有说明标签用 DOM 叠层并避让，不在 3D 世界中使用模糊的小字号贴牌。

## 必须展示的项目亮点

1. 20×20 / 400 格与 `(列+层) mod 3 = 0` 的 133 个预占禁用位。
2. 硬约束优先：预占/占用/检修、承重、体积；不合格候选先剔除。
3. 多目标 score：效率、稳定、能耗、位置价值预留；δ 不宣称已证明生命周期收益。
4. AWRA-LS：Rank、可行 Assign、重定位与交换，每步重新检查约束。
5. 在线策略边界：初始批量用 AWRA-LS，新到货逐件用 score；高密度不可行必须 fail closed。
6. 混合流双命令：一趟完成“存一件 + 顺路取一件”，减少空驶。
7. 公平比较：策略共享相同 arrivals / seed；频次噪声检验鲁棒性。
8. 负结果：档3 P50 当前 AWRA 最低，但 P95 由 near 最低；不宣称全面最优。
9. 故障边界：现有真实故障语义是设备停机—维修—恢复；库位故障在线改派需另做算法扩展后才能展示为真值。

## 实施前两道口径闸

1. 统一 U2：`canonical_assumptions.md` 声称档3使用载货垂直速度 0.80 与 6/8.5/12.5s 三档装卸，但 `realism.py` 仍使用 15s 常数且未调用 `travel_time_laden()`。必须先拍板“升级 realism 到 U2”或“回退 canonical 的档3表述”。推荐升级 realism 并重跑档3基线。
2. 故障语义：第一版只展示设备停机/维修/恢复。库位故障改派作为后续可选扩展，完成真实维护位屏蔽、重新选位、约束测试后再进入画面。

## 建议决策

选择方案 C 作为终态；内部按 A 的事件 schema 设计，演示按 B 的固定 seed 离线回放。这样可以同时满足真实性、可解释性与录制稳定性。先做 20 秒“事件链竖切”验证：让 `G016` 的 arrival、队列位置、dequeue、carrier、slot map 和 KPI 全部同源，再扩到 6–9 个作业；未通过该竖切前不继续美术细化。
