### 1.3 严格 20×20 的能力边界

严格定义：单巷道、单个可见堆垛机、20 列×20 层、400 个同时存在且独立寻址的物理储位；同一货物全程由同一堆垛机连续搬运，不接受分页、投影或终点瞬移。

- Factory I/O 原生 Stacker Crane 的 Numerical 目标固定为 `1..54`；单 Rack 18 格；车体行程 10.5 m、平台行程 6.625 m。
- Analog X/Z 只能在固定行程内连续定位，不能延长行程或生成新储位。20 个中心点铺满行程时，水平最大间距约 0.553 m、垂直约 0.349 m，且尚未扣除端部余量。
- Ultimate SDK、Web API 和驱动只开放 I/O/tag，不开放受支持的三维部件、关节和碰撞体创建接口。
- 厂商在 2025 年仍明确当前版本不能导入自定义部件；2.5.10 只增加 OPC UA 热修复，没有改变该边界。
- 结论：**2.5.10 内严格 400 格物理方案 No-Go（高置信度）**。地址容量不是瓶颈，固定几何和运动学才是瓶颈。

## 2. 方案比较与拍板结果

### A. 几何隔离 + 控制诊断 + 故障联锁（采用）

复制内置场景，在副本中逐项验证装载口几何、托盘落稳、运输和放货；只外移被证明干涉的护栏段/立柱/柜体；增加承载台货物检测和跌落故障联锁。优点是能区分几何、控制和物理求解根因，保留工业安全外观；代价是要多做 A/B 场景和近景证据。

### B. 只增加延时和物理精度（不单独采用）

可以降低抖动，却无法修复真实几何穿插，也容易把偶然成功误判为稳定。仅在几何 A/B 后作为单变量实验使用。

### C. 删除全部防护网（否决）

可能暂时避免碰撞，但会掩盖托盘未落稳、柜体干涉或运输动态问题，并削弱演示的工业合理性。只允许在隔离副本中删除已证明冲突的单段，用作反证；不得删除整圈围栏。

## 3. F3 碰撞闭环设计

### 3.1 场景版本

所有场景进入 `l2_factoryio/scenes/`，不覆盖内置场景：

- `F3_baseline_54`: 内置 Automated Warehouse 另存副本，只修正相机和证据标记，不改几何。
- `F3_clearance_A`: 仅外移装载口疑似冲突的单段 safeguard/黄色立柱，其他不变。
- `F3_clearance_B`: A 仍失败时，额外外移实际进入托盘扫掠包络的控制柜；若 B 仍失败，判定护栏不是主根因并回到 baseline。

每次编辑前复制上一版本；禁止反编译或修改 Unity asset bundle；禁止在一个版本里同时改几何、控制延时和物理参数。

### 3.2 控制器诊断相位

`f3_stacker_control.py` 在现有 `snapshot/stop/probe/demo` 基础上增加可停点诊断，不另起重复控制器：

1. `feed`: 只把一个托盘送到 At Load，然后安全清零。
2. `pick`: 完成 Left 侧取货、回中、Lift=False 落到承载台，等待落稳后停止。
3. `travel`: 在 `pick` 通过后只移动至指定货位，不伸叉放货。
4. `store`: 完整放货并返回 rest。

诊断相位必须共享同一组基础 I/O、等待函数、安全清零和日志格式，避免“诊断脚本与正式脚本行为不同”。

### 3.3 稳定与联锁

- Lift=False 且 Moving Z 停稳后增加 1.5 s 初始落稳窗口；该延时只是保守下限，不作为货物仍在位的证明。
- 在承载台布置一个可读的货物存在传感器：运输前必须为真，抵达后仍必须为真；任一点丢失立即 `Target=0`、C0..C6 清零、记录 `CARGO_LOST`，不继续下一箱。
- 在巷道下方/装载口外侧布置不与正常托盘路径相交的窄 Remover 故障区。检测到跌落物时触发 `DROPPED_LOAD` 并终止本轮；Remover 只负责防止散落物污染后续任务，触发过的运行一律判失败。
- 只有几何 A/B 仍有轻微振动时才试物理参数：`physics.timestep=1/240`、`max_substeps=16`、`solver_iterations=40`。最终验收使用正常 1×，并保持 FPS 高于 15。

### 3.4 视觉证据

- baseline 先用近景/Follow a Part 记录首次接触点；禁止再以宽视角和命令日志单独签收。
- 每个相位至少留：动作前、首次接触/停稳、动作后近景；失败必须保留，不覆盖。
- 正式验收录屏同时显示托盘、装载口、堆垛机和目标货位；画面外的 Remover 触发也要在日志中可见。

## 4. 数据流与错误处理

```text
Factory I/O sensors
  -> pymodbus snapshot/interlocks
  -> feed/pick/travel/store state machine
  -> actuator writes
  -> two-phase motion waits
  -> cargo-present + drop-zone checks
  -> PASS evidence or safe-stop failure
```

安全原则：

- 任何 Modbus 错误、互锁断开、动作未起动、动作不停止、货物丢失或跌落区触发，都执行安全清零并保留原始错误。
- F6 只用于开始新实验前恢复干净场景；不得用 F6 抹掉失败后继续同一批验收。
- `position_hint` 只能优化同一进程内幂等命令，不能替代真实位置反馈。
- 离线测试必须覆盖：正常相位、无起动、超时、货物丢失、跌落触发、safe-stop 部分失败和 `0/55` 边界。

## 5. 验收门

F3 只有同时满足以下条件才可签绿：

1. 正常 1×、FPS 持续高于 15。
2. 目标 `[1, 30, 54]` 跑 3 轮，共 9 次；每轮先执行一次干净 F6，避免复用已占货位；覆盖近端底层、中部高层、远端顶层。
3. 9/9 无护栏/柜体接触、无侧翻、无散落、无穿透、无 `Moving` 卡死。
4. 承载台货物传感器运输前后保持有效，Remover/跌落故障区 0 次触发。
5. 每次结束 C0..C6=False、Target=0、叉臂 Middle；下一次不依赖人工拖动物体。
6. 离线回归全绿，点表、日志和实际方向一致。
7. 素材只标“Python 直控技术预演 / 54 格物理 I/O 与时序验证”。
