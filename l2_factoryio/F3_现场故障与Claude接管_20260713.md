## 4. 关键现场语义

### 4.1 I/O 与目标

- Modbus TCP/IP Server：Factory I/O 为 server，Python `pymodbus` 为 client。
- 绑定：`127.0.0.1:502`，Slave 1；网卡必须是 Software Loopback，不是 Meta Tunnel。
- Cargo sensors 为 active-low：`At Entry/Load/Unload/Exit=False` 表示有货，`True` 表示空。
- `Forks Left`：装载输送带侧；`Forks Right`：货架侧。
- `Forks Left/Right` 是保持型输出：True 保持伸出；False 后自动回 Middle。不能到限位就撤销，否则会在交接中把托盘/箱体一并拉回。
- `Lift=True` 是微抬承载状态；带载运输期间必须保持 True。
- `Target 1..54` 为原生货位；`0` 停止；`55` rest/load。

### 4.2 信号不能证明的事情

- `At Load=True` 只表示载入位不再遮挡，不证明箱子在承载台。
- `Lift=True/False` 只是命令读回，不证明实体平台已处于高/低位。
- `At Middle/Left/Right` 只证明叉机构限位，不证明货物水平、未勾挂、未落地。
- `Moving X/Z=False` 可能表示“已完成”，也可能表示“从未启动/被物理约束”；必须区分 START 边沿和 SETTLE 边沿。
- Factory I/O 场景没有现成的 carriage cargo-present/目标货位占用反馈，当前只能依赖近景人工门，或后续在副本中增加传感器/跌落区。

### 4.3 Emitter/Web API 陷阱

- 现场 Emitter tag id：`8e4549f8-74ed-4dd8-ab3d-b8ddb795e8c7`。
- 首次读取发现 `isForced=true, forcedValue=true`。普通 `PUT /api/tag/values` 写 false 不会覆盖强制值。
- 必须使用 `PUT /api/tag/values-force` 把 Emitter 强制为 false；生成单箱时短暂 force true，轮询 `At Entry` 由空变有后立刻 force false。
- Runtime console 可用 `app.web_server = True` 开 Web API；每次完整重启后都要重新确认，不能假定持久化。
- 一次错误就是 Emitter 未解除强制，导致自动生成两托货，后续 F6 才清掉。
